Audit date: 2026-05-13. Target: `tome` v0.1.0 ahead of v0.2.0 ship.

## Summary

- **blocker**: 0
- **major**: 4
- **minor**: 4
- **nit**: 2

- **cargo audit**: PASS — 3 allowed unmaintained warnings, all already enumerated in `deny.toml` `[advisories].ignore` with rationale. No CVE-class advisories.
- **cargo deny check**: PASS — `advisories ok, bans ok, licenses ok, sources ok`. One `multiple-versions = "warn"` entry remains (wit-bindgen 0.51.0 / 0.57.1 via tempfile/getrandom transitives). Not blocking — `bans` policy is `warn` by design.

- **Vendored sqlite-vec**: v0.1.9 pinned 2026-05-12 in `vendor/sqlite-vec/README.md`, both `LICENSE-MIT` and `LICENSE-APACHE` present.

## Findings

| Severity | Area | Finding | File:line or command | Suggested action |
|----------|------|---------|----------------------|------------------|
| major | Scrubbing — registry persistence & output | `tome catalog add` stores the resolved URL **verbatim** in `config.toml` and re-emits it on success (`Added catalog '<name>' from <url> …`). If the user supplies `https://user:token@host/repo`, the credentials land on disk and on stdout/stdin-JSON unscrubbed. Only the `TomeError::GitFailed.catalog` field is scrubbed (via `display_source`). Same data path feeds `tome catalog list` and `tome catalog show`, so the leak persists for the lifetime of the catalog. | `src/commands/catalog/add.rs:80,126-138`; `src/catalog/store.rs:30-52`; re-read by `src/commands/catalog/list.rs:88,117` and `src/commands/catalog/show.rs:31-46`. | Run `scrub_to_string(url.as_bytes())` before assigning to `CatalogEntry.url`, or scrub at the emit boundary. Add a `tests/scrubbing.rs` case exercising `tome catalog add https://u:p@host/repo`. |
| major | File permissions | `config.toml` is written with umask defaults (0644 on macOS/Linux). It holds catalog URLs — which (per the finding above) may include embedded credentials. The Phase 1 SECURITY.md (line 119) already documents this as "permissive POSIX defaults"; flagging here because the URL-scrubbing gap upgrades the impact. Models directory, `manifest.json`, lockfile, and `index.db` are similarly umask-default, but they don't carry user-supplied secrets. | `src/catalog/store.rs::write_atomic` (no `set_permissions(0o600)` before `persist`). | Tighten `config.toml` to 0600 via `tempfile::Builder::permissions` or a `set_permissions` call on the persisted file. Document the choice in `SECURITY.md`. Other Tome-managed files (index DB, model artefacts) can stay umask-default. |
| major | Read-only DB opens | `commands::plugin::open_index_for_read` (used by `plugin list`, `plugin show`, `plugin interactive`, `query`) **and** `commands::status::assemble_report`/`check_drift` both call `index::open` → `Connection::open(db_path)`, which opens **read-write** (no `OpenFlags::SQLITE_OPEN_READ_ONLY`). They never write under the present implementation, but the naming/comment ("read-only-ish") understates the actual flag. A future regression in any of these read paths could mutate the DB without holding the advisory lock and corrupt a concurrent writer's transaction. | `src/commands/plugin/mod.rs:165-171`; `src/commands/status.rs:150,198`; transitively `src/index/db.rs:45`. | Either (a) plumb `OpenFlags::SQLITE_OPEN_READ_ONLY \| SQLITE_OPEN_NO_MUTEX` via a new `index::open_read_only` helper used by all read paths, or (b) keep the current shape but add a `#[deny]`-style assertion comment + a test that opens the index with another process holding the lock and confirms the read paths succeed. Option (a) is more defensive. |
| major | `.gitignore` coverage | Working tree contains `.claude/settings.local.json` and `.claude/agent-foundry/supply-chain-defence.local.json` — both untracked, both regenerate between harness runs. `.gitignore` already covers `.midnight-expert/settings.local.json` and `.claude/scheduled_tasks.lock` but not the `.claude/settings.local.json` family. P9 retro called out the same problem class. | `.gitignore:1-16`. | Add `.claude/settings.local.json`, `.claude/**/*.local.json`, and `.midnight-expert/` (whole dir) to `.gitignore`. Optionally also add `*.local.json` as a wildcard if no checked-in file matches. |
| minor | Scrubbing — long-hex false positives | `KV_SECRET` regex now matches `Authorization: Bearer <token>` as one unit, but the `(?:(?:token\|password\|api[-_]?key\|bearer\|authorization)\s+)?` lookbehind alternative omits **`signature`** despite `signature` being in the outer key alternation. A presigned-URL header in the form `Signature: AWS4-HMAC-SHA256 …` would have its keyword and value matched but the chained-keyword form `signature: AWS4 <blob>` won't pre-consume the `AWS4` keyword. Probably harmless because the regex matches greedily up to the next whitespace anyway, but the asymmetry is a foot-gun for a future maintainer extending the alternation. | `src/catalog/git.rs:64-65`. | Add a test for `signature: AWS4-HMAC-SHA256 Credential=…` to lock current behaviour, or include `signature` in the chained-keyword group for symmetry. |
| minor | Open-DB error message includes raw path | `index::db::open` formats `open <path>: <error>` into `TomeError::IndexIntegrityCheckFailure`. The path is `paths.index_db` which is XDG-derived and benign on most systems, but on a system where `$XDG_DATA_HOME` itself encodes user data (e.g. corporate setups with username in the path) the path lands in error output. Not a credential leak, but a privacy nit. | `src/index/db.rs:46-47`; same for `pragma_update` map_err calls. | Consider eliding the path or using `.file_name()` only — the user already knows their data dir from `tome status`. Low priority. |
| minor | Model URL stored in `manifest.json` verbatim | The downloaded model manifest stores `source_url` verbatim from `MODEL_REGISTRY` (no scrubbing). The pinned URLs are clean today, but if a future model entry uses a presigned-URL CDN (e.g. R2/S3 with `?X-Amz-Signature=…`), that signature would be persisted to disk. The download error path already routes through `scrub_for_diag`; the *success* path does not. | `src/embedding/download.rs:139-150`. | Strip query strings or scrub on write, OR add a documented invariant on `ModelEntry.source_url` that it must be query-string-free. The latter is cheaper and the right call given the registry is compile-time. |
| minor | Advisory-lock contract — `cascade_disable_for_catalog` reads pre-lock | The pre-check that lists enabled plugins for a catalog runs **without** the advisory lock (see `commands/catalog/remove.rs:133-145`), which is documented in CLAUDE.md as intentional (TOCTOU at worst drops an extra plugin's rows, still correct). The corresponding code in `commands/reindex.rs::resolve_targets` (Scope::All and read_enabled_plugins) is the same shape and is NOT documented. A future reader could mistake the pattern for a bug. | `src/commands/reindex.rs:114-177`; `src/commands/catalog/update.rs:109-118`. | One-line comment at each pre-lock open noting "read-only enumeration; TOCTOU acceptable because writer ops re-check inside the lock". |
| nit | `deny.toml` source-trust scope | `[sources] unknown-registry = "deny"` and `unknown-git = "deny"` are strict, but `allow-registry` is single-element (`crates.io`). No allowlist comment; future maintainer might be tempted to add a private registry. | `deny.toml:55-58`. | Add a one-line comment explaining the strict posture. |
| nit | `vendor/sqlite-vec` versioning | README pins v0.1.9 and dates 2026-05-12, but the `vendor/sqlite-vec/sqlite-vec.c` source itself has no header comment binding it to that version. If `README.md` were edited out of sync, the bind would silently break. | `vendor/sqlite-vec/sqlite-vec.c` (head). | When updating, also paste the upstream release tag into a leading C comment in `sqlite-vec.c`. |

## Verified-clean items

- **`scrub_credentials` itself** — comprehensive (4 ordered regexes, signed-URL keys included, safe-context vs unsafe-context branching for long hex). Sound for the documented surface.
- **`git` stderr capture** — every `Git::run` failure routes through `scrub_to_string` (`src/catalog/git.rs:142-148`). No new shell-outs introduced in Phase 2.
- **`reqwest` error scrubbing** — `embedding::download::stream_to_partial` wraps both the `get()` failure and the non-200 status path in `scrub_for_diag` (`src/embedding/download.rs:95-105`).
- **Atomic writes** — `config.toml` (`store::write_atomic`), `manifest.json` (`download::write_manifest`), and the catalog clone (`add::persist_clone` via `tempfile::TempDir`) all use `NamedTempFile::persist` or `rename`. Model body uses `partial/` directory then `rename` — atomic at the directory level.
- **Advisory lock invariants — write paths** — every mutating index access (`plugin::lifecycle::{enable, disable, reindex_plugin, cascade_disable_for_catalog, auto_disable_orphan}`) acquires `acquire_lock(&paths.index_lock)` before opening the DB. No write path bypasses the lock.
- **Path validation** — `paths.model_path` rejects empty / `..` / separators / absolute names; `catalog/manifest.rs::validate_source` covers six escape vectors with 11 test cases.
- **SIGINT propagation** — the cancellation flag is peeked inside both the model-download read loop and the embedder closure inside the transaction (`plugin/lifecycle.rs:413-417`), so a Ctrl-C aborts the SQLite transaction cleanly rather than mid-write.
- **MODEL_REGISTRY shape** — both entries have 64-char hex SHA-256s, non-zero `size_bytes`, MIT licence recorded and surfaced by `tome models list`. The `has_placeholder_checksum()` guard refuses installs against an all-zero hash.
- **`cargo deny` exceptions** — `webpki-roots` (CDLA-Permissive-2.0), `option-ext` (MPL-2.0), `libfuzzer-sys` (NCSA) all explained in `deny.toml` with the pull-in path.

## Top-3 blocker substitutes (for the recap)

There are no blockers. The three highest-impact items, in order:

1. **catalog-add URL scrubbing gap** — major. User-supplied creds in catalog source URLs land on disk and on stdout. Fix at the boundary in `commands/catalog/add.rs`.
2. **`config.toml` permission 0644** — major. The URL-scrubbing gap upgrades this from "nit" to "secondary defence". Tighten to 0600.
3. **Read-only DB opens are actually read-write** — major. Today's code never writes through these paths, but the naming invites regression. Add `OpenFlags::SQLITE_OPEN_READ_ONLY` or document/assert.
