### R-1 — `rustix` transitive→direct promotion + the symlink-primitive spike (FR-007, NFR-004)

- **Decision**: Promote `rustix` from transitive to a **direct** dependency (`rustix = { version = "1", features = ["fs"] }`, matching the `1.1.4` already resolved in `Cargo.lock`). Before any hardening lands, run a short **spike** confirming that, under the already-enabled feature set, `rustix::fs` exposes (a) `openat2` with `ResolveFlags::NO_SYMLINKS` on Linux and (b) a portable per-component `openat` + `OFlags::NOFOLLOW` directory-walk usable on macOS. If the spike confirms both, FR-007 hardens the **intermediate** directory walk for every Tome-managed write sink. If it fails (the primitive is unreachable without a *new* package), FR-007 **degrades** to final-node `O_NOFOLLOW` plus the documented trust-model mitigation — no new package either way.
- **Rationale**: `cargo tree -i rustix` confirms `rustix v1.1.4` is already in the graph via `tempfile` (`default` feature) and `crossterm`←`comfy-table`; `tempfile` enables `rustix/fs`. Promoting it adds **no new package** (NFR-004) and stays on the licence allowlist (rustix is `Apache-2.0 OR MIT OR Apache-2.0-WITH-LLVM-exception` — the first two are allowlisted). This is the exact `filetime`/`encoding_rs` "make-it-explicit" precedent (Phase 4): both were promoted from transitive to direct because the code needed a type/API the parent didn't re-export, using the version the tree already pulls. The spike is mandatory because the Phase 5 "promote regex" research item turned out to be a **myth** (regex was already direct since Phase 1) — so we verify against `cargo tree`, never the planning doc. The deferred cap-std work (`CONCERNS.md` T151, l.235–264) explicitly rejected a no-new-dep `O_NOFOLLOW`-on-final-open route because it only re-hardens the final node; the real win is the **intermediate-component** walk, which needs `openat`-style relative opens — exactly what `rustix::fs` provides without cap-std's heavier `Dir` capability layer.
- **Alternatives**: `cap-std` (rejected — a *new* top-level package + a full capability `Dir` abstraction; rustix gives the same `openat`/`O_NOFOLLOW` primitives at lower cost, and is already present); final-node-only `O_NOFOLLOW` with no `rustix` (rejected — buys zero intermediate-dir protection, the half-measure CONCERNS.md already rejected); a manual `libc::openat` FFI (rejected — `rustix` is the safe, already-present wrapper, and adding `libc` directly would itself be a new direct dep).
- **Spike outcome (F2, T010–T016, 2026-06-01)**: **PASS.** Promotion verified zero-package: `cargo tree -i rustix -e features` resolves a single `rustix v1.1.4` (transitive via `tempfile`'s `default`/`fs` feature and `crossterm`←`comfy-table`), and after adding `rustix = { version = "1", features = ["fs"] }` the only `Cargo.lock` change is `+ "rustix"` under the `tome` package's own dep list — `[[package]]` count unchanged (474→474), one rustix entry, no name added/removed; `cargo check` clean; `cargo deny check licenses` → `licenses ok`. Reachability confirmed against the locked 1.1.4 source: `rustix::fs::openat`/`openat2`/`CWD`/`OFlags{NOFOLLOW,DIRECTORY,RDONLY,CLOEXEC}`/`Mode`/`ResolveFlags::NO_SYMLINKS` all exported under `fs`; `openat2` is gated `#[cfg(linux_raw_dep)]` (Linux-backend only — exactly the contract's "Linux-only" expectation, so the test arm is `#[cfg(target_os = "linux")]`). The portable per-component `openat`+`O_NOFOLLOW` walk **runs and refuses a symlinked intermediate component on macOS** (`tests/symlink_intermediate_guard.rs`, 2 portable tests green). **R2 approach FR-007 takes**: spike PASSED → the **full intermediate-walk primitive** (Linux `openat2(RESOLVE_NO_SYMLINKS)` + portable per-component `openat`+`NOFOLLOW` walk), NOT the degraded final-node-only `O_NOFOLLOW` fallback. **No new package either way** (NFR-004 holds). Two findings the R2 SSOT primitive (`src/util/symlink_safe.rs`) must fold in: (1) the refusal errno is **not** a single value — macOS returns `ENOTDIR` for `O_NOFOLLOW | O_DIRECTORY` on a symlinked directory component but `ELOOP` for `O_NOFOLLOW`-only and for a symlinked final node, while Linux `RESOLVE_NO_SYMLINKS` returns `ELOOP`; the guard must treat the **set** {`ELOOP`, `ENOTDIR`} as "refused symlinked component" (the spike asserts membership, not equality). (2) the walk must anchor at a **trusted, canonical** root dir (operator-owned, opened by canonical path, no `O_NOFOLLOW` on the anchor itself) and validate only the components *below* it — otherwise it trips on irrelevant system symlinks (macOS `/tmp → /private/tmp`); this matches the FR-007 trust model (operator owns the tree; plugin content supplies only the final filename) rather than weakening it.

### R-2 — F-KNN: bounded over-fetch + widen, **no schema change** (FR-001)

- **Decision**: Fix the silent under-fetch in `src/index/query.rs::knn` by **over-fetching then post-trimming with a bounded widen loop**: bind the vec0 `k` to a multiple of `top_k`, apply the existing workspace/`searchable`/`--catalog`/`--plugin` `JOIN`/`WHERE`, and if fewer than `min(top_k, total matches)` post-filter rows survive, re-query with a larger `k` (geometric growth) until either `top_k` matches are collected or the candidate set is exhausted (k reaches the table row count). No SQLite schema migration, no new column, no vec0 `PARTITION KEY`.
- **Rationale**: The bug is that vec0's `k = ?` chooses the global-nearest neighbourhood *before* the filters apply (`query.rs:84` binds `k = ?2` to `top_k`; the `searchable = 1` + workspace + catalog/plugin predicates are a post-`JOIN` `WHERE`). Over-fetch+widen is a contained change to one function and its two callers (`commands/query.rs`, `mcp/tools/search_skills.rs`) with **zero migration risk** — the spec forbids schema changes this phase (FR/Key-Entities: "No schema change"), and a v0.x beta should not ship an avoidable migration. A `PARTITION KEY`/metadata-column approach would be more efficient asymptotically but requires a schema migration and a re-embed/reindex — out of proportion for a beta gate. The widen ceiling (filtered corpus genuinely smaller than `top_k`) returns the true smaller set, never an error and never global-neighbourhood leakage (spec Edge Case).
- **Alternatives**: vec0 `PARTITION KEY` on workspace/searchable + a `rowid IN (...)` constraint (rejected for beta — schema migration + reindex; revisit post-beta if the widen loop's worst-case fan-out is ever shown hot, which it is not at current corpus sizes); a fixed large multiplier with no widen (rejected — the spec explicitly requires it MUST NOT under-fetch as the corpus grows; a fixed multiplier still misses on a large enough corpus, which is exactly the regression test's job to prove).

### R-3 — Read-only `doctor` opens read-only and degrades, never migrates unlocked (FR-002)

- **Decision**: Change `src/doctor/checks.rs` (the schema/index check at l.58–68) to use `index::open_read_only` and swallow a stale/future-schema error into a **degraded report**, mirroring the existing `check_index` pattern at `doctor/mod.rs`. `doctor --fix` keeps the real lock-held migration via the existing `repair_schema` path. No lock is taken on the read-only path.
- **Rationale**: The read-write `index::open` runs `apply_pending` migrations + WAL pragmas, so plain `tome doctor` on a stale DB performs an **unlocked** migration (FR-124 violation + a race with a concurrent `reindex`/`enable` holding `index.lock`), and on a future-schema DB propagates `SchemaVersionTooNew` (exit 73) and **aborts the whole run** — contradicting "doctor never crashes." The read-only-open-then-degrade pattern is already proven in the sibling check; this is a one-site alignment, not a new mechanism. The doctor module already opens read-only elsewhere, so the safe path exists.
- **Alternatives**: Make `index::open` not migrate (rejected — every other caller relies on auto-migrate-on-open; doctor is the exception, so doctor changes); keep read-write but catch the error (rejected — still runs the unlocked migration on a stale DB before the catch).

### R-4 — Cache key + reuse refcount by **scrubbed** URL (FR-003)

- **Decision**: In `src/commands/catalog/add.rs`, key the cache directory (`paths.cache_dir_for`) and the reuse refcount (`workspace_catalogs::refcount_by_url`) by the **scrubbed** URL (the same string every reader resolves by), while `git.clone_shallow` continues to clone from the **raw** URL for authentication. Add an SSH-source round-trip test.
- **Rationale**: Today the cache dir + refcount use the raw `url` (`add.rs:33,67`) but the DB persists `scrubbed_url` and *every reader* (`show`/`update`/`remove`/`list`/doctor/sync) resolves by the scrubbed string. For any URL where scrubbing changes the string — plain SSH `git@host:owner/repo`, `ssh://`, `https://user:token@…` — the clone lands at one sha256 path and is looked up at another (ENOENT on `show`, orphaned clone on `remove`, reuse never triggers). Keying by the scrubbed URL everywhere makes the writer agree with every reader. Cloning still needs the raw URL because the credential lives there. The common case (plain `https://host/owner/repo`, raw==scrubbed) is unaffected (spec Edge Case). Existing fixtures only use plain HTTPS, which is why the divergence was invisible — the new test must use an SSH source.
- **Alternatives**: Persist the raw URL and scrub only at display (rejected — violates "never store credential-shaped strings"; constitution §XIII); have readers re-derive the raw URL (rejected — impossible, the credential is gone after scrubbing, by design).

### R-5 — MCP prompt names assigned against a single global taken-set (FR-004)

- **Decision**: In `src/mcp/prompt_collision.rs` + `prompts.rs`, assign every final prompt name against **one global taken-set**, suffixing `{base}{n}` until free *and re-checking the suffix against the same taken-set*, so no user-invocable entry is dropped on a cross-kind collision. `doctor`'s prompts report reflects the true resolution.
- **Rationale**: The current loser-suffix (`{base}{idx+1}`) is never checked against other buckets' names, so the terminal `HashMap::insert` silently discards a duplicate — reachable with non-hostile input (Command `foo` + user-invocable Skill `foo` + Command `foo2` → one entry vanishes from `prompts/list` and returns `prompt_not_found` on `prompts/get`). A single taken-set with suffix-until-free is the standard fix and preserves both entries. This is the same "single namespace, suffix backstop" discipline Phase 6 personas already joined (006 R-10/FR-066) — the fix makes the *implementation* match that documented intent. Verified end-to-end by the FR-012 harness (the fix may land before the harness exists; the harness gates its verification, not its implementation).
- **Alternatives**: A per-kind namespace (rejected — slash names share one MCP namespace; collisions across kinds are real); fail the server on collision (rejected — a beta user's two innocuously-named entries must not break enumeration).

### R-6 — Workspace settings emitted via `toml_edit`; control chars rejected at the manifest boundary (FR-005)

- **Decision**: Emit `settings.toml` in `src/workspace/init.rs` via `toml_edit` (deleting the hand-rolled `escape_toml_basic`, matching the sibling `rename`/`regen_summary` paths) **and** reject control characters (newline et al.) in catalog names at the manifest-parse boundary (`catalog/manifest.rs`).
- **Rationale**: `escape_toml_basic` escapes only `\` and `"`, so a newline-bearing catalog name (free-form third-party text; the manifest validator only rejects empty) produces unparsable `settings.toml` → `WorkspaceMalformed` (exit 70) on *every* harness op for that workspace, reachable via `workspace init --inherit-global` re-emitting a poisoned global catalog name. `toml_edit` escapes correctly and is already the emission path for the sibling rename flows, so this also removes a bespoke escaper (single-source-of-truth). Defence in depth pairs the correct emitter with a boundary reject so the bad value never enters the system; rejecting control chars at the manifest boundary is consistent with the §IV strict-schema posture on the *recognised* `name` field (a value reject, not an unknown-field reject — third-party manifests stay lenient on unknown fields).
- **Alternatives**: Extend `escape_toml_basic` to cover control chars (rejected — re-implements what `toml_edit` already does correctly; the Rule-of-Three says delete the bespoke escaper); reject only, keep `escape_toml_basic` (rejected — leaves a second emission path that can drift; both halves are cheap).

### R-7 — Bounded third-party reads via the **existing per-class caps**, swept across the whole class (FR-006)

- **Decision**: Route every unbounded `std::fs::read`/`read_to_string` on a third-party path through `crate::util::bounded_read(path, <existing per-class cap>)` — `PLUGIN_MANIFEST_MAX` (256 KiB) for manifests/frontmatter, `HARNESS_MCP_MAX` (1 MiB) for settings/hooks — **not** a single new cap. Sweep the whole class: at least `plugin/manifest.rs:61`, `catalog/manifest.rs:46`, `plugin/lifecycle.rs:958`, `plugin/components.rs:170`, and `doctor/checks.rs:174`.
- **Rationale**: `plugin/manifest.rs` uses raw `std::fs::read` (allocates the whole file), bypassing the 256 KiB cap the sibling `SKILL.md` path already enforces (`frontmatter.rs:292`) — a multi-GiB `plugin.json` OOM-crashes Tome at `enable`/`show`/`list`. The cap and the bounded-read helper already exist; the fix is mechanical routing, one line per site. The Phase-6 "fix the class, not the instance" + "fix one sink, miss the parallel" lessons mandate sweeping every sibling read, especially the `doctor` read-only/CI surface (`doctor/checks.rs:174` reads `tome-catalog.toml` unbounded). Reusing the per-class caps (not inventing one) keeps the named-error story (`ManifestInvalid`/size error) intact per site.
- **Alternatives**: One global `MAX_THIRD_PARTY_READ` cap (rejected — the spec is explicit: per-class caps, not a single new cap; a 1 MiB settings file is legitimate where a 1 MiB frontmatter is not); a streaming parser (rejected — over-engineered for files that are legitimately small; the cap+named-error is the YAGNI fix).

### R-8 — OpenCode rules: lowest-common-denominator **inline** body (FR-008)

- **Decision**: In the shared-`AGENTS.md` reconciliation (`harness/sync.rs::compute_rules_body`), pick the lowest-common-denominator body style across live sharers: if **any** sharer requires an inline body (OpenCode, which can't resolve `@`-includes), write the body **inline** (valid for include-capable harnesses too) rather than the first sharer's style.
- **Rationale**: Today the dedup writes only the *first* sharer's body style; registry order makes Codex (`AtInclude`) win, so OpenCode receives `@.tome/RULES.md` as literal prose and Tome's rules are never delivered — silently, no error. The parallel *guardrails* reconciler already does this correctly via a union, so this aligns hooks/rules with the proven sibling pattern (another instance of the "fix one sink, match its parallel" discipline). Inline is a strict superset: `AtInclude` harnesses also resolve an inline body, so writing inline whenever any sharer needs it harms no one.
- **Alternatives**: Write per-harness separate files (rejected — defeats the shared-`AGENTS.md` design; duplicates content, NFR-009); detect OpenCode and special-case it (rejected — the union/LCD rule is more general and matches the guardrails reconciler).

### R-9 — `catalog remove --force` re-derives its cascade inside the lock (FR-009)

- **Decision**: In `src/commands/catalog/remove.rs`, move the enabled-plugins read **inside** the `index.lock`-held closure so the cascade input is re-derived after the lock is acquired, not reused from a stale pre-lock `Vec`.
- **Rationale**: The enabled list is read before the lock; the locked closure reuses the stale `Vec`, so a concurrent `plugin enable` (serialising on the same lock) is a lost update → a ghost-enabled plugin whose catalog enrolment is deleted. The contract already says "acquire lock, *then* check enabled plugins"; the fix makes the code match. Maps to `CONCERNS.md` TD-017. The single-process common case is unaffected; the fix matters for scripted/parallel CI usage.
- **Alternatives**: A second post-lock validation that aborts on drift (rejected — re-deriving is simpler and always correct; aborting would surface a confusing error on a benign race).

### R-10 — `harness/sync.rs` decomposition: file-move into `reconcile/{hooks,guardrails,agents}.rs`, strictly behaviour-preserving (FR-011, NFR-005)

- **Decision**: Decompose the 1,737-LOC `src/harness/sync.rs` into a thin orchestrator plus per-sink modules `src/harness/reconcile/{hooks,guardrails,agents}.rs`, **lifting the already-factored `reconcile_hooks`/`reconcile_guardrails`/`reconcile_agents` functions and their private helpers verbatim** into their own files. The orchestrator (`sync_project` + `SyncDeps`/`SyncOutcome`/`SyncSubsystem`/`HarnessDecision`/`Action` + the shared `HarnessSnapshot`/`collect_harness_snapshots`/`group_by_path`/`read_*_settings`/`relative_path` helpers + the rules/mcp write helpers) stays in `sync.rs` (or a `reconcile/mod.rs` re-export). The fixed sink order (`first_clash` → hooks → guardrails → agents), `first_error` precedence, idempotence, and the `SyncOutcome` JSON wire-pin are unchanged.
- **Rationale**: `sync.rs` is **already internally factored** into the `reconcile_<sink>` shape (hooks ≈l.752, guardrails ≈l.1007, agents ≈l.1196, orchestrator l.187–447) — the brief's target structure is therefore a **file move**, not a redesign. The single biggest behaviour-preservation risk is the **mass-delete safeguard** (open the central DB read-only and *propagate* on an existing DB, never `.ok()`-swallow → that empties the enabled set and mass-deletes every reconciled file; `CONCERNS.md`/006 R-1) — the move must carry it intact per module. The decomposition lands **first** (before FR-007/FR-008) so those fixes land in the clean structure (spec Sequencing). RUST-1/RUST-2 efficiency cleanups are **explicitly out of scope** and are NOT folded in (they would risk behaviour change in a "strictly behaviour-preserving" refactor).
- **Alternatives**: A full redesign of the reconcile pipeline (rejected — out of scope, behaviour-change risk; the functions are already cleanly separable); leaving `sync.rs` monolithic and only adding FR-007/008 (rejected — the spec sequences the decomposition first precisely so the symlink/OpenCode fixes land in modules a person can hold in their head, and the project has twice shipped a one-sink-fix that missed its parallel).

### R-11 — In-process MCP test harness closes GAP-1 on the library side (FR-012)

- **Decision**: Build an in-process MCP test harness (a `tests/common/` driver that constructs and drives a real server instance via the library API, no real model load — `StubEmbedder`) and a new `tests/exit_codes_e2e_mcp.rs` that exercises the MCP-internal exit codes lacking end-to-end coverage (`GAP-1`: 9, 26–29) and verifies the FR-004 prompt-collision fix end-to-end through `prompts/list` + `prompts/get`.
- **Rationale**: `tests/exit_codes_e2e_mcp.rs` confirmed absent; the MCP-internal codes are covered only at library level (`tests/exit_codes.rs` + `tests/mcp_prompts*.rs`), and the CLI binary can't reach them (the MCP server is the surface). Rich in-process MCP test infrastructure already exists (`tests/mcp_server.rs`, `mcp_lifecycle.rs`, `mcp_prompts.rs`) to build the driver on. The harness sits on the **library** side (drive the server in-process, no ONNX load) — the established library-vs-CLI-binary test boundary (Phase 4 P3) is why it can finally cover codes the CLI binary can't. This is a pre-named, pre-scoped backlog item, not new scope.

### R-12 — Crate rename `tome` → `tome-mcp`, command stays `tome` (FR-017; complexity-budget note)

- **Decision**: Rename the published **crate** to `tome-mcp` (`Cargo.toml [package] name`), and add an explicit `[[bin]] name = "tome"` so the installed **command** stays `tome`. Treat the rename as its own slice with a full `cargo check` + `--locked` re-verification, and grep every `tome`-as-package-name reference: `Cargo.lock` `name`, CI workflows, the binary-size assertion (`target/release/tome` — stays valid *because* `[[bin]] name = "tome"`), `tests/exit_codes_e2e.rs` (invokes the CLI binary by name), README install lines, and the `--version`/`-V` pre-parse (`CARGO_PKG_NAME`).
- **Rationale**: `cargo publish` to `tome` is rejected — the name is taken on crates.io (created 2021, all versions yanked, ownership permanent; verified live, B1). `cargo install tome-mcp` yielding a `tome` binary is the fast, certain path vs. a discretionary, timeline-unsafe name-reclaim request. The `[[bin]]` decoupling is the standard Cargo idiom and keeps every binary-name consumer (the size check, the exit-code e2e harness, install instructions for the command) working unchanged. *Complexity-budget justification*: the rename touches packaging metadata only, adds no module and no dependency, and is forced by an external namespace collision — the minimum change that unblocks the crates.io channel while preserving the user-facing command.
- **Alternatives**: `tome-cli`/`tomekit`/`tome-plugins` (rejected — `tome-mcp` best signals the dual CLI+MCP identity and the MCP-server differentiator); request the abandoned name (rejected — discretionary, not timeline-safe for a beta); change the command too (rejected — gratuitous user-facing churn; the binary name is decoupled from the crate name for exactly this case).

### R-13 — Release pipeline via `cargo-dist`; self-contained Linux+macOS binaries verified per-target (FR-018/019/020, NFR-008/010)

- **Decision**: Use **`cargo-dist`** to produce per-platform archives (Linux + macOS × x86_64 + aarch64) with checksums, push a Homebrew formula to the tap, and publish to crates.io. Each target's CI job **verifies the absence of an application-specific sidecar** (`ldd`/`otool -L` shows no `libonnxruntime`) by inspecting the binary's dynamic dependencies. Builds are `--locked`; `Cargo.lock` is committed and shipped in the tarball. The generated release workflow is subject to the same gates as CI (fmt/clippy/`cargo-deny`, version-pinned actions including the upgraded checkout). The actual `cargo publish`, the git tag, the tap-PR merge, and release-notes posting remain **user-reserved** actions — the pipeline is wired but not triggered.
- **Rationale**: `cargo-dist` is the de-facto Rust release-automation tool; it targets the required platforms and pushes a cross-owner tap given a PAT secret. "Self-contained" means no application-specific sidecar: `ort`'s default `download-binaries` statically links ONNX Runtime, and `llama.cpp`/`sqlite-vec` are vendored/static — linking the platform's own C/C++ runtime + libc is acceptable, so the Linux build targets a glibc baseline so the binary runs across mainstream distributions (spec Edge Case). The per-target `ldd`/`otool` check makes SC-007 mechanically verifiable. `--locked` (NFR-008) guarantees a tagged artifact matches the audited dependency set — no transitive bump silently enters a release. The macOS binary is already self-contained (verified, RELEASE-READINESS); Linux is expected likewise after the glibc-baseline build, confirmed per-target. Publish/tag staying user-reserved matches the standing project discipline (every prior Polish handover).
- **Alternatives**: Hand-rolled `release.yml` matrix (rejected — re-implements cargo-dist's checksum/archive/formula/installer generation; more surface to keep gated); `cargo-binstall`-only metadata (rejected — doesn't produce the binaries or the Homebrew formula); dynamic-load ONNX (`ort-load-dynamic` + `brew install onnxruntime`) (rejected — pushes a runtime-library install onto the user, the opposite of "one command, self-contained"; out of scope to re-architect the inference linking for a beta).

### R-14 — Homebrew tap pushed with a least-privilege cross-owner PAT (FR-019, NFR-010)

- **Decision**: cargo-dist pushes the formula to `aaronbassett/homebrew-tap` using a **least-privilege** classic/fine-grained PAT scoped to write only the tap repo (cross-owner: the tap lives under a different owner than the crate repo). The PAT is a CI secret, never logged, and is an **operator-provided** secret.
- **Rationale**: cargo-dist's default `GITHUB_TOKEN` cannot write a repo under a *different* owner, so a cross-owner tap requires an explicit PAT. Least-privilege (tap-repo-write only) bounds the blast radius; never-logged satisfies §XIII and NFR-010. Providing the secret + creating the tap is an operator-only action (the orchestrator cannot mint credentials).
- **Alternatives**: A tap under the same owner as the crate repo (rejected — the tap already lives under `aaronbassett/`; not the orchestrator's to relocate); a broadly-scoped PAT (rejected — violates least-privilege/NFR-010).

### R-15 — Aggregated third-party-licence bundle covering cargo graph + native components (FR-019, NFR-007)

- **Decision**: Generate an aggregated `THIRD-PARTY-LICENSES` document with **`cargo-about`** (config pinned to the existing allowlist), covering the cargo dependency graph, and **manually append** the licence notices for the statically-linked/vendored native components (ONNX Runtime, llama.cpp, vendored sqlite-vec) that `cargo-about` cannot see. Attach it to each tagged binary release.
- **Rationale**: The binary statically links MIT/BSD/Apache-2.0 code; those licences require carrying upstream notices in binary redistributions (a standard compliance obligation, not a legal blocker — all permissive). `cargo-about` enumerates the cargo graph and is itself a CI tool (no runtime dep). The native components are linked outside the cargo graph (ONNX via `ort`'s downloaded binary; llama.cpp via `llama-cpp-2`'s C++ static lib; sqlite-vec vendored under `vendor/`), so their notices are appended deterministically. `cargo-deny` stays green (NFR-007).
- **Alternatives**: `cargo-bundle-licenses` (acceptable equivalent; `cargo-about` chosen for its templating + allowlist-config reuse); shipping only the repo LICENSE files (rejected — doesn't cover transitive/native notices the binary redistribution obligation requires).

### R-16 — docs.rs build made to succeed (FR-025)

- **Decision**: Make the docs.rs build succeed despite `ort`'s build-time ONNX-Runtime download by adding `[package.metadata.docs.rs]` (feature-gating the fastembed embedder off for the docs build, or setting the docs.rs build to skip the network step); if that proves brittle, point the `documentation` Cargo field at the repository instead.
- **Rationale**: docs.rs builds in a network-isolated sandbox, so `ort`'s `download-binaries` fails there (PKG-2). docs.rs isn't on the install path (downgraded major→minor in the audit), so the acceptable floor is "the `documentation` field resolves" — either a working docs.rs build (preferred) or a repository pointer (fallback).
