# Phase 6 / Phase 7 (US5 — Privilege governance + doctor extensions) Retro

**Branch**: `006-us5-doctor`
**Baseline**: US4 merged @ 39153b2 (PR #135).

## What this phase landed

Plugin-agent privileged fields (`hooks`/`mcpServers`/`permissionMode`) pass
through to Claude Code by default but are auditable via doctor's
`PrivilegeEscalationReport` and strippable via the opt-in layered
`strip_plugin_agent_privileges` setting. `tome doctor` gains five emit-only
reports (Hooks/Guardrails/Agents/PrivilegeEscalation/Persona) with byte-stable
JSON pins; `--fix` repairs only the safe derivable cases (re-render stale
guardrails, re-emit missing agents, remove orphaned `<plugin>__*`) and never
removes a non-matching hook or user content. `tome plugin show` lists agents
+ hooks.json/GUARDRAILS.md presence + resolved persona name.

## What Worked Well

- **Doctor as a read-only projection over the US1-US4 harness infra**: the
  five reports reuse the production read helpers (`read_rewritten_entries`,
  `present_region_keys`, `CanonicalAgent::parse`, `agent_name_clash_set`,
  `owned_agent_files`) rather than re-deriving state — minimal new surface.
- **`--fix` via the idempotent `sync_project`** inherits all of US1-US3's
  write safety for free (structural-match-only hook removal, marker-bounded
  guardrails, `<plugin>__*`-only agent removal, symlink refusal). Security
  came back APPROVE with 0 findings.
- **Privilege strip via a per-emission clone** keeps the audit source intact
  (the borrow checker enforces it — `prepared` is borrowed immutably), so the
  `PrivilegeEscalationReport` always sees what a plugin ships even when strip
  is on (FR-051).

## What Didn't Work

- **Per-branch refresh gating left a stale report**: `--fix` refreshed the
  Phase 6 surfaces only in the Phase 6 branch (gated on `!harness_sync_ran`),
  so a combined harness+Phase-6 fix emitted a stale Phase 6 report even
  though the shared sync had already repaired the files. Fix: refresh after
  ANY successful project sync, regardless of which branch ran it.
- **A report field's semantic was drift-only**: `GuardrailsReport.suppressed`
  derived from on-disk regions, so a correctly-suppressed plugin (region
  intentionally absent) never appeared — a dead branch. The useful audit is
  the steady-state set (ships both GUARDRAILS.md + hooks.json).
- **Tests proved behaviour at the wrong granularity**: hooks-drift tested
  event-presence (a buggy event-granularity impl would pass) instead of
  entry-identity; `dropped_fields` was pinned by hand because claude-code
  drops nothing — neither exercised the real path.

## Workarounds & Solutions

- **Pre-stage on-disk state for read-only check-fn tests.** Running the real
  `sync_project` in a doctor test is ~65s (it triggers summarisation/index
  work); plant the `settings.local.json` hooks / marker regions / `<plugin>__*`
  files directly and call the check fn, reserving the real sync for the few
  `--fix` end-to-end cases.

## Packages & Dependencies

- No new top-level dependencies. Reuses the US4 scalar resolver for
  `strip_plugin_agent_privileges`; the US1 removal glob + US2/US3 ownership
  predicates for `--fix`.

## Patterns & Code

- **Refresh all affected report surfaces after a shared side-effecting fix.**
  When a `--fix` operation (here `sync_project`) can be triggered from
  multiple branches and touches multiple subsystems, refresh EVERY affected
  surface after it succeeds — gate the refresh on "the operation ran", not on
  "my branch ran it".
- **Read-only doctor projection** over the production reconcilers: a check fn
  that re-reads the same on-disk/source state the writer produced, never
  writing, is the honest auditor (FR-124).

## For Next Time

- When adding a report field, write the test that distinguishes the correct
  semantic from a plausible-but-wrong one (entry-identity vs event; real drop
  vs hand-built pin) — a happy-path assertion that both impls satisfy proves
  nothing.
