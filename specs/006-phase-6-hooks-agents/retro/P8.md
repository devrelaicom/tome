## What Worked Well

- **The phase-wide pass earned its keep even after five clean per-US passes.**
  Both majors live at US-interaction seams (hooks-vs-guardrails exit-code policy;
  the hooks→guardrails→agents `first_error` composition) that no single-US
  reviewer could see. This is the Phase 5 P8 lesson, re-confirmed.
- **Reviewers reading the `us{1..5}-disposition.md` docs FIRST** kept them from
  re-litigating settled items and focused them on genuine cross-US drift. The
  Rust + security passes confirmed the named backlog (TD-061/063/064, R5-3) stays
  deferred rather than re-deriving it.
- **One `StubHarness` driving all three sinks** (`RealJson` hooks +
  `with_hook_settings` + `with_native_agents` + the default non-suppressing
  in-file guardrails region) made the idempotence / e2e / precedence tests
  hermetic, fast, and free of real-harness path coupling. One enabled `agent`
  row also surfaces the plugin in the hooks + guardrails enumerations (DISTINCT
  catalog/plugin), so a single seed feeds all three sinks.
- **Pre-running `cargo clippy --all-targets -- -D warnings` before committing new
  test files** caught three `doc_lazy_continuation` lints that `cargo test` (no
  `-D warnings`) happily compiled — turning a would-be failed-commit cycle into a
  quick fix.

## What Didn't Work

- **A background commit failed transiently** mid-clippy (likely build-lock /
  resource contention with the subagent's just-finished cargo run). All three
  gates passed on a clean re-run, confirming it was infra, not a code defect.
  Lesson: a failed pre-commit aborts the commit (nothing lands), so the fix is to
  confirm-transient-or-fix-root-cause then make a NEW commit — never `--amend`,
  never `--no-verify`.
- **`cargo test` green is NOT sufficient pre-commit validation.** `-D warnings`
  promotes clippy doc-lints (here `doc_lazy_continuation` on a numbered list
  followed by an unindented paragraph, and a `/// +`-prefixed line read as a
  markdown bullet) to errors that plain `cargo test` ignores. New test files need
  a clippy pass, not just a test run.

## Workarounds & Solutions

- **Cargo.lock must be regenerated (`cargo check`) after a version bump BEFORE
  committing** — otherwise the commit's own clippy run updates the lock *after*
  `git add`, leaving a dirty `Cargo.lock` post-commit and failing any CI
  `--locked` check. The version bump itself triggers a recompile of the
  `CARGO_PKG_VERSION` consumers (`--version`, doctor `tome_version`).
- **e2e agent-surface assertion vs the stub.** `StubHarness::translate_agent`
  deterministically echoes the frontmatter-STRIPPED body (it does not re-emit
  YAML), so the e2e test asserts the translated body (source `description:`
  absent, body present) rather than rendered frontmatter. Real MD+YAML frontmatter
  fidelity stays covered against the actual claude-code harness in
  `harness_sync_stub.rs`. Using claude-code in the e2e would have suppressed the
  guardrails region (hooks present), breaking the whole-flow proof — so keeping
  all three sinks under one stub was the right tradeoff.

## Patterns & Code

- **Cross-sink exit-code reconciliation.** When one sink's symlink-refusal code
  was corrected (guardrails 7→46 in US3 C3-1), the parallel sink (hooks) had to be
  brought in line with the authoritative exit-codes contract too — a
  cross-contract consistency the per-US pass could not see. Audit "what's the
  policy for the SAME operation class across all sinks" at phase assembly.
- **cap-std / `O_NOFOLLOW` DEFER discipline.** When the clean fix needs a new
  top-level dependency the constitution gate forbids, document the deferral with
  the trust-model rationale (operator owns the harness dirs; plugin content never
  supplies directory path components; final-node symlink refusal + atomic write +
  mode preservation are the current mitigations) AND why the no-dep alternative
  (`O_NOFOLLOW` on the final open only) is inadequate — rather than forcing the
  dep or silently dropping the item.
- **mtime-capture is the load-bearing half of an idempotence proof.** An
  empty-`added`/`updated` outcome alone misses a "rewrites identical bytes"
  regression; capturing each sink output's mtime before a >1s sleep and asserting
  it is unchanged after the second sync catches exactly that class.
