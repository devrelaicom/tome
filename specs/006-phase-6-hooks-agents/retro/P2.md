# Phase 6 / Phase 2 (Foundational) Retro — F1+F2+F3

**Branch**: `006-foundational`
**Phase 6 baseline**: planning PR #130 merged @ 384a95a (on main after #128 v0.5.0 squash).

## What this phase landed

Pre-allocate the four Phase 6 `TomeError` variants + exit codes 43–46 (F1),
land the load-bearing `EntryKind::Agent` widening with every exhaustive match
site updated + a marker-only schema migration (F2), and extend the
`HarnessModule` trait surface (`hooks_strategy`, `hook_settings_path`,
`guardrails_target`, `supports_native_agents`, `agent_dir`, `agent_format`,
`translate_agent`) plus the new shape enums, with safe defaults and a
`StubHarness` implementation (F3).

## What Worked Well
<!-- Effective patterns, tools, approaches -->

- **One contract-grounded subagent for tightly-coupled foundational work.**
  F1+F2+F3 touch overlapping concerns (the closed error set, the entry
  schema, the harness trait) and a single `devs:rust-dev` agent with a
  precise brief (exact file:line match-sites, exact exit codes, the
  no-catch-all rule) produced a clean implementation in one pass: fmt +
  clippy + the full 153-binary suite all green, 0 failures, first try.
- **Marker-only migration for a free-text domain widening.** `kind` is a
  free-text TEXT column, so admitting `'agent'` needs no DDL — a no-op
  `apply` that only advances `SCHEMA_VERSION` 3→4 keeps the migration
  registry and doctor's schema check honest without touching data (R-11).
- **Safe-by-default trait methods.** The seven new `HarnessModule` methods
  all have conservative defaults (GuardrailsOnly / None / false / in-file
  region, no suppression), so the five real harness modules inherit correct
  behaviour and every existing harness test stayed green without edits.

## What Didn't Work
<!-- Problems, issues, limitations encountered -->

- **Subagent left a stale background `cargo` process.** The F-implementation
  subagent's final report was garbled and it left a `cargo check
  --features test-loopback` running after returning; that held the `target/`
  build lock and made my verification `cargo test` hang at 0% CPU. Fix: kill
  the stale process, then re-run. Lesson: brief subagents to NOT leave
  background processes, and always verify with my own gate run.
- **Full `cargo test` exceeds the background-task lifetime.** Piped through
  `tail` in a `run_in_background` Bash task, the suite got SIGKILLed (~exit
  144) before finishing, and `tail` buffering hid all progress. See
  Workarounds.

## Workarounds & Solutions
<!-- Bug fixes, non-obvious solutions, clever workarounds -->

- **Squash-merge divergence (planning PR):** `main` advanced via the #128
  squash while the planning branch carried the unsquashed phase-5-polish
  commits. Resolved by `git rebase --onto origin/main e8d7bae <branch>` to
  replay only the two phase-6 planning commits, then pushing to a fresh
  branch (`006-phase-6-planning`) because force-push to the already-pushed
  branch is correctly disallowed. Per-phase branches off fresh `main` avoid
  recurring divergence.
- **Running the full suite locally:** launch detached
  (`nohup cargo test > /tmp/log 2>&1 < /dev/null & disown`) and watch the
  logfile with a `Monitor` (failure signatures + a terminal summary on
  process exit). Survives the background-task timeout; gives live progress
  the buffered-`tail` approach hid.

## Patterns & Code
<!-- Reusable patterns, helpful code snippets, architectural insights -->

- **Test-configurable unit-struct upgrade:** `StubHarness` went from a unit
  struct to a `#[derive(Default)]` struct with `with_*` builder setters; all
  existing `StubHarness` literals became `StubHarness::default()`. The
  Default reproduces the old behaviour + the trait's safe defaults, so the
  change is backward-compatible and tests opt into new capabilities
  explicitly. Reusable shape for the next "make this test double
  configurable" need.
- **Skeleton type module ahead of behaviour:** `harness/agents.rs` lands the
  real `CanonicalAgent`/`TranslatedAgent` types in F3 (so the trait method
  signature compiles) with the parsing/translation deferred to US1 — the
  public surface is stable, US1 only fills in bodies.

## For Next Time
<!-- Improvement suggestions, lessons learned, things to try differently -->

- **Verify slices with targeted tests, not the full suite.** The full
  `cargo test` is ~20-25 min locally and the git hooks don't run it (only
  fmt/clippy/typos). Run `cargo test --test <file>` for the changed area and
  let CI be the authoritative full-suite gate.
- **Brief subagents to leave no background processes** and to give a
  structured final report; verify their work via my own gate run regardless.
