# Phase 6 / Phase 6 (US4 — Agent personas via MCP prompts) Retro

**Branch**: `006-us4-personas`
**Baseline**: US3 merged @ 9fac54f (PR #134).

## What this phase landed

An opt-in `expose_agents_as_personas` setting (default false, first-declarer-
wins project→workspace→global) exposes each enabled agent as a
`<name>-persona` MCP prompt (clash-prefixed where required) plus a single
global reserved `drop-persona`, reusing the Phase 5 prompt + substitution
machinery (no parallel substitution path). `prompts/get` returns the
frontmatter-stripped body wrapped in the role-assumption template with Phase 5
built-in/env substitution + a single catch-all `args`. Effective value read
from the MCP server's single startup scope (project layering has no effect on
a running server).

## What Worked Well

- **Personas as a parallel path on the Phase 5 prompt machinery**: agent rows
  feed `collect_persona_identities` into the SAME `resolve_collisions`
  namespace + `build_context_for_entry` substitution — no parallel
  substitution path (NFR-007), `drop-persona` reserved via an empty-seed
  collision tie-break, body wrapped in the role template. Clean reuse.
- **Generic scalar resolver** (`resolve_scalar_with` over a field accessor) —
  US5's `strip_plugin_agent_privileges` adds a one-line call site, no second
  resolver.
- Security review came back PASS (the persona path inherits Phase 5's audited
  substitution/arg handling; double opt-in trust gate).

## What Didn't Work

- **A 529 overload interrupted the impl subagent mid-work** — it had changed
  `build_for_workspace`'s signature but not finished updating all ~15 callers,
  leaving a compile error. I scoped-fixed the callers (a line-scoped perl
  matching only `build_for_workspace` lines, appending `false`) and re-ran the
  gates myself. Lesson: a returned-with-error subagent may have left the tree
  uncompilable — always run the gates before trusting it.
- **Row-projection gap**: `enabled_agents_for_workspace` did not SELECT
  `plugin_version`/`indexed_at` that the command/skill registry query already
  had, so personas rendered `${TOME_PLUGIN_VERSION}` empty AND every persona
  won base-name collisions (empty `indexed_at` sorts first) — two reviewer
  MAJORs from one omission.

## Workarounds & Solutions

- When a returned subagent's work fails to compile, find the un-updated call
  sites with `grep` and fix mechanically; the Bash safety classifier can be
  briefly unavailable during an overload (read-only ops still work — retry the
  write/compile after a beat).

## Packages & Dependencies

- No new top-level dependencies. Reuses the Phase 5 prompt/substitution layer
  (`build_context_for_entry`, the single-sweep substitution) unchanged.

## Patterns & Code

- **When extending a row-projection struct for a new consumer, mirror the
  sibling query's full column set.** The persona path needed exactly the
  `plugin_version` + `indexed_at` columns the command/skill registry query
  already selected; omitting them silently degraded substitution + collision
  tie-breaking.
- **SSOT scope-loaders** (`settings::scopes`): the project→workspace→global
  settings-file loaders are now one canonical copy consumed by the MCP startup
  resolver, the harness list command, and harness sync — was triplicated.

## For Next Time

- Brief the impl subagent that a `build_for_workspace`-style signature change
  must update every caller (production + all test call sites) in the same
  pass; and to mirror the sibling registry query's columns when adding a
  parallel row projection.
