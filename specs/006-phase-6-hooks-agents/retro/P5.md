# Phase 6 / Phase 5 (US3 — Guardrails fallback + rules-file correction) Retro

**Branch**: `006-us3-guardrails`
**Baseline**: US2 merged @ a991312 (PR #133).

## What this phase landed

`GUARDRAILS.md` renders as per-plugin marker regions
(`<!-- START GUARDRAILS: <catalog>:<plugin> -->` … `END`) in each harness's
rules file (Cursor → standalone sibling), suppressed on `CLAUDE.md` when the
plugin ships JSON hooks; plus the Phase 4 correction making Claude Code's
rules sink `CLAUDE.md` (> `.claude/CLAUDE.md`, dropping `AGENTS.md`).
Deterministic placement (rules block first, then regions in lexicographic
order), overwrite-in-place, orphan removal, Cursor sibling deletion when
empty; atomic mode-preserving symlink-refusing writes; suppression computed
hooks-first.

## What Worked Well

- **One coherent US3 subagent** (correction + guardrails + tests) — like US2,
  the right grain; gates green first pass.
- **Parameterised `MarkerSpec` engine** generalised `rules_file.rs`'s single
  `tome:begin/end` block to a keyed marker family, so multiple guardrails
  regions coexist with the rules block and a future keyed marker family
  reuses it without touching `rules_file.rs` internals.
- **The 4-reviewer pass caught a real BLOCKER** independently flagged by the
  Rust, test, AND security lenses: verbatim third-party content written
  between managed markers with no validation → region escape / file wedge /
  rules-block corruption. The single-lens happy-path tests would never have
  found it.

## What Didn't Work

- The "verbatim content between managed markers" pattern has an INHERENT
  injection/wedge risk that the first implementation missed entirely — the
  body must be validated for marker-shaped lines at the trust boundary.
- Two contract docs disagreed on the symlink exit code (guardrails.md said 7;
  the authoritative exit-codes-p6.md said 46). Reconciled to 46.

## Workarounds & Solutions

- **Fail closed on marker-shaped verbatim bodies**: since the body is
  contractually verbatim (escaping it is wrong), reject (exit 46, naming the
  source, loud-but-isolated) any `GUARDRAILS.md` body containing a guardrails
  START/END or `tome:begin/end` marker line. Refusal, not escaping.

## Packages & Dependencies

- No new top-level dependencies. Region machinery generalises the existing
  `rules_file.rs` block find/replace to a parameterised marker pair.

## Patterns & Code

- **Validate verbatim third-party content for managed-marker collisions at
  the boundary.** Whenever plugin/frontmatter content is copied verbatim into
  a marker-delimited region of a file Tome re-parses, scan it for lines
  matching ANY managed marker regex and fail closed — otherwise the content
  can escape its region (persistent injection) or wedge the file. General
  companion to the path-segment-validation pattern.
- **Deterministic placement + overwrite-in-place** reconciles "lexicographic
  order" with "never reorder existing content": overwrite surviving regions in
  position, append only NEW regions in lex order (a fresh file lands fully
  ordered; re-syncs never reorder → idempotent).

## For Next Time

- Tell the implementation subagent up front to validate verbatim content for
  marker collisions and to write the atomicity / cross-file-invariant tests —
  the reviewer pass repeatedly discovers these rather than confirming them.
