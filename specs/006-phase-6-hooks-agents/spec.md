# Feature Specification: Phase 6 — Hooks and Agents: Real Hooks, Soft Guardrails, Native Agent Translation, and Personas

**Feature Branch**: `006-phase-6-hooks-agents`
**Created**: 2026-05-28
**Status**: Draft
**Input**: User description: "Phase 6 — real Claude Code hooks plus a GUARDRAILS.md soft fallback, native agent translation across four harnesses, optional agent-as-MCP-prompt personas, and the Phase 4 rules-file correction so Claude Code reads CLAUDE.md, not AGENTS.md. Per PRDs/phase-6.md."

**Codebase Documentation**: See [.sdd/codebase/](../../.sdd/codebase/) for technical details. The eight codebase documents were refreshed at the Phase 5 Polish closeout against the post-v0.5.0 tree and remain current for Phase 6 (this spec introduces no code change).

**Source PRD**: [PRDs/phase-6.md](../../PRDs/phase-6.md) — the implementation-level brief is authoritative for HOW; this spec is authoritative for WHAT.

## Overview

Phase 5 made plugin commands and skills first-class, portable, and slash-invocable across harnesses. Phase 6 closes out the two component types Tome has deferred since the start — **hooks** and **agents** (subagents) — and corrects a latent Phase 4 mistake about where Claude Code reads its rules. Both component types sit at the divergent end of the cross-harness spectrum: neither has a clean universal mapping. Phase 6 is therefore deliberately asymmetric about how far it commits per harness.

Hooks are about enforcement, and enforcement does not port. A real Claude Code hook is a process the harness runs and obeys — a non-zero pre-tool-use exit blocks the tool call. No other harness shares that protocol, and a rules/memory file is explicitly context, not enforced configuration. So Phase 6 makes a clean, honest split. Where real enforcement is available — Claude Code with JSON hooks — Tome installs the real thing: a plugin's `hooks/hooks.json` is rewritten (resolving plugin-root path variables to absolute paths) and merged into the project's local Claude settings file by structural match, idempotently, never deleting a hook it cannot prove it owns. Everywhere else, and for Claude Code plugins that ship no JSON hooks, Tome degrades to **`GUARDRAILS.md`**: a soft, prose-level fallback rendered into each harness's rules file as a Tome-managed, per-plugin region. Guardrails are advisory by construction — the agent can ignore them — which is worse than a real hook and strictly better than nothing. Guardrails never masquerade as enforcement.

Agents are about portability, and agents mostly port. Four harnesses — Claude Code, Codex, Cursor, OpenCode — converged on the same shape: a file with metadata plus a system-prompt body, stored under the harness's agents directory, description-driven delegation, fresh context per spawn. Phase 6 gives agents **full native translation** across those four, emitting each harness's own file format with a best-effort field-and-value mapping that passes through only what the target supports and drops the rest — never inventing cross-vendor model substitutions. For harnesses without native agent support, and (optionally) everywhere, agents can also be exposed as **MCP-prompt "personas"**: a global, off-by-default fallback that lets a user assume and drop an agent's persona through slash commands, reusing Phase 5's prompt and substitution machinery.

Phase 6 also corrects Phase 4: Claude Code does not natively read `AGENTS.md`, so Tome's rules-include block and guardrails for Claude Code must target `CLAUDE.md`. Codex, Gemini, and OpenCode keep sharing one `AGENTS.md`; both pointers resolve the same `.tome/RULES.md`, so there is no content duplication and no dependence on Claude Code ever shipping `AGENTS.md` support.

After Phase 6, enabling a plugin in a workspace reconciles, per harness in the effective list: real hooks (Claude Code), guardrails regions (all harnesses), and native agent files (the four). The MCP server optionally exposes agent personas as prompts alongside Phase 5's command and skill prompts. Tome never executes a hook command or an agent body itself — it rewrites and places files; the harness owns runtime behaviour.

What this phase explicitly does not do: real hooks for any harness other than Claude Code (Codex and Gemini have real hook systems but are signposted for a later phase); native agent files for Gemini/Antigravity or the Phase 2 harnesses (Gemini gets guardrails and optional personas; Antigravity's agent schema is undocumented); emulation of Claude Code-only agent semantics (agent teams, worktree isolation, persistent agent memory, auto-submission, effort) — unmappable fields are dropped; semantic search over agents (agents are indexed for provenance and collision detection only, never surfaced through the search tool in this version); server-side shell execution of the harness-native backtick syntax (still deferred); hook-authoring or scaffolding tooling; and any new harnesses beyond Phase 4's five.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Use a plugin's agents natively in any supported harness (Priority: P1)

A developer enables a plugin that ships one or more agents (for example, a `reviewer` subagent) in a workspace bound to a project. They work in any of the four natively-supported harnesses. They want the plugin's agents to appear in that harness exactly as if the plugin had been authored for it — in the harness's own agent file format, with the fields the harness understands carried across and the ones it cannot understand quietly dropped, never silently mis-translated.

**Why this priority**: Agents are the larger portable win of Phase 6 — four harnesses already converged on the same agent shape, so this is where Tome delivers the most cross-harness value per unit of effort. It is independently demonstrable end-to-end and is the slice most plugin authors and users will feel first.

**Independent Test**: From a fresh install, the developer enrols a catalog containing a plugin with an `agents/` directory, binds a project to a workspace whose effective harness list includes all four natively-supported harnesses, and enables the plugin. Each harness's agent directory then contains the plugin's agent emitted in that harness's format, namespaced by plugin in the filename, with the body placed where that harness expects it (a dedicated instructions field for the TOML harness, the file body for the others) and harness-default behaviour applied where the source omitted a field. A model value that has a same-vendor target in a harness is mapped; a model value with no sane same-vendor target is dropped; an unsupported field is dropped and logged. Disabling the plugin removes exactly that plugin's agent files from every harness's agent directory and leaves other plugins' agents in place. Re-running sync changes nothing.

**Acceptance Scenarios**:

1. **Given** an enabled plugin shipping an agent file, **When** the workspace's effective harness list includes each of the four natively-supported harnesses, **Then** that agent is emitted into each harness's agent directory in that harness's format, with the system-prompt body placed in the harness-appropriate location and a plugin-namespaced filename.
2. **Given** a translated agent whose source omits the harness's required descriptor, **When** the agent is emitted for a harness that requires a description, **Then** Tome supplies a documented fallback (the first line of the body) and records the fallback for diagnostics.
3. **Given** an agent declaring a model value, **When** it is translated for a harness with a legitimate same-vendor target for that value, **Then** the value is mapped to that target; **When** translated for a harness whose only candidates are different-vendor identifiers, **Then** the field is dropped and the harness's default is inherited; in no case is a value mapped across vendors.
4. **Given** an agent declaring a field the target harness does not support and that does not map cleanly to one it does, **When** the agent is translated, **Then** the field is dropped from the emitted file and the drop is recorded for diagnostics.
5. **Given** two enabled plugins each shipping an agent with the same source name, **When** both are emitted, **Then** each agent's file is namespaced by its plugin and both are present; the harness-facing displayed name is plugin-prefixed for the clashing agents only, while non-clashing agents keep their clean name.
6. **Given** an enabled plugin is disabled, or its harness leaves the effective list, **When** sync reconciles, **Then** that plugin's namespaced agent files are removed from the affected harness's agent directory and no other plugin's agent files are touched.
7. **Given** a workspace already reconciled, **When** sync runs again with no underlying change, **Then** no agent file is rewritten or removed (idempotence).

### User Story 2 — Real hooks for a Claude Code plugin (Priority: P2)

A developer enables a plugin that ships real JSON hooks in a workspace whose effective harness list includes Claude Code. They want those hooks to actually run and actually enforce — a pre-tool-use guard that blocks a dangerous tool call must block it — exactly as they would if the plugin had been installed through Claude Code's native plugin manager, with the plugin's bundled scripts resolved to the right place on this machine.

**Why this priority**: Real enforcement is the one capability nothing else in Phase 6 can substitute for, and it is only available on Claude Code. Delivering it where it is possible is the honest core of the hooks half of the phase, and it is self-contained and independently testable. It ranks below agents because it touches a single harness, where agents touch four.

**Independent Test**: From an install with a Claude Code project, the developer enables a plugin whose `hooks/hooks.json` references the plugin-root path variable. The project's local Claude settings file then contains the plugin's hook with the plugin-root variable rewritten to the absolute installed-plugin path while other Claude-native variables are left untouched for the harness to resolve at runtime. Re-running sync adds no duplicate. A hook the user had already written by hand identical to the plugin's hook is not duplicated. Disabling the plugin removes exactly the plugin's hook entries by structural match and leaves a user-edited copy untouched. The local (gitignored) settings file is used, never the team-shared committed one.

**Acceptance Scenarios**:

1. **Given** a plugin shipping `hooks/hooks.json` that references the plugin-root path variable (and, where present, the plugin-data variable), **When** the plugin is enabled in a Claude Code project, **Then** the project's local Claude settings file contains the hook with those two variables rewritten to their absolute installed paths and every other Claude-native variable left verbatim.
2. **Given** a plugin's hook has been merged, **When** sync runs again, **Then** no structurally identical hook entry is appended a second time under the same event.
3. **Given** the user has hand-authored a hook entry structurally identical to one the plugin ships, **When** the plugin is enabled, **Then** the plugin's hook is not appended as a duplicate.
4. **Given** an enabled plugin is disabled, or Claude Code leaves the effective list, or the plugin leaves the workspace, **When** sync reconciles, **Then** exactly the plugin's structurally-matching hook entries are removed; any entry that does not exactly match a re-derived plugin entry — including one the user edited after Tome wrote it — is left in place.
5. **Given** the project has no local Claude settings file, **When** a plugin's hooks are merged, **Then** Tome creates the local settings file with a single hooks object; the team-shared committed settings file is never written.
6. **Given** removing a plugin's hooks empties an event's hook array, **When** sync reconciles, **Then** the empty event array is pruned while an otherwise-empty hooks object is left in place.

### User Story 3 — Soft guardrails everywhere a real hook can't run (Priority: P3)

A plugin author ships a `GUARDRAILS.md` describing pre/post-action constraints in prose. A developer enables that plugin across a mix of harnesses. They want each harness to surface the plugin's guardrails in the rules file that harness actually reads, as a clearly Tome-managed and per-plugin region — and, on Claude Code specifically, they want real JSON hooks to take over and suppress the prose fallback when the plugin ships both.

**Why this priority**: Guardrails are the honest degradation path that makes Phase 6's hook story cross-harness at all; without them, only Claude Code users with JSON-hook-shipping plugins get any pre/post-action guidance. They depend on the Phase 4 rules-file correction (so Claude Code's guardrails actually land in a file Claude Code reads), which makes this the natural home for that correction.

**Independent Test**: A plugin shipping `GUARDRAILS.md` and no JSON hooks is enabled across all five harnesses. A clearly-delimited, provenance-tagged per-plugin region appears in each harness's rules-file target — `CLAUDE.md` for Claude Code, the shared `AGENTS.md` for the AGENTS-based harnesses, and a Tome-owned sibling file for the standalone-rules harness. A second plugin shipping guardrails produces a second distinct region in the shared file. A plugin shipping both guardrails and JSON hooks has its guardrails region present in the shared `AGENTS.md` for the other harnesses and absent from `CLAUDE.md` (suppressed by the real hooks). Disabling a plugin removes only its region. Re-syncing overwrites the content between a plugin's existing markers in place. Separately, a Claude Code project receives its Phase 4 rules-include block in `CLAUDE.md`, not `AGENTS.md`, while a project shared with an AGENTS-based harness still has one `AGENTS.md` block, both resolving the same underlying rules file.

**Acceptance Scenarios**:

1. **Given** a plugin shipping `GUARDRAILS.md` and no JSON hooks enabled across all five harnesses, **When** sync reconciles, **Then** a per-plugin guardrails region delimited by provenance-carrying markers (naming the catalog and plugin) appears in each harness's resolved rules-file target, with the plugin's guardrails content copied verbatim between the markers.
2. **Given** a plugin shipping both `GUARDRAILS.md` and JSON hooks enabled in a project whose harnesses include Claude Code and at least one AGENTS-based harness, **When** sync reconciles, **Then** the plugin's guardrails region is present in the shared `AGENTS.md` and absent from `CLAUDE.md`, with no contradiction because the two files are distinct.
3. **Given** two enabled plugins each shipping `GUARDRAILS.md`, **When** they render into the same shared rules file, **Then** the file contains two distinct per-plugin regions, each independently delimited.
4. **Given** an enabled plugin with a guardrails region is disabled, or its harness leaves the effective list, or (on Claude Code) it begins shipping JSON hooks, **When** sync reconciles, **Then** that plugin's region is removed from the affected file and other plugins' regions remain.
5. **Given** a plugin's guardrails region already exists, **When** sync runs again, **Then** the content between the existing markers is overwritten in place rather than duplicated, and the operation is idempotent when the source content is unchanged.
6. **Given** the standalone-rules harness has no enabled plugin contributing guardrails, **When** sync reconciles, **Then** the Tome-owned guardrails sibling file is deleted entirely.
7. **Given** a Claude Code project under Tome management, **When** the rules integration is applied, **Then** the Phase 4 rules-include block targets `CLAUDE.md` (not `AGENTS.md`); a project also shared with an AGENTS-based harness keeps a single `AGENTS.md` block for those harnesses; both include directives resolve the same underlying `.tome/RULES.md` with no duplicated content.

### User Story 4 — Assume an agent's persona in any connected client (Priority: P4)

A user works in a harness with no native agent support, or simply wants a plugin agent's behaviour as a slash command. They enable a global, off-by-default option that exposes each enabled agent as a "persona" prompt. They invoke the persona by slash command with a free-form instruction, the agent's role is assumed for the conversation, and a single global prompt drops any assumed persona.

**Why this priority**: Personas extend the agent story to harnesses the native translation cannot reach, but they are an advisory approximation (conversational context, not isolation or tool scoping) and are off by default. They build directly on Phase 5's prompt and substitution machinery, so they sequence naturally after the native agent work.

**Independent Test**: With the persona option off (the default), no persona prompt appears in the MCP prompt list. With the option enabled globally, each enabled agent appears as a `<name>-persona` prompt (clash-prefixed by plugin when two plugins' agents share a name) plus a single global drop-persona prompt. Invoking a persona prompt returns the agent's body with its frontmatter stripped, wrapped in a documented role-assumption template, with Phase 5 built-in and environment substitution applied and a free-form argument resolved through the same Phase 5 argument pipeline (single catch-all argument plus the documented append fallback). The persona name is taken from the agent's frontmatter name when present, else the agent filename stem.

**Acceptance Scenarios**:

1. **Given** the persona option is off (default), **When** the MCP prompt list is enumerated, **Then** no persona prompts and no drop-persona prompt appear.
2. **Given** the persona option is enabled, **When** the prompt list is enumerated, **Then** each enabled agent appears once as a `<name>-persona` prompt and a single global, unnamespaced drop-persona prompt is present exactly once regardless of agent count.
3. **Given** two enabled plugins' agents share a source name with the persona option on, **When** their persona prompts are named, **Then** only the clashing personas are plugin-prefixed (`<plugin>-<name>-persona`); non-clashing personas keep the clean form.
4. **Given** a persona prompt is fetched, **When** the body is rendered, **Then** the agent's frontmatter is stripped, the body is wrapped in the documented role-assumption template, Phase 5 built-in and environment substitution is applied to the body, and a caller-supplied free-form instruction is resolved through the Phase 5 argument pipeline.
5. **Given** an agent whose frontmatter has no name, **When** its persona name is resolved, **Then** the agent's filename stem is used, subject to the Phase 5 sanitisation, length, and collision rules.
6. **Given** the persona option is enabled in a workspace that also gets native agent files, **When** clients connect, **Then** personas are exposed to all connected clients (the redundancy is harmless and is why the option is global and off by default).

### User Story 5 — Audit and govern plugin-agent privilege escalation (Priority: P5)

A cautious developer or org administrator wants visibility into, and control over, the fact that installing a plugin's agents through Tome can grant those agents privileged frontmatter (hooks, MCP-server declarations, permission-mode) that Claude Code's native plugin manager would refuse. They want a report of every such agent and an opt-in setting that restores native-plugin-manager parity by stripping those privileged fields from emitted Claude Code agents.

**Why this priority**: Passing privileged fields through by default is a genuine capability advantage of installing via Tome, but it removes a guardrail the native plugin manager enforces. Making that trade-off auditable and reversible is a governance concern that matters for security-conscious users; it is lower priority than the core capabilities it governs and is cleanly separable.

**Independent Test**: A plugin agent declaring one or more privileged frontmatter fields is enabled in a Claude Code project. By default it is emitted to Claude Code's agent directory with those fields intact, and the diagnostic command lists that agent under a privilege-escalation report grouped by plugin. With the strip option set (at workspace or global scope), the same agent is emitted to Claude Code's agent directory without those three fields, and the change requires no plugin edit.

**Acceptance Scenarios**:

1. **Given** a plugin agent declaring privileged frontmatter fields and the strip option off (default), **When** the agent is emitted for Claude Code, **Then** those fields are emitted intact.
2. **Given** such an agent has been emitted, **When** the diagnostic command runs, **Then** the agent appears in a privilege-escalation report grouped by plugin so the escalation surface is auditable.
3. **Given** the strip option is set at workspace or global scope, **When** the same agent is emitted for Claude Code, **Then** the privileged fields are stripped from the emitted file; the setting follows the same project → workspace → global layering as the harness list so it can be enforced org-wide.
4. **Given** the strip option is set, **When** an agent without any privileged fields is emitted, **Then** its emission is unchanged (the strip is a no-op for non-privileged agents).

### Edge Cases

- A plugin ships `hooks/hooks.json` that is malformed or unparsable. The failure surfaces with a dedicated error code naming the file; sibling components of the same plugin still reconcile where possible, consistent with loud-but-isolated parse-failure handling.
- A plugin ships both `hooks/hooks.json` and `hooks/GUARDRAILS.md`. On Claude Code the JSON hooks win and the guardrails region is not rendered into `CLAUDE.md`; on every other harness the guardrails region is rendered normally. Per-event merging of "JSON for some events, prose for others" is not attempted.
- A user hand-edits a Tome-written hook entry in the local Claude settings file. On removal, the edited entry no longer matches a re-derived plugin entry and is left in place; Tome never deletes a hook it cannot prove it owns.
- An agent's frontmatter is malformed YAML, or translation fails for a target harness. The failure surfaces with a dedicated error code naming the agent; the agent is not partially emitted.
- An agent declares a model value that maps cleanly for one harness and has no sane target for another. The value is mapped where a same-vendor target exists and the field is dropped (default inherited) where it does not; no cross-vendor mapping is ever performed.
- The standalone-rules harness (Cursor) hosts all enabled plugins' guardrails in one Tome-owned sibling file; each plugin is still individually marker-wrapped so per-plugin removal works, and the sibling file is deleted when no plugin contributes to it.
- The filename-derived-name harness (OpenCode) cannot hide the plugin prefix; its agents are always named with the plugin prefix. This is an accepted wart of that harness's model.
- A user-authored agent in a harness's agent directory happens to use the same plugin-prefixed filename convention Tome uses. Removal globs the plugin-prefixed pattern; the probability of trampling such a file is accepted as lower-risk than the alternative of a provenance frontmatter key that could break a harness's parser.
- A plugin ships an agent but the workspace's effective harness list includes only harnesses without native agent support. No native agent files are written; the agent is still indexed for provenance and collision detection, and is exposed as a persona only if the persona option is enabled.
- A guardrails render or write fails for a rules file or the standalone sibling. The failure surfaces with a dedicated error code; partially-written guardrails state is not left between markers.
- A workspace's enabled plugins include agents whose privileged fields are present but the target harness is not Claude Code. The privilege-strip setting and the privilege report concern Claude Code emission specifically; other harnesses' agent dialects do not carry those three fields in the first place.
- A Claude Code project has an `AGENTS.md` (shared with AGENTS-based harnesses) but no `CLAUDE.md`. Under Phase 6 the rules-include block and any guardrails for Claude Code are written to `CLAUDE.md` so Claude Code actually reads them; the courtesy of scaffolding a `CLAUDE.md` that imports a user's hand-written `AGENTS.md` is out of scope.

## Requirements *(mandatory)*

### Functional Requirements

#### Real hooks (Claude Code only)

- **FR-001**: Tome MUST read a plugin's `hooks/hooks.json` on plugin enable and on harness sync for any project whose effective harness list includes Claude Code; a plugin shipping no `hooks/hooks.json` MUST fall through to the guardrails path for Claude Code with no error.
- **FR-002**: Tome MUST merge rewritten plugin hooks into the project's local (gitignored) Claude settings file under the hooks key; Tome MUST NOT write the team-shared committed Claude settings file for this purpose. If the local settings file is absent, Tome MUST create it with a single hooks object.
- **FR-003**: At copy time Tome MUST rewrite, within string values of the hook JSON, the plugin-root path variable to the absolute installed-plugin root path and the plugin-data path variable to the absolute plugin persistent-data path (the same values Phase 5 exposes as the corresponding built-ins). All other Claude-native variables MUST be left verbatim for the harness to resolve at runtime. The rewrite MUST be a targeted textual substitution of exactly those two variables, not the full Phase 5 substitution pipeline.
- **FR-004**: On add, for each rewritten plugin hook entry keyed by event, Tome MUST append the entry under that event only if no structurally identical entry (deep equality of the post-rewrite entry object) already exists under that event; an existing structurally identical entry MUST be skipped. The operation MUST be idempotent across repeated syncs and MUST NOT duplicate a hook the user already authored by hand.
- **FR-005**: On disable, when Claude Code leaves the effective list, or when the plugin leaves the workspace, Tome MUST re-derive the plugin's rewritten hook entries and remove each structurally identical entry under its event; an entry with no exact structural match MUST be skipped (never removed). Tome MUST NOT use a sidecar or provenance marker for hooks — ownership is established solely by re-deriving and structurally matching.
- **FR-006**: After removal, Tome MUST prune empty hook event arrays; an otherwise-empty hooks object MUST be left in place.

#### Guardrails soft fallback

- **FR-010**: Tome MUST treat a plugin's optional `hooks/GUARDRAILS.md` as plain Markdown copied verbatim into the target; Tome MUST NOT parse or interpret its content.
- **FR-011**: For each enabled plugin shipping `GUARDRAILS.md`, Tome MUST render its content into a per-plugin, marker-delimited region in the rules-file target for each harness in the effective list, using provenance markers that name the catalog and plugin. The regions MUST be placed adjacent to — not inside — the Phase 4 rules-include block, in a deterministic order (the Phase 4 include block first, then one guardrails region per contributing plugin in lexicographic `<catalog>:<plugin>` order) so that re-syncs never reorder existing content.
- **FR-011a**: Guardrails regions MUST use a literal marker pair distinct from the Phase 4 `tome:begin`/`tome:end` rules-block markers, carrying the catalog and plugin in the marker text (the PRD form `<!-- START GUARDRAILS: <catalog>:<plugin> -->` … `<!-- END GUARDRAILS: <catalog>:<plugin> -->`); the exact literal MUST be pinned in the contracts and is the sole per-plugin removal key.
- **FR-012**: Guardrails targets MUST follow the Phase 4 harness modules: Claude Code → `CLAUDE.md`; Codex → `AGENTS.md`; Gemini → `AGENTS.md` (preferred) else `GEMINI.md`; OpenCode → `AGENTS.md`; Cursor → a Tome-owned standalone sibling file under the Cursor rules directory, distinct from the Phase 4 skills sibling, with each plugin still individually marker-wrapped.
- **FR-013**: For Claude Code specifically, if a plugin ships any `hooks/hooks.json`, Tome MUST NOT render that plugin's guardrails region into `CLAUDE.md` (the real hooks supersede the prose fallback). This suppression MUST apply only to Claude Code's own file; the same plugin's guardrails MAY be present in the shared `AGENTS.md` for other harnesses simultaneously. Per-event merging of JSON-for-some/prose-for-others MUST NOT be attempted.
- **FR-014**: Guardrails reconciliation MUST be per target file: for each target derived from the effective harness list, Tome MUST ensure exactly one region per enabled-plugin-with-guardrails (minus any plugin suppressed for that file's harness) with current content, overwriting the content between existing markers in place; and MUST remove any region whose marker corresponds to a plugin no longer enabled, a harness no longer in the effective list, or (on Claude Code) a plugin now shipping JSON hooks.
- **FR-015**: The Cursor guardrails sibling file MUST be deleted entirely when no enabled plugin contributes a guardrails region to it. Guardrails state MUST be filesystem-inferred from the marker pairs with no sidecar.
- **FR-016**: Within a single harness sync, the hooks-presence determination that drives the Claude Code guardrails suppression predicate (FR-013) MUST be computed before guardrails are reconciled for `CLAUDE.md`, so the predicate never reads stale state. The reconciliation MUST handle both suppression transitions symmetrically: a plugin that begins shipping `hooks/hooks.json` between syncs MUST have its `CLAUDE.md` guardrails region removed (FR-014) while its hooks are merged (FR-004); a plugin that ceases to ship `hooks/hooks.json` between syncs MUST have its hook entries removed (FR-005) and its `CLAUDE.md` guardrails region (re-)rendered, in the same sync.

#### Phase 4 rules-file correction

- **FR-020**: Claude Code's harness module MUST target `CLAUDE.md` (not `AGENTS.md`) for both the Phase 4 rules-include block and Phase 6 guardrails; the Claude Code include directive MUST point at the same `.tome/RULES.md` the AGENTS-based harnesses point at.
- **FR-021**: Codex, Gemini, and OpenCode MUST continue to share a single `AGENTS.md` rules-include block; the Claude Code `CLAUDE.md` block and the shared `AGENTS.md` block MUST resolve the same underlying rules file with no duplicated rules content.
- **FR-022**: Claude Code's rules-file candidate precedence MUST become `CLAUDE.md` > `.claude/CLAUDE.md` (first existing wins; create `CLAUDE.md` when none exist). `AGENTS.md` MUST NOT appear in Claude Code's candidate set under any precedence — this is the substance of the Phase 4 correction, and it changes the existing `claude_code` rules-file candidate array, which today lists `AGENTS.md` first. The existing-rules-file precedence continues to apply only within this corrected Claude-Code-own candidate set (an existing `CLAUDE.md` is reused and its block updated in place). Tome MUST NOT introduce a transitive Claude-Code-imports-AGENTS chain and MUST NOT depend on Claude Code ever shipping native `AGENTS.md` support. Scaffolding a `CLAUDE.md` that imports a user's hand-written `AGENTS.md` is out of scope.

#### Native agent translation

- **FR-030**: Tome MUST emit native agent files for Claude Code, Codex, Cursor, and OpenCode from a plugin's `agents/*.md` source files (Markdown plus YAML frontmatter). Each natively-supported harness module MUST declare its agent directory, its agent file format, and a translation from the canonical (Claude Code) field set to the harness's dialect.
- **FR-031**: Agent files MUST be written at project scope into each harness's agent directory (Claude Code `.claude/agents/`, Codex `.codex/agents/`, Cursor `.cursor/agents/`, OpenCode `.opencode/agent/` — note OpenCode's directory is the singular `agent/`; all directory paths are verified against current harness documentation at implementation time, per the Phase 4 caveat). Tome MUST emit native files even for harnesses that cross-read Claude Code's agent directory, rather than relying on the cross-read.
- **FR-032**: Translation MUST pass through only the frontmatter fields the target harness supports or that map cleanly to a field it supports, and MUST drop every other field; no field may be passed through on the assumption the harness tolerates unknown keys. Dropped fields MUST be recorded for diagnostics.
- **FR-033**: The system-prompt body MUST be placed where each harness expects it: in the file body for the Markdown-plus-YAML harnesses, and in the harness's dedicated developer-instructions field (as a TOML triple-quoted string) for the TOML harness.
- **FR-034**: Field value mapping MUST be same-vendor only. A model value with a legitimate same-vendor target in a harness MUST be mapped to that target; a model value with no sane same-vendor target MUST be dropped (harness default inherited); an inherit-style value MUST be dropped everywhere. Cross-vendor or strongest-to-strongest heuristics MUST NOT be used. Dropped model fields MUST be recorded for diagnostics.
- **FR-035**: For the OpenCode harness, a translated agent MUST default to the subagent mode, and when the source lacks a description (which OpenCode requires) Tome MUST fall back to the first non-empty line of the body, trimmed, and record the fallback for diagnostics; when the body has no non-empty line, a documented placeholder description MUST be used.
- **FR-036**: Read-only intent MUST be reconstructed per harness from the source agent's canonical tool posture (its tool allowlist and/or disallowed-tools fields) using a documented inference rule — for example, the absence of any write/edit/execute-class tool from the allowlist denotes read-only — expressed in each harness's own mechanism (sandbox mode, a read-only flag, or per-tool permission entries). Where the rule is indeterminate or the harness cannot express the intent, the field MUST be dropped (harness default inherited) and the drop recorded for diagnostics.
- **FR-037**: Field value mapping for `model` (FR-034) MUST be driven by a per-harness, same-vendor model-alias table (source canonical value → harness-native identifier) declared as a single source of truth in the harness modules and pinned in the contracts; a source value with no entry for a given harness MUST be dropped (default inherited). This table is the named artefact against which the no-cross-vendor success criterion is verified. Exact identifiers are confirmed against current harness documentation at implementation time (Phase 4 caveat).

#### Agent file naming, provenance, and removal

- **FR-040**: Emitted agent filenames MUST always be namespaced as `<plugin>__<name>.<ext>`; the namespaced filename MUST be the sole provenance mechanism. Tome MUST NOT add any provenance frontmatter key.
- **FR-041**: The harness-facing displayed/registered agent name MUST use the clean `<name>` normally, and the plugin-prefixed `<plugin>-<name>` form only when two or more enabled plugins in the workspace clash on `<name>`, applied to the clashing agents only; the filename MUST remain `<plugin>__<name>` regardless of clash.
- **FR-042**: For the filename-derived-name harness (OpenCode), the agent's harness-facing name is necessarily the plugin-prefixed filename stem; this is an accepted constraint of that harness.
- **FR-043**: Agent removal MUST be reconciled on sync and on disable by globbing `<plugin>__*.<ext>` in each natively-supporting harness's agent directory and removing files for plugins no longer enabled or harnesses no longer in the effective list, leaving other plugins' agents untouched.

#### Plugin-agent privilege handling

- **FR-050**: Tome MUST, by default, pass the privileged frontmatter fields (the hooks, MCP-server-declaration, and permission-mode fields that Claude Code's native plugin manager forbids for plugin-shipped subagents) through to emitted Claude Code agent files.
- **FR-051**: The diagnostic command MUST report every installed agent carrying any of those privileged fields, grouped by plugin, so the escalation surface is auditable.
- **FR-052**: Tome MUST provide a configuration setting that, when set, strips those three privileged fields from emitted Claude Code agent files (restoring native-plugin-manager parity). The setting MUST default to off and MUST follow the same project → workspace → global scope layering as the harness list, so it can be enforced org-wide. The setting MUST be a no-op for agents that carry none of the privileged fields.
- **FR-053**: Both new boolean settings (`strip_plugin_agent_privileges` and the persona-exposure toggle of FR-060) MUST be added as typed `bool` fields, defaulting to `false`, on the Tome-owned settings structs (the global, workspace, and project-marker forms), which sit on the strict side of the strictness boundary. Their layering MUST be a first-declarer-wins priority walk across project → workspace → global (the nearest scope that declares the field wins; a project `false` overrides a global `true`). The composition reference/exclusion grammar that applies to the `harnesses` list MUST NOT apply to these scalar booleans.

#### Agent personas (MCP-prompt fallback)

- **FR-060**: Tome MUST provide a configuration setting that, when enabled, exposes each enabled agent as an MCP-prompt persona plus a single global drop-persona prompt. The setting MUST default to off and MUST follow the same layering as the harness list. Because MCP prompts are not harness-scoped, enabling it MUST expose personas to all connected clients.
- **FR-061**: A persona prompt's name MUST be `<name>-persona` normally and `<plugin>-<name>-persona` only for agents whose `<name>` clashes across two or more enabled plugins, applied to the clashing agents only. The persona `<name>` MUST be taken from the agent's frontmatter name field (read before frontmatter stripping) and MUST fall back to the agent filename stem when absent. Persona names MUST be subject to the Phase 5 sanitisation, length-limit, and collision-counter rules as a final backstop.
- **FR-062**: A persona prompt body MUST strip the agent's frontmatter, wrap the body in the documented role-assumption template, apply Phase 5 built-in and environment substitution to the body, and resolve a single catch-all free-form argument through the exact Phase 5 argument pipeline (including the documented append fallback when the template consumes no argument reference). No parallel substitution path may be introduced.
- **FR-063**: When the persona setting is enabled, Tome MUST expose exactly one global, unnamespaced drop-persona prompt whose body instructs the agent to stop acting as any assumed persona and return to default behaviour. The drop-persona name MUST be reserved: if any command, skill, or persona would derive to the same name, the other entry MUST be counter-suffixed and drop-persona MUST remain unnamespaced and unique.
- **FR-064**: Persona prompts MUST be emitted by the MCP server's prompt-list and prompt-get layer from agent rows as a specialised path (template-wrapped body, persona naming) parallel to — not folded into — Phase 5's command and skill prompts.
- **FR-065**: User-facing documentation MUST state plainly that a persona is advisory conversational context, not the isolation or tool-scoping a native subagent provides; personas MUST NOT be presented as enforced configuration.
- **FR-066**: Persona prompt names MUST participate in the same single prompt-name collision namespace as Phase 5's command and skill prompts; the collision-resolution pass MUST run over the union of command, skill, and persona derived names. The agent-clash plugin prefix (FR-061) MUST be applied before the Phase 5 counter-suffix backstop, so the backstop only ever fires on residual collisions the prefix did not resolve.
- **FR-067**: The persona exposure toggle MUST be resolved against the MCP server's single startup scope (the resolved workspace, or global fallback) — the same scope the server binds its prompt registry to — because the running MCP server is not project-bound. Project-scope layering of `expose_agents_as_personas` therefore has no effect on a running server; this MUST be documented. Where FR-060 states the setting follows "the same layering as the harness list," that layering governs the on-disk resolution; the persona toggle's effective value for a session is whatever the server's startup scope resolves.

#### Agent indexing and the data model

- **FR-070**: Tome MUST index a plugin's `agents/*.md` files into the existing entry table with the entry-kind set to the agent value (a third permitted value of the Phase 5 kind discriminator); agent rows MUST always be non-searchable (excluded from the agent search tool in this version). The plugin scan MUST walk the agents directory alongside the Phase 5 skills and commands directories.
- **FR-070a**: The Rust `EntryKind` enum MUST be widened to include the agent variant, and every exhaustive match over it — including the per-kind count aggregation behind `tome plugin list`/`tome plugin show` and the doctor entry-count surface — MUST be updated to account for agent-kind rows (counted separately or explicitly excluded). No `EntryKind` match may regress to a catch-all that re-hides schema drift; the canonical-enum-dispatch discipline established in Phase 5 MUST be preserved. This widening is load-bearing: a `kind='agent'` row introduced without it would make those surfaces fail with the index-integrity error code.
- **FR-071**: Agent indexing MUST require no new columns and no new tables: agent rows MUST reuse existing columns and the existing (catalog, plugin, kind, name) uniqueness constraint. The change is confined at the storage layer to widening the `kind` column's permitted domain and, at the Rust layer, to widening the `EntryKind` enum and its exhaustive matches (FR-070a). Agent rows MUST carry the agent name (frontmatter name, else filename stem), description, and content hash; embedding MUST be skipped for agents.
- **FR-072**: Agent indexing MUST exist for provenance/enabled-state tracking and for cheap cross-plugin name-collision detection. The clash set MUST be defined as the set of `<name>` values held by two or more agent-kind rows enabled in the resolved workspace (joined through the workspace-enablement junction), computed once per sync from the workspace scope. The same clash set MUST govern filename display-name prefixing (FR-041), harness-facing displayed-name prefixing (FR-041), and persona naming (FR-061) identically across all of the workspace's bound projects.
- **FR-073**: Hooks MUST NOT be indexed; hook and guardrails files MUST be read during harness sync (not during indexing), consistent with the filesystem-inferred, no-sidecar model. A schema-version bump MUST occur only if the migration registry requires a marker for the widened kind domain; no data migration is required for the free-text kind column.

#### Sync, CLI surface, and reconciliation

- **FR-080**: Phase 6 MUST add no new top-level CLI commands; the Phase 1–5 command surface MUST be unchanged.
- **FR-081**: Harness sync MUST additionally reconcile, per harness in the effective list, real hooks (Claude Code), guardrails regions, and native agent files; sync MUST be idempotent across all three (a second run with no underlying change MUST change nothing).
- **FR-082**: Plugin enable and plugin disable MUST trigger the same reconciliation for the affected plugin across the workspace's bound projects' effective harnesses.
- **FR-083**: The plugin-show command MUST list a plugin's agents, whether the plugin ships JSON hooks and/or a `GUARDRAILS.md`, and (for agents, when the persona setting is on) the resolved persona name.
- **FR-084**: A failure reconciling one sink (hooks, guardrails, or agents) for one harness MUST NOT roll back sinks already reconciled successfully in the same sync; reconciliation MUST continue across the remaining harnesses and sinks where possible and surface the first failure's exit code (the forward-progress discipline Phase 4 established for binding-then-sync). Each individual file write MUST remain all-or-nothing (atomic), so a failure never leaves a partially-written file or a partially-written marker region.

#### Diagnostics (doctor) and errors

- **FR-090**: The diagnostic command MUST report: for Claude Code hooks, which enabled plugins contributed hooks to the local settings file and any plugin-derived hook entries Tome expected but could not find (drift from user edits); for guardrails, per target file the present per-plugin regions, orphaned regions, and regions suppressed by JSON hooks on Claude Code; for agents, per harness the present and orphaned namespaced agent files, fields dropped during translation (informational), and the privilege-escalation report grouped by plugin; and, when personas are enabled, the effective persona prompt list with resolved and clash-prefixed names.
- **FR-091**: The diagnostic command's repair mode MUST repair only the safe, derivable cases — re-render stale guardrails regions, re-emit missing agent files, and remove orphaned namespaced agent files — and MUST NOT remove a hook entry that does not exactly match a re-derived plugin entry, nor delete any user-authored content.
- **FR-092**: Each new failure mode introduced in Phase 6 MUST either reuse an existing closed-set error variant or claim a dedicated new exit code: a malformed/unparsable plugin `hooks/hooks.json`; a failure to read/merge/write the local Claude settings file; a malformed agent frontmatter or failed agent translation; and a failed guardrails render/write. The numeric exit codes MUST be assigned in the contracts phase as a single free contiguous run of four, pairwise-unique, reusing none of the codes already in the closed enum (the PRD-proposed 30–33 collide with the model-on-disk cluster and 34–37 are also already taken by the inference/vector cluster). The chosen run MUST be recorded in `contracts/exit-codes-p6.md` and pinned by the exit-code test, following the same reassignment precedent used for the Phase 4 summariser code and the Phase 5 cluster.

### Non-Functional Requirements

- **NFR-001**: Hook merge and removal MUST be deterministic and idempotent: given the same plugin `hooks.json`, the same installed-plugin paths, and the same existing local settings file, repeated reconciliation MUST converge to the same file content and MUST NOT append or remove anything on a no-change second run.
- **NFR-002**: Tome MUST NOT execute any hook command or any agent body at any point; Phase 6 only reads, rewrites, and places files, and the host harness owns all runtime behaviour.
- **NFR-003**: Hook ownership MUST be established purely by re-deriving and structurally matching what Tome would have written — no sidecar, no provenance marker — so that a user-edited hook is conservatively left in place on removal.
- **NFR-004**: Guardrails and agent reconciliation state MUST be filesystem-inferred (guardrails via marker pairs, agents via the namespaced filename glob) with no sidecar, consistent with Phase 4's state model.
- **NFR-005**: Agent translation MUST never map a model value across vendors and MUST prefer dropping a field (inheriting the harness default) over emitting a value the target harness cannot interpret; every dropped field MUST be observable via diagnostics.
- **NFR-006**: The privilege pass-through MUST be visible and reversible: the escalation surface MUST be auditable through the diagnostic command and the strip behaviour MUST be available through the layered configuration without editing any plugin.
- **NFR-007**: The persona surface MUST reuse the Phase 5 prompt and substitution machinery without a parallel substitution implementation; caller-supplied persona arguments MUST be treated as opaque strings exactly as Phase 5 treats prompt arguments.
- **NFR-008**: Personas MUST be off by default; a workspace whose configuration does not enable them MUST behave exactly as it did in Phase 5 with respect to the prompt surface.
- **NFR-009**: The Claude Code rules-file correction MUST NOT duplicate rules content across `CLAUDE.md` and `AGENTS.md`; both include directives MUST resolve the same underlying rules file.
- **NFR-010**: All new declarative inputs Tome owns MUST follow the existing strictness boundary; third-party plugin inputs (agent frontmatter, `hooks.json`, `GUARDRAILS.md`) MUST be read leniently, consistent with the Phase 2–5 plugin-frontmatter boundary, while failing loudly on malformed recognised structures.
- **NFR-011**: All new MCP response and diagnostic record shapes introduced in Phase 6 (the persona prompt entries, the hooks/guardrails/agents/persona doctor records) MUST be pinned by byte-stable serialisation tests across the human and JSON output paths where applicable, consistent with the Phase 3–5 wire-shape pinning discipline.

### Key Entities *(include if feature involves data)*

- **Hook set** (concept): A plugin's `hooks/hooks.json` after plugin-root/plugin-data variable rewriting, keyed by event. Merged into the project's local Claude settings file by structural match; not indexed; ownership inferred by re-derivation. Claude Code only.
- **Guardrails region** (concept): A per-plugin, marker-delimited block of verbatim `GUARDRAILS.md` content rendered into a harness's rules-file target (or the Cursor standalone sibling). Provenance carried by markers naming the catalog and plugin. Suppressed on Claude Code's `CLAUDE.md` when the plugin ships JSON hooks.
- **Agent** (concept): A plugin `agents/*.md` source (Markdown plus YAML frontmatter), indexed as an entry with the agent kind, non-searchable, identified by (catalog, plugin, kind, name). Translated per-harness into a native agent file named `<plugin>__<name>.<ext>`; optionally exposed as a persona prompt.
- **Agent translation** (concept): The per-harness mapping from the canonical agent field set to a target dialect — declaring the agent directory, file format, body placement, supported-field pass-through, and same-vendor value mapping; everything unmappable is dropped and logged.
- **Persona prompt** (concept): A view onto an agent row exposed through the MCP prompts capability when the persona setting is on — a `<name>-persona` prompt whose body is the agent body with frontmatter stripped, wrapped in a role-assumption template, Phase 5 substitution applied, plus a single global drop-persona prompt.
- **Privilege-escalation report** (concept): The diagnostic grouping of installed agents that carry the privileged frontmatter fields, by plugin; paired with the layered strip setting that removes those fields from emitted Claude Code agents.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A plugin agent enabled in a workspace whose effective list includes all four natively-supported harnesses is emitted into each harness's agent directory in that harness's format, plugin-namespaced, with the body in the harness-appropriate location and harness defaults applied to omitted fields — with no per-harness manual editing of the plugin source.
- **SC-002**: A model value with a legitimate same-vendor target (per the named model-alias table of FR-037) is mapped to that target for the relevant harness and is dropped (default inherited) for a harness whose only candidates are a different vendor; no emitted agent file ever carries a cross-vendor model identifier.
- **SC-003**: Disabling a plugin removes exactly its namespaced agent files from every harness's agent directory and leaves all other plugins' agents intact; re-running sync after a no-op change rewrites and removes nothing across hooks, guardrails, and agents.
- **SC-004**: A plugin shipping `hooks/hooks.json` referencing the plugin-root variable, enabled in a Claude Code project, results in the local Claude settings file containing the hook with the plugin-root variable rewritten to the absolute installed path and other Claude-native variables left intact; the team-shared committed settings file is never written.
- **SC-005**: Re-running sync adds no duplicate hook entry, a user-authored hook identical to a plugin hook is not duplicated on add, and disabling the plugin removes only the structurally-matching plugin hooks while leaving a user-edited copy untouched.
- **SC-006**: A plugin shipping `GUARDRAILS.md` and no JSON hooks, enabled across all five harnesses, produces a per-plugin guardrails region in `CLAUDE.md`, the shared `AGENTS.md`, and the Cursor sibling file; a plugin shipping both has its region present in `AGENTS.md` and absent from `CLAUDE.md`; two guardrails-shipping plugins produce two distinct regions in the shared file; disabling a plugin removes only its region; re-sync overwrites between markers in place.
- **SC-007**: A Claude Code project receives its rules-include block in `CLAUDE.md` rather than `AGENTS.md`; a project also shared with an AGENTS-based harness keeps a single `AGENTS.md` block; both directives resolve the same `.tome/RULES.md` with no duplicated content.
- **SC-008**: With personas off (default) no persona prompt appears in the prompt list; with personas enabled each agent appears as `<name>-persona` (clash-prefixed where required) plus exactly one global drop-persona, and a persona fetch returns the wrapped, frontmatter-stripped body with Phase 5 substitution applied and a free-form argument resolved through the Phase 5 pipeline.
- **SC-009**: A plugin agent declaring privileged frontmatter fields is emitted to Claude Code's agent directory with those fields intact by default and appears in the diagnostic privilege-escalation report; with the strip setting enabled at workspace or global scope, the same agent is emitted without those three fields and no plugin edit is required.
- **SC-010**: The diagnostic command accurately reports hooks contributed and missing, guardrails regions present/orphaned/suppressed, agent files present/orphaned with their dropped-field information and the privilege report, and the persona list when enabled; the repair mode fixes the safe cases (stale guardrails, missing agents, orphaned agent files) without deleting any unowned hook entry or user-authored content.
- **SC-011**: Every Phase 1–5 success criterion still holds after Phase 6 ships; the existing agent search and read surfaces are unchanged and agents do not appear in search results.

## Assumptions

- The PRD's "Resolved decisions" table is authoritative for the design questions it covers; this spec encodes those decisions as requirements and does not re-open them. Per-harness specifics (exact agent directories, supported frontmatter fields, model identifiers) are verified against current harness documentation at implementation time, because the ecosystem moves fast — the same caveat Phase 4 carried.
- The PRD-proposed Phase 6 exit codes 30–33 collide with the existing model-on-disk codes (30 model-missing, 31 model-corrupt, 32 model-checksum-mismatch, 33 model-registration-parse-error), and 34–37 are also already occupied (the inference/vector cluster). The four new Phase 6 codes will be assigned to a free contiguous run in the contracts phase, exactly as the Phase 4 summariser code was moved off its colliding proposal and the Phase 5 cluster was moved to 25–29. This is an implementation/contract detail, not a scope question.
- Real hooks for Codex and Gemini, native agent files for Gemini/Antigravity and the Phase 2 harnesses, semantic search over agents, server-side shell execution of the harness-native backtick syntax, hook-authoring/validation tooling, persistent-data-directory lifecycle/cleanup, and any new harnesses are out of scope and signposted for Phase 7+.
- Gemini's sunset for individuals (June 2026 per the PRD) is the reason it gets guardrails and optional personas but no native agent files; Antigravity's undocumented agent schema is the reason it is excluded from native agents. Neither decision is subject to clarification here.
- The local Claude settings file is the correct sink for rewritten hooks because it is gitignored (machine-specific absolute paths must not be committed) and keeps Tome's writes out of the team-shared file; this matches Claude Code's own local-scope conventions.
- The privilege pass-through is a deliberate capability advantage of installing via Tome over the native plugin manager; the default is pass-through-with-visibility, and the strip is opt-in — reversing the defensive-by-default stance considered during design review, on the rationale that stripping by default throws away the advantage.
- Personas are an advisory approximation only and are documented as such; they are global and off by default precisely because MCP prompts cannot be scoped to specific harnesses, so the redundancy with native agent files on the four supporting harnesses is harmless.
- The Phase 5 prompt-naming, sanitisation, length, and collision machinery and the Phase 5 substitution pipeline are reused unchanged for personas; no parallel implementation is introduced.
- The Phase 3 migration framework is the right place for the kind-domain widening if a marker is required; the `kind` column itself needs no data migration. The widening is not purely a storage change, however: the Rust `EntryKind` enum and its exhaustive matches must be widened in step (FR-070a), or the first agent row would break the per-kind count surfaces — this is in scope, not deferred.
- The codebase documents under `.sdd/codebase/` are current as of the Phase 5 Polish closeout and are not re-generated for this spec, since the spec introduces no code change (the same approach the Phase 5 spec took).

## Development Standards

- New declarative inputs Tome owns follow the existing strictness boundary; third-party plugin inputs (agent frontmatter, `hooks.json`, `GUARDRAILS.md`) are read leniently, failing loudly only on malformed recognised structures — consistent with the Phase 2–5 plugin-frontmatter boundary.
- All new error classes use the existing closed-enum discipline; each documented Phase 6 failure mode either reuses an existing variant or claims a dedicated exit code whose number is documented in the contracts (and reassigned off the colliding 30–33 proposal).
- Hooks, guardrails, and agent files are reconciled with no sidecar — ownership inferred by structural re-derivation (hooks), marker pairs (guardrails), and the namespaced-filename glob (agents) — consistent with Phase 4's filesystem-inferred state model.
- All file writes (local Claude settings, rules files, the Cursor guardrails sibling, agent files) follow the existing atomic-write discipline, including mode preservation and symlink refusal at every read/write entry point that Phase 4 established.
- The persona surface ships through the existing Phase 5 MCP prompt machinery and substitution module — no templating-engine dependency, no parallel substitution path.
- All new tests cover both human and JSON output modes for any new diagnostic surface; the existing JSON wire-shape pinning discipline carries forward to the persona prompt entries and the new hooks/guardrails/agents/persona doctor records.
- Phase 6 introduces no new top-level dependency unless the contracts phase establishes one is unavoidable for a target harness's file format; any such addition is constitution-gated like Phase 4's `toml_edit`/`llama-cpp-2` additions.
