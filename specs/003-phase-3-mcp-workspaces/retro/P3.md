# Phase 3 / P3 Retrospective: User Story 1 (MCP Server)

**Phase**: User Story 1 (T072–T103)
**Spec**: [spec.md](../spec.md)
**Plan**: [plan.md](../plan.md)
**Started**: 2026-05-14
**Closed**: 2026-05-14 (US1.a–US1.d landed in one branch session)

## What Worked Well

- **The query::pipeline / run_with_deps split** (T083): refactoring `commands::query` to extract a silent `pipeline()` function let the MCP `search_skills` handler reuse the full KNN+rerank logic without duplicating it. `run_with_deps` becomes a thin emit-wrapper. The split was forced by FR-221 (stdout = MCP protocol only) but the cleaner factoring is generally useful — every future caller that needs a query without emit can use `pipeline`.
- **`#[tool_router(vis = "pub")]` for test introspection**: the rmcp-macros `vis` argument made `Server::tool_router()` reachable from integration tests, enabling the FR-108 description-purity check + the "exactly two tools" assertion without an MCP handshake. Cheaper than spawning the binary.
- **`tokio::sync::OnceCell::new_with(Some(...))` for test prepopulation**: the production code lazy-loads the reranker on the first `search_skills` call; tests skip the lazy path by handing in a pre-populated cell with a StubReranker. Same pattern works for any `OnceCell`-backed lazy resource.
- **`spawn_blocking` inside the MCP handlers** (US1.b, US1.c): every sync operation (rusqlite, fastembed, fs::read) runs on the blocking pool so the single-threaded reactor isn't held up by inference / DB latency. Pattern: build the deps inside the handler, capture by `move`, return the outcome; the await is cheap.
- **`if !matches!(cli.command, Command::Mcp(_))` in main.rs** to skip `logging::init` + ctrlc handler installation: the global `tracing` registry only accepts one subscriber, and the CLI's stderr-based subscriber conflicts with the MCP file subscriber. Same for ctrlc: tokio's async `signal::ctrl_c()` races with the sync handler. Catching this at the dispatcher is cleaner than trying to remove a subscriber later.

## What Didn't Work

- **The `#[tool(description = ...)]` macro accepts only string literals**, not `const &str` references. The macro's `darling`-driven parser is strict on the `description` field type. Workaround: put the description as a `///` doc comment on the handler method — rmcp-macros falls back to doc comments by design (`rmcp-macros/tests/test_doc_comment_description` confirms). The doc-comment route is slightly less DRY (descriptions live in `server.rs` rather than next to the schemas in `tools/`) but it works.
- **`ServerInfo` / `Implementation` are `#[non_exhaustive]`**: struct literal construction fails to compile outside the rmcp crate. Fix: use `ServerInfo::new(caps).with_server_info(Implementation::new(name, version)).with_instructions(...)`. The builder chain is fine; just adds a small surprise the first time you write it.
- **`MutexGuard<File>` doesn't impl `Write`**: the obvious `Mutex<File>` → `MakeWriter` route doesn't compile because `MutexGuard` deref to `&File`, and `&File: Write` exists but `MutexGuard<File>: Write` does not. F8's `LockedFile<'a>(MutexGuard<'a, File>)` adapter forwards `Write` via `&File`. Already documented in F8's notes but worth flagging again — any future file-backed `tracing-subscriber` MakeWriter consumer will hit this.
- **rmcp's `large_enum_variant` clippy lint surprised the get_skill `LookupOutcome` enum**: `SkillRecord` is 208 bytes; the other variants are unit. Fix: `Box<SkillRecord>` for the `Found` variant. Three lines.

## Workarounds & Solutions

- **The `tool_router` clippy "dead_code" false positive**: the `#[tool_handler]`-generated `ServerHandler` impl accesses `self.tool_router` through the macro expansion, but clippy's dead-code analysis doesn't see it. Solution: `#[allow(dead_code)]` on the field with a comment explaining why. Cleaner than re-routing through a getter.
- **MCP handler signature must take `Arc<McpState>` not `&McpState`**: `#[tool]`-decorated methods are called by rmcp's router which doesn't know the state's lifetime relative to the request. Cloning the Arc per call is cheap; the state internals (`Arc<dyn Embedder>`, `Arc<dyn Reranker>`, etc.) are themselves shared-pointer types so the second-level clone is cheap too.
- **`McpState.scope` is `ResolvedScope`, not `&ResolvedScope`**: handlers need owned access for `spawn_blocking`-captured closures. `ResolvedScope` is `Clone` so the cost is one alloc per call.

## Packages & Dependencies

- **`schemars = "1"`** (direct dep): the `JsonSchema` derives on `Input` / `Output` / `SkillMatch` types need it. rmcp also depends on schemars 1.x; one version in the tree.
- **`rmcp` features `transport-io` + `schemars`**: `transport-io` brings `tokio/io-std` and exposes `rmcp::transport::stdio()` — the only transport we use. `schemars` exposes the schema-generation surface needed by the `#[tool_router]` macro.
- **rmcp 1.7.0** is the latest at start of US1; the `#[tool_router(vis = "pub")]` syntax landed in this version (older versions defaulted to private with no override).

## Patterns & Code

- **Silent compute / emit wrapper pattern** (T083): when a CLI command's compute path is reused by a non-CLI surface (MCP, library API, RPC), split the function into a `pipeline(args, deps) -> Result<Outcome, Error>` (no I/O side-effects) plus a `run_with_deps(args, deps, mode) -> Result<Outcome, Error>` that calls `pipeline` then emits per `mode`. Tests + alternative callers use `pipeline`; the CLI uses `run_with_deps`.

- **rmcp tool registration pattern**: declare per-tool input/output types in `tools/{name}.rs` with `#[derive(Deserialize, Serialize, JsonSchema)]`. Put the actual `#[tool]`-decorated handler methods on an `impl Server` block in `server.rs` and delegate to `tools::{name}::handle(state, input)` free functions. Keeps each tool's logic in its own file while centralising the registration table.

- **MCP error envelope pattern**: every domain-specific error code goes into `ErrorData.data` as `{"code": "<contract-name>", ...field}` JSON. The JSON-RPC numeric code (`INVALID_PARAMS` / `INTERNAL_ERROR`) is generic; the precise identifier is in `data.code`. Tests assert against `data.code` (typed `serde_json::Value` lookup).

- **`async fn` handler + sync work via `spawn_blocking`**: every MCP handler is `async fn` (required by rmcp), but the actual work (rusqlite queries, fastembed inference, fs::read) is sync. Pattern: do cheap async-friendly work directly (validation, config load), then `spawn_blocking(move || { ... sync work ... }).await`. Single-threaded runtime stays responsive.

## For Next Time

- **Expose seeds on `McpState`** for testability (TD-014 in CONCERNS.md). Today the handlers derive `embedder_seed` / `reranker_seed` from `state.embedder_entry.name/version` which are hard-coded to MODEL_REGISTRY entries. Adding `embedder_seed: MetaSeed` and `reranker_seed: MetaSeed` fields to `McpState` would let tests bootstrap an index with stub seeds and run the full happy-path pipeline without tripping drift detection.
- **Reconcile the mcp-server.md "Index DB missing → exit 35" drift** (TD-015): the contract says 35 but `IndexIntegrityCheckFailure` is 51 in error.rs and the code currently surfaces 60 (`McpStartupFailed { reason: "index_missing" }`). The 60-with-reason approach is more accurate but the contract should be amended.
- **A real MCP handshake driver in CI** is the gating piece for T093 / T094 / T095. Options: spawn the binary with real BGE models in CI (slow, ~20 GB caches), or build a minimal stub harness that runs the rmcp client side against the in-process server (faster but needs more rmcp-client integration). Defer to Phase 3 / US4 (doctor) or polish.
