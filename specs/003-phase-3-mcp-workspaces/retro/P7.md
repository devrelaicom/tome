# Phase 7 / P7 Retrospective: User Story 5 (forward schema migrations)

**Spec**: [spec.md](../spec.md)
**Plan**: [plan.md](../plan.md)
**Contract**: [contracts/schema-migration.md](../contracts/schema-migration.md)
**Started**: 2026-05-14
**Closed**: 2026-05-14 (US5.a landed in one commit on `003-phase-3-us5`)

## What Worked Well

- **The framework was already there**: Foundational F7 had already landed `Migration { from, to, name, apply: fn(&Transaction) -> Result<(), TomeError> }`, `apply_pending`, and the `MIGRATIONS_OVERRIDE` thread_local as `#[doc(hidden)] pub static`. US5 had zero production code to write — every test wrote against an interface that had been waiting for it since F7. Pre-emptive instrumentation paid off.
- **Generate-at-setup over committed binary fixtures**: `tests/common/mod.rs::write_index_db_with_schema_version(path, version)` fabricates a minimal `meta`-only synthetic DB at the requested version. The intermediate fixture-build script the contract referenced (`tests/fixtures/older-schema.db` etc.) wasn't needed — the helper generates the DB inline at test setup. Consistent with Phase 2's `Fixture::build_sample`, `fabricate_models`, and other generate-at-setup helpers.
- **RAII `MigrationsGuard` for thread-local cleanup**: a 4-line `MigrationsGuard::install(&[...])` whose `Drop` impl clears the slot makes per-test injection idempotent and panic-safe. Survives `assert!` failures so subsequent tests on the same thread never see a stale override. Used identically in `schema_migration_e2e.rs`, `atomicity.rs`, and `doctor.rs`.
- **Minimal `meta`-only fixture DB**: by fabricating *only* the `meta` table (not the full v1 schema), each test's migration step is free to create whatever tables it wants without conflicting with production tables. The "v1_marker", "v2_marker", "doctor_fix_marker" tables are entirely test-private. No fixture-vs-production-schema collision possible.
- **Synthetic `SuggestedFix` injection for dispatch-path testing**: the doctor `--fix` test exercises `repair_schema` end-to-end despite Phase 3 shipping zero registered migrations (so `build_suggested_fixes` never emits `subsystem: "schema"`). Manually appending a synthetic `SuggestedFix { auto_fixable: true, ... }` to the assembled report is a minimal-but-faithful way to test the dispatch + execute halves of a framework when the trigger half ships separately.
- **SIGINT mid-tx modelled as deliberate closure-level `Err`**: per-step transaction rollback is identical regardless of whether the abort came from a signal handler or a closure-level failure. The assertion targets (table absent, schema_version unchanged) are the same either way. `catalog::git::CANCELLED` stays unmanipulated, no test isolation hazard. Same discipline `atomicity_enable.rs` codified in P3.

## What Didn't Work

- **First-pass doctor test bootstrapped with stub seeds**: `tome::index::open` with `MetaSeed { name: "stub-embedder", ... }` writes the stub seeds into `meta.embedder_name`. Then `check_drift` reads those vs the configured `MODEL_REGISTRY` seeds (which the model fabricators install) and flags `EmbedderNameDrift` → Unhealthy. The classification assertion failed. Fix: bootstrap with `commands::plugin::registry_seeds()` so the stored side matches the configured side. The stub seeds are still the right choice for tests that don't pair with `fabricate_all_installed_models` — but when both layers are present they must agree.
- **`Migration.apply` is `fn`, not a closure**: spent a moment trying to write `apply: |tx| { ... }` before realising the signature is a function pointer. Named `fn` items are required. The contract said this; I missed it on first read. No real cost — moved the closure bodies to named functions in five minutes — but worth highlighting for Phase 4+ migration authors.

## Workarounds & Solutions

- **Doctor `repair_schema` runs `index::open` which re-runs `apply_pending`**: the `--fix` schema repair path calls `index::open` (which itself reads schema_version + invokes `apply_pending`), THEN reads schema_version again and calls `apply_pending(current, SCHEMA_VERSION)`. The second call is a no-op (current == target after the first). Belt-and-braces — the inner `index::open` already migrated, and the outer call covers the future case where `repair_schema` opens a DB through a different path. No correctness issue; documented in `fixes::repair_schema` if a Phase 4+ author wonders.
- **Stray `use std::cell::Cell;`** was caught by clippy on the discipline-check test (`migrations_override_is_thread_local_and_clears_on_drop`); the `Cell::new(0u8)` line at the bottom was defensive scaffolding I added to keep the asserts from being optimised away in `--release`. clippy didn't flag it — left in place as a low-overhead belt-and-braces.

## Packages & Dependencies

**No new dependencies.** Reuses `rusqlite`, `tempfile`, `time`, `serde_json`, `sha2`, `hex` — all pre-existing.

## Patterns & Code

- **RAII guard wraps `pub static thread_local!` injection**: pair every test-only `thread_local!` slot with a `Guard` struct in the consumer test file. `Guard::install(...)` flips the slot; `Drop` clears it. Survives panics. Reusable for any future test injection point — there's no reason to repeat the per-test manual swap-and-restore boilerplate.
- **Generate-at-setup fixture DBs**: `write_index_db_with_schema_version` is one helper, three tests use it. The contract referenced committed binary `.db` files but the helper pattern is strictly better — no binary churn, no opaque blobs in PR diffs, no surprise when the production schema layout drifts. Adopt this for any future "I need a synthetic SQLite fixture at version X" test.
- **Synthetic `SuggestedFix` injection**: when a framework SHIPS the dispatch + execute halves but DOESN'T YET produce the trigger (no production code emits the relevant `SuggestedFix`), the test manually appends a synthetic suggested fix to the assembled report. Production wiring is exercised; only the trigger is mocked. The same shape applies to any list-of-tagged-items dispatch (the doctor's `subsystem` string-routing being the parent pattern).
- **"Framework ships empty, e2e tests on synthetic fixtures" is now a repeated pattern**: F7 shipped `MIGRATIONS = &[]` and `MIGRATIONS_OVERRIDE`. US5 ships e2e tests against synthetic registrations. Phase 4+ adds the first real migration; the rails are battle-tested. The same shape should apply to any future framework whose first production consumer is a future phase — ship the test harness now so the rails are stress-tested before any real user runs against them.
- **`#[doc(hidden)] pub static` for test injection** (F7 convention, now re-confirmed): the discipline holds. `MIGRATIONS_OVERRIDE` is reachable from three integration test files without `#[cfg(test)]` visibility tricks. The doc-hidden gate keeps it out of the published API surface; the doc comment marks the test-only intent. Phase 4+ migration authors who add new test injection points should follow the same shape.

## For Next Time

- **Phase 4+ migration authors**: write the migration closure as a named `fn` item (not a closure literal) — `Migration.apply` is a function pointer, not a closure type. The pattern is illustrated in `tests/schema_migration_e2e.rs::migrate_v0_to_v1_create_marker` etc. Three lines per migration: the `fn` body, an entry in the `MIGRATIONS` slice, and a one-line `Migration { from, to, name, apply: my_fn }`.
- **The doctor `--fix` schema path has a redundant `apply_pending` call**: `repair_schema` calls `index::open` (which migrates) THEN reads schema_version + calls `apply_pending` again. Second call is a no-op when `current == target`. Worth a refactor only if/when a Phase 4+ migration takes long enough that the double-attempt costs more than the safety it buys. Today's cost is negligible.
- **Build a "schema needs migration" `SuggestedFix` builder for Phase 4+**: when the first real migration registers, `build_suggested_fixes` should grow a branch that emits `SuggestedFix { subsystem: "schema", auto_fixable: true, ... }` when `current < SCHEMA_VERSION`. Today there's no production trigger (zero migrations) so the test injects synthetically. Phase 4 should drop the synthetic injection and rely on the natural trigger.
- **The closeout could merge into the slice commit for test-only phases**: US5 was small enough (~540 lines of new tests, zero production code) that the test commit and the closeout (docs refresh + CLAUDE.md update + tasks.md tick) could plausibly be one PR. Phase 3's convention was "feat commit + closeout commit + final docs commit" but US5 fits cleanly into a feat + closeout 2-commit shape. Worth considering for any future test-only phase.
