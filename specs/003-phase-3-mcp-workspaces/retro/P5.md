## What Worked Well

- **F1's mechanical sweep paid off**: tasks T131–T140 (the per-command refactor) were already complete from Foundational F1 in PR #44. The actual deliverable for slice US3.a turned out to be the *verification artefact* — `tests/workspace_commands.rs` — rather than any production code changes. The lesson generalises: when a Foundational slice does a wholesale mechanical sweep, the downstream user-story slices that "use" that sweep often reduce to "prove it works end-to-end."
- **`reuse_existing: bool` flag controlling the rollback path**: in `catalog::add`, the "did we clone, or reuse?" distinction is small (3 lines) but load-bearing — `store::save` failure rollback must NOT delete a cache directory we don't own. Naming the boolean after the decision (not "should_rollback") keeps the call site readable.
- **Cache-reuse + refcount compose cleanly with the existing cascade flow**: the cascade-disable path (Phase 2) already runs INSIDE `commands::catalog::remove::run`. Falling through to the new refcount check at the end means no separate update to `cascade_disable_for_catalog`; ordering is implicit in the flow. T146 reduced to a doc-comment task.
- **CLI-binary + library-API bridge test pattern**: the cross-scope cascade scenario (`cascade_remove_in_workspace_does_not_remove_shared_clone`) needed both real-git-clone `catalog add` (CLI binary) AND library-API enable (StubEmbedder, no real BGE load). Sharing the same `paths_for(env)` Paths between the two sides made the bridge mechanical: CLI calls write the disk state, library calls operate on it, CLI calls verify the final state. Reusable for any future "needs both real flow and stubbed compute" scenario.
- **`make_workspace_opt_in` helper**: tests that want the workspace registry MUST touch `workspaces.txt` BEFORE running `tome workspace init` (init's append step is opt-in — it doesn't create the file). Hiding the order in a helper keeps test bodies focused on the assertion, not the setup ritual.

## What Didn't Work

- **The Phase 1 cache-collision error message ("`<source>` (cache path collision)") had to be removed** to enable cross-scope sharing. No callers in the test suite asserted on the specific message, so removal was clean — but it's a category of "Phase N test relied on Phase N-1 error message, Phase N+1 removed it, test still passed by coincidence" risk worth flagging. Tighter exit-code/message assertions in the original test would have surfaced this earlier.
- **Initial test for `catalog_list_returns_only_scope_catalogs` and `catalog_remove_affects_only_resolved_scope` failed** because they tried to add the same URL twice (once per scope) and hit the not-yet-lifted Phase 1 cache-collision error. The fix: use two distinct `Fixture::build_sample()` instances. Lesson: when the cross-scope-sharing feature lives in slice US3.b, slice US3.a's cross-product tests can't assume sharing works yet — keep them scope-isolation-only, not URL-sharing.

## Workarounds & Solutions

- **`reuse_existing` flag on the rollback path**: covered above.
- **Manifest re-parse from cache on reuse**: `catalog::add`'s reuse path re-reads `tome-catalog.toml` from the existing cache directory. Same parser, same validation. The display-name collision check then sees the same `manifest.name` the original clone recorded — same UX as a fresh clone.
- **Cascade-disable's pure-deletion property bridges seed spaces**: in the cross-scope cascade test, the library-API enable writes StubEmbedder seeds (`stub-embedder`, `stub-reranker`) into the index's `meta` table. The subsequent CLI `catalog remove` would normally construct `OpenOptions` with `registry_seeds()` (BGE seeds) — which would trip drift detection. But `cascade_disable_for_catalog` is pure SQL `DELETE` statements; it opens the DB but doesn't actually instantiate the embedder. So the stub-vs-real-seed mismatch is invisible.

## Packages & Dependencies

- **No new direct dependencies.** All work uses `tempfile`, `sha2`, `hex`, `serde`, and `toml` — all pre-existing.

## Patterns & Code

- **Content-addressed shared resource with reference counting**: `catalogs/<sha256>/` cache directories are now shared across scopes; `reference_count(url, paths) -> Vec<Scope>` is the ad-hoc reference table (walks every known config); deletion is conditional on `refs.is_empty()`. The reference enumeration is unlocked — the TOCTOU profile is named, accepted, and documented (matches Phase 2's `cascade_disable_for_catalog` pre-check). Reusable for any future shared-on-disk resource (downloaded model artefacts, vendored extensions, etc.).
- **Opt-in tracking with documented degradation**: `workspaces.txt` is opt-in (`tome workspace init` only appends if the file exists). Without it, `reference_count` sees only the global config — a workspace's reference is invisible, so a global remove can delete a clone that a workspace still points at. The recovery path (`tome catalog update` re-clones) is documented in the contract.
- **`reuse_existing` boolean controlling rollback paths**: when a function may either CREATE a resource or REUSE an existing one, name the boolean after the decision ("did we take the reuse path?") rather than after the action ("should we rollback on save failure?"). Read-time intent is clearer.
- **Cross-scope cascade test pattern**: when a scenario requires both `catalog add` via CLI (real git clone) AND `lifecycle::enable` via library API (StubEmbedder), share `paths_for(env)` between them and intersperse CLI binary calls with library calls. The single Paths value means both surfaces touch the same on-disk layout. Works only when the CLI's downstream operations don't need the real embedder (`catalog remove --force` cascade is pure deletion; this property is what makes the bridge work).

## For Next Time

- **The dangling-reference race in `reference_count`** (process A observes empty, process B re-adds, A deletes clone, B's reference dangles) is documented but not actively tested. The recovery path is `tome catalog update` re-cloning. Worth adding to `tome doctor`'s catalog-clone report (US4) as a detectable state: cache absent + config references present → "missing, run update to re-clone."
- **The opt-in registry is now load-bearing for clone-cleanup correctness**. Doctor should report registry opt-in status prominently — a user who has multi-scope catalogs but no `workspaces.txt` is silently losing clone-cleanup safety. The contract already calls this out; doctor will surface it.
- **`catalog::store::reference_count` is unaware of orphan clones** (cache dir on disk, but no config references it). That's a separate doctor concern (per the catalog-extensions-p3.md §Doctor reporting section). When implementing doctor's catalog section, enumerate `catalogs/` on disk and cross-reference; an orphan is informational, not auto-fixable.
- **Same-URL-but-different-name across scopes** works because the display-name collision check is scope-local. This is the right behaviour — a user might want "midnight-experts" globally and "midnight-experts-prod" in a workspace. Document this in user-facing docs when they exist.
