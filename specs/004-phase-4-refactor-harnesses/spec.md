## Overview

Phase 1 gave Tome catalogs. Phase 2 made the inventory inside those catalogs queryable through a local semantic index. Phase 3 turned that index into an MCP-consumable surface for agentic coding harnesses, and introduced workspaces as a way of scoping state per project.

Phase 4 does two things in one phase, both of which were learned through dogfooding Phase 3:

1. **It corrects the architecture.** Phase 3 made every workspace its own self-contained universe — its own `.tome/` directory carrying its own catalog config, its own index database, its own everything — and made "global" a special-case flag. That shape didn't hold up under scrutiny. The Phase 4 architecture moves all of Tome's storage under a single `~/.tome/` tree, centralises the SQLite database, makes workspaces named first-class objects stored centrally, and reduces a project's `.tome/` directory to a thin binding pointer plus a copied rules file. "Global" is no longer a flag — it is a workspace named `global` that other workspaces are equal to. This is a refactor, not an additive feature; it touches every path, every storage call, and every workspace-aware code path that landed in Phase 2 and Phase 3.
2. **It adds cross-harness agent integration on top.** With the central architecture in place, a single command — `tome workspace use <name>` from inside a project — binds the project to a workspace, drops a rules file into the project, and writes the Tome MCP server entry into the configured agentic coding harnesses' MCP configs. Plugins enabled in the workspace become discoverable from any of those harnesses without the developer touching a config file. Five harnesses are supported in Phase 4: Claude Code, Codex CLI, Gemini CLI, Cursor, and OpenCode.

A third local model joins the embedder and reranker: a small instruction-tuned language model that generates two natural-language summaries per workspace — a short one for the MCP search tool's description (so the agent's host LLM knows when to call `search_skills`) and a long one for the workspace's rules file body (so the agent gets a brief, current rundown of which topics the workspace's enabled plugins cover). Both summaries are cached in the workspace's settings file and regenerated whenever the enabled plugin set changes.

Phase 4 also ships layered, composable per-scope settings. A project, the workspace it is bound to, and the global workspace each may declare a list of harnesses to integrate with. The effective list for any given project is computed by walking that priority chain. Inside each list, lightweight composition syntax (`[workspace]`, `[workspaces.<name>]`, `[global]`, plain names, and `!`-prefixed exclusions) makes it possible to combine and override without copy-paste.

The Phase 3 schema migration framework, shipped empty in Phase 3, has its first real registered migration in Phase 4: the database schema moves from version 1 to version 2 to introduce the new junction tables. Pre-release, there is no automated migration of user data from Phase 3's per-workspace databases — users wipe their old state directories by hand.

What this phase explicitly does not do: translate commands, agents, or hooks across harnesses (Phase 5+); add harnesses beyond the five listed (Phase 5+); add escape-hatch config knobs for the summarisation model; provide tooling for cross-machine workspace synchronisation; offer backward-compatibility shims for prior phases' on-disk layouts; or support Windows-native or WSL1 environments.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Bind a project to a workspace and get cross-harness integration automatically (Priority: P1)

A developer is starting work on a new project. They want their agentic coding harness (or several of them) to be able to find and load every skill in a particular set of Tome plugins as soon as they open the project, with no per-project boilerplate, no copy-pasted MCP config entries, and no rules files they have to maintain by hand. They want one command in the project directory to make the whole thing work.

**Why this priority**: This is the headline Phase 4 deliverable. Every other Phase 4 surface (workspace lifecycle, summarisation, layered settings, harness modules) exists to make this one command flow possible end-to-end. Without it, Phase 4 has nothing to demonstrate.

**Independent Test**: From a fresh install of Tome on a machine where at least one supported agentic coding harness is present, the developer creates a workspace by name, enrols a catalog into it, enables a plugin from that catalog, and runs the bind command in a project directory. The bind command writes a project-scope marker that names the workspace, drops a rules file into the project, and writes the Tome MCP server entry into each configured harness's MCP config. Launching the harness in that project surfaces a Tome MCP server that returns search hits for skills in the enabled plugin. The auto-generated rules file content tells the host LLM which topics this workspace covers and that it should call the search tool when the conversation touches those topics. None of this requires the developer to edit a config file by hand.

**Acceptance Scenarios**:

1. **Given** a fresh install of Tome on a machine where at least one supported harness's per-user directory exists, **When** the developer creates a named workspace, enrols a catalog into it, enables a plugin from that catalog, and runs the bind command in a project directory, **Then** the project's marker directory is created with a binding pointer to the workspace and a copy of the workspace's current rules file; the configured harness's MCP config gains a Tome entry that names the workspace; and rerunning the same bind command is a no-op.
2. **Given** the bind command has run in a project, **When** the developer launches the harness in that project, **Then** the harness sees a Tome MCP server configured against the workspace's name, the agent can call the search tool from within a session, and the agent receives results for skills enabled in that workspace.
3. **Given** the workspace's settings declare more than one harness, **When** the developer runs the bind command, **Then** every named harness gets the MCP entry written into its config and every named harness's rules-file target receives the workspace's rules content; harnesses not named in the settings receive nothing.
4. **Given** the bind command has run, **When** the developer enables an additional plugin in the workspace, **Then** the workspace's cached summaries regenerate, the workspace's rules file is rewritten, and the bound project's rules file is updated to match without further developer action; subsequent search calls in the harness reflect the new plugin's skills.
5. **Given** a project bound to one workspace, **When** the developer rebinds the project to a different workspace, **Then** the project's marker pointer changes, the project's rules file is replaced with the new workspace's content, the harness MCP config's Tome entry is rewritten to name the new workspace, and the old binding is removed from the central registry.
6. **Given** the bind command would touch a harness MCP config file that already contains an entry named "tome" whose contents do not match Tome's expected shape, **Then** the command refuses with a dedicated error naming the conflicting entry and points the developer at the explicit override flag; passing the override flag rewrites the entry and continues.
7. **Given** the developer runs the bind command outside of any directory the bind command considers a project (for example, in their home directory), **Then** the command refuses with a clear error rather than creating a binding at an unexpected path.

### User Story 2 — Manage named workspaces through their lifecycle (Priority: P2)

A developer accumulates workspaces over time — one for each project family they work in, one for occasional one-off exploration, one for whatever the maintainer of a particular tool just released. They want to list them, inspect them, rename them, retire them, and propagate their state to the projects that are bound to them. They want all of this to live in one command surface that is independent of any particular harness.

**Why this priority**: Without the lifecycle commands, workspaces exist only as a side-effect of initial creation and bind; a developer can never see what they have, rename anything as their thinking evolves, or remove anything without manual surgery on the central registry. This story is what makes the named-workspace model usable past the first hour.

**Independent Test**: From an install with at least two workspaces and at least one bound project, the developer runs each lifecycle command in turn. The list command shows every workspace with a count of bound projects and enabled plugins. The info command (with no arguments) names the currently resolved workspace; with a workspace name, it names that specific one. The rename command updates the central directory, the database row, and every bound project's marker. The remove command refuses without an override when projects are bound and cascades safely with an override. The sync command propagates the workspace's current rules content to every bound project. The regen-summary command regenerates and re-syncs without changing the enabled plugin set.

**Acceptance Scenarios**:

1. **Given** at least two workspaces exist, **When** the developer runs the list command, **Then** the output names every workspace and reports for each: the count of enrolled catalogs, the count of enabled plugins, the count of indexed skills, the count of bound projects, and the last-used timestamp; structured output is available.
2. **Given** a developer is inside a bound project, **When** they run the info command with no arguments, **Then** the output identifies the currently resolved workspace, the resolution method, and the same per-workspace counts the list command reports plus the central path of the workspace's settings file and rules file.
3. **Given** the developer runs the info command with a specific workspace name, **When** the workspace exists, **Then** the same per-workspace report is returned for that workspace regardless of which workspace the developer is currently inside.
4. **Given** a workspace exists and is referenced by zero bound projects, **When** the developer renames it, **Then** the workspace's central directory is renamed, the workspace's database row is updated, the central database's workspace record carries the new name, and no other surface needs updating.
5. **Given** a workspace is referenced by one or more bound projects, **When** the developer renames it, **Then** the same central rename happens and every bound project's marker file is updated to name the new workspace; if any bound project's directory has been deleted from disk since binding, the command fails the rename with a clear error and changes nothing.
6. **Given** a workspace is referenced by at least one bound project, **When** the developer removes it without the override flag, **Then** the command refuses with a dedicated error that names every bound project's path.
7. **Given** the same situation, **When** the developer removes it with the override flag, **Then** the command tears down the integration in every bound project (removes the Tome rules-file block or rules file, removes the Tome MCP entry from every configured harness), removes each bound project's marker, deletes the workspace's central directory, deletes the workspace's database rows, refcount-cleans any catalog clone no longer referenced by any workspace, and exits cleanly.
8. **Given** the reserved workspace `global` exists at all times, **When** the developer attempts to remove it, **Then** the command refuses with a dedicated error and no operation runs.
9. **Given** a workspace's central rules file has been regenerated, **When** the developer runs the sync command for that workspace, **Then** every bound project's copy of the rules file is brought up to date; the command is idempotent and re-running it produces no further changes.
10. **Given** the developer wants to regenerate the cached summaries against the current enabled-plugin state, **When** they run the regen-summary command, **Then** both the short and long summaries are recomputed from the current plugin set, written to the workspace's settings file, and the rules file is re-synced to bound projects in the same invocation.
11. **Given** the developer attempts to create a workspace whose name is reserved, contains characters not permitted by the documented name rule, or matches the name of an existing workspace, **Then** the command refuses with the appropriate dedicated error and creates nothing.

### User Story 3 — Control which harnesses each project, workspace, and install integrates with, using composable layered settings (Priority: P2)

A developer wants different projects to be integrated with different sets of harnesses. They mostly use one harness, but a particular project should also be configured for another harness because they use that harness when pairing with a colleague. A different workspace, used across multiple projects, should be configured for two harnesses uniformly. They want to declare these things in one place per scope (project, workspace, install) and compose without copy-paste.

**Why this priority**: A flat per-project harness list is enough to work, but it forces copy-paste between projects in the same family and forces every project to be touched when the developer adds a new harness to their environment. The composition shape solves both problems and is the same shape developers already understand from many other layered-config systems.

**Independent Test**: From an install where multiple harnesses are detected on the local machine, the developer declares a harness list at the global scope, a different harness list at the workspace scope (with the workspace referencing global via composition), and a third declaration at a project scope (with the project referencing the workspace via composition and an explicit exclusion). Running the harness-list command in the project reports the effective list with each harness annotated by which scope contributed it. Removing the project-scope declaration falls through to the workspace; removing both falls through to global. A composition cycle (one workspace's list references another whose list references the first) is detected and refused. An attempt to use the `[workspace]` reference from project settings that are not bound to any workspace is refused.

**Acceptance Scenarios**:

1. **Given** the developer has declared a harness list at the global scope only, **When** they run any command that needs the effective list in a project bound to a workspace that has no harness list of its own and no project-scope declaration, **Then** the effective list comes from global.
2. **Given** the workspace's settings file declares a harness list and the project has no declaration, **When** the same command runs, **Then** the effective list comes from the workspace's declaration; the global declaration is not consulted unless the workspace's declaration explicitly references it.
3. **Given** the project's marker file declares a harness list, **When** the same command runs, **Then** the effective list comes from the project's declaration regardless of what the workspace or global declare; the workspace and global declarations are consulted only if the project's declaration references them via composition.
4. **Given** any scope's declaration includes an explicit composition reference (to another workspace, to the bound workspace, or to global), **When** the effective list is resolved, **Then** every referenced source's list is expanded recursively, with `!`-prefixed names subtracted from the union after expansion; the order of items in the array does not matter.
5. **Given** any declaration includes a composition reference that produces a cycle (workspace A includes B which includes A), **When** the effective list is resolved, **Then** the command refuses with a dedicated cycle-detected error naming the path of references that closes the loop.
6. **Given** project settings include the `[workspace]` reference but the project's marker does not name a workspace, **When** the effective list is resolved, **Then** the command refuses with a dedicated error.
7. **Given** a workspace's settings file or the global settings file include the `[workspace]` reference, **When** the effective list is resolved, **Then** the command refuses with the same dedicated error; the `[workspace]` reference is valid only at project scope.
8. **Given** a declaration explicitly sets the harness list to empty, **When** the effective list is resolved, **Then** the result is empty for that scope and falls through no further; no integration is performed.
9. **Given** the developer runs the harness-list command without arguments from a bound project, **When** the command runs, **Then** the output reports the effective list and annotates each entry with the scope (project, workspace, or global) and the reference path that brought it into the list; entries removed by `!`-prefixed exclusions are reported separately.
10. **Given** the developer runs the harness-list command with a workspace name argument, **When** the command runs, **Then** the output reports that workspace's directly-declared list verbatim, without expanding composition references; this is the "as-written" view for inspecting one workspace's intent.

### User Story 4 — Have each workspace describe its enabled plugins to the agent in plain English, automatically (Priority: P2)

A developer enables a few plugins in a workspace. They want the agent in their harness to know — without the developer typing anything — which topics this workspace covers and that the agent should call the Tome search tool when the conversation moves into those topics. They want this to update itself as they enable, disable, or update plugins.

**Why this priority**: An agent's host LLM decides whether to call a tool based on the tool's description. Without a current, plain-English summary of what the workspace's plugins cover, the host LLM has no reason to call `search_skills` and the workspace's value never reaches a conversation. The rules file solves the same problem from the agent-rules side: a brief workspace overview the host LLM reads at session start. Both sides need the same authoritative summary, and that summary needs to track the workspace's actual plugin state, not a stale moment-in-time blurb.

**Independent Test**: From an install with the summarisation model downloaded, the developer enables a plugin in a workspace and watches two summaries generate: a short one bounded by a documented character window and a long one within a separate documented window. Both are written to the workspace's settings file under a dedicated section. The short summary appears inside the MCP search tool's description. The long summary appears as the body of the workspace's rules file. Disabling the plugin regenerates both summaries; enabling a different plugin regenerates them again; reindexing a plugin whose contents materially changed regenerates them again; an explicit regen command regenerates them on demand. The summaries are descriptive of the actual plugin set; they name no specific plugin or skill by literal identifier in the short form, and the long form invites the agent to call the search tool when working on tasks involving those topics.

**Acceptance Scenarios**:

1. **Given** a workspace exists, the summarisation model is present on disk, and the developer enables a plugin in the workspace, **When** the enable operation completes, **Then** the workspace's settings file contains a short summary within the documented character window and a long summary within its own documented window; both summaries are generated from the workspace's current enabled-plugin state.
2. **Given** the same situation, **When** the bind command writes (or rewrites) the harness MCP config, **Then** the Tome MCP entry's tool description for the search tool includes the workspace's short summary inline alongside a fixed scaffold; the description as a whole fits the documented agent-host limit.
3. **Given** the same situation, **When** any of the lifecycle operations that change the enabled plugin set runs (plugin enable, plugin disable, plugin reindex with content changes, catalog update affecting enabled plugins, explicit regen-summary), **Then** both summaries are regenerated, the settings file is rewritten, and any bound project's rules file is re-synced.
4. **Given** the workspace's settings file already contains cached summaries from a prior regeneration, **When** an operation runs that does not change the enabled plugin set (for example, plugin reindex with no content changes), **Then** the cached summaries are reused; the summarisation model is not invoked.
5. **Given** the summarisation model is missing or fails its integrity check at the moment summarisation is required, **When** any operation that would trigger regeneration runs, **Then** the operation exits with a dedicated non-zero status; the rest of the operation does not silently succeed without a summary.
6. **Given** the summarisation model produces output that exceeds the documented length window for the short summary, **When** the result is written, **Then** the system logs a non-fatal warning naming the workspace and the observed length; the value is still cached and used, on the principle that a too-long description is better than no description, but the warning surfaces a tuning issue.
7. **Given** the developer manually edits the workspace's central rules file by hand, **When** they run the sync command (without regen-summary), **Then** the manually-edited content is propagated to bound projects; regeneration is not triggered as a side-effect of sync.
8. **Given** the developer runs the regen-summary command, **When** the model produces both summaries successfully, **Then** the settings file is rewritten and the bound projects are re-synced in the same invocation; the developer does not need to follow up with a separate sync command.

### User Story 5 — Diagnose binding and harness-integration state with one command (Priority: P3)

A developer is debugging odd behaviour after setting up a workspace and binding a project to it. The agent in their harness does not seem to see Tome. They want a single command that tells them: which workspace this project is bound to, whether the binding is well-formed, whether each configured harness's rules file and MCP config contain the expected Tome entries, and where any drift lies. They want non-destructive automatic repairs for the obvious problems.

**Why this priority**: Phase 3 shipped the doctor command for the index and model subsystems. Phase 4 introduces three more subsystems that can drift: the workspace ↔ project binding, the rules-file integration in each configured harness, and the MCP config entry in each configured harness. Without a doctor extension that covers them, "my agent doesn't see Tome" becomes an unstructured debugging exercise across multiple harness config files. This story is what makes Phase 4 supportable in the field.

**Independent Test**: From an install with a workspace, a bound project, and two configured harnesses, the developer runs the doctor command in the project. The report identifies the bound workspace and the resolution method (no flag, environment variable, project marker). It reports the effective harness list with each harness's rules-file target and MCP config target. For each harness, it reports whether the rules-file content is current and whether the MCP config entry is Tome-shaped and points at the correct workspace. Deleting the project's rules file by hand makes the report flag the file as missing; running with the fix flag re-copies it from the workspace's central rules file. Replacing the harness MCP entry's command field by hand makes the report flag the entry as user-owned; the fix flag does not overwrite without the explicit override. From outside any project, the report reports the global workspace and reports no project binding.

**Acceptance Scenarios**:

1. **Given** the developer is inside a bound project with all configured harnesses correctly integrated, **When** they run the doctor command, **Then** every subsystem (workspace, binding, rules-file integration per harness, MCP integration per harness, models, index, catalog cache) is reported as healthy and the command exits zero.
2. **Given** the same situation, **When** they pass the explicit "use global state" syntax (per the new workspace model, by naming the global workspace), **Then** the doctor reports the global workspace as resolved and reports no project binding; subsystems unrelated to the binding continue to report normally.
3. **Given** the project's marker file references a workspace that does not exist in the central registry (deleted out from under the project), **When** the doctor command runs, **Then** the binding subsystem is reported as broken with a dedicated diagnosis and the suggested fix is to either rebind to a different workspace or recreate the named one; the fix flag does not auto-resolve this case.
4. **Given** the project's rules-file copy is missing or its contents differ from the workspace's central rules file, **When** the doctor command runs, **Then** the binding subsystem is reported as drifted with a dedicated diagnosis; the fix flag re-copies the file from the central source.
5. **Given** a harness's rules-file target is missing the Tome block (or, for standalone-file harnesses, the Tome file), **When** the doctor command runs, **Then** that harness's rules integration is reported as broken with a dedicated diagnosis; the fix flag re-runs the harness-sync logic for that harness.
6. **Given** a harness's MCP config contains an entry named "tome" whose shape does not match Tome's expected shape (different command, different first argument), **When** the doctor command runs, **Then** that harness's MCP integration is reported as broken with a "user-owned entry" diagnosis; the fix flag does not overwrite without the explicit override.
7. **Given** the same situation, **When** the developer combines the fix flag with the explicit override flag, **Then** the user-owned entry is replaced with Tome's expected shape; the override is required and explicit.
8. **Given** the effective harness list for the current project is empty (every layered scope opts out), **When** the doctor command runs, **Then** the harness subsystems are reported as "not applicable" rather than as broken; the report's overall classification is unaffected by harnesses being deliberately disabled.
9. **Given** the local machine has a per-user directory for a supported harness but that harness is not in the effective list, **When** the doctor command runs, **Then** the report informationally notes the detected harness and reports that no integration was configured; this does not affect the overall classification.
10. **Given** the developer pipes the doctor output to a file, **When** the command runs, **Then** colours, glyphs, and progress are suppressed and structured output is available with the same per-subsystem classifications and suggested fixes.

### Edge Cases

- **Bound project's directory is deleted from disk between bind and the next command**: the central database still records the binding. The next operation that walks bound projects skips the missing path with a debug-verbosity log line; the workspace's rename, remove, and sync commands continue past the missing entry.
- **Two parallel bind commands from two terminals in the same project**: standard atomic-write discipline applies — both writes go to disk through temp-file renames; the last commit wins; the central database's `workspace_projects` row insert is idempotent.
- **Project marker present but its config file is empty or names a nonexistent workspace**: any command that reads the binding exits with a dedicated error pointing the developer at the doctor command rather than silently treating the project as unbound.
- **Workspace renamed while an MCP server is running against it**: the MCP server has an open handle on the central database. Operations continue serving from the open handle. The rename completes successfully because the workspace name is decoupled from the file location of any database — the rename only moves the central settings directory and updates the workspace's row in the central database; the next process to launch the MCP server sees the new name. The harness's MCP config entry is rewritten by the rename command so subsequent agent sessions launch the server with the right workspace argument.
- **Workspace removed via the override flag while an MCP server is running against it**: same handle semantics; the running server continues serving its open handle until terminated; the harness's MCP config entry is removed by the cascade so a subsequent launch will fail with a "workspace not found" error pointing the developer at rebinding.
- **Composition reference to a workspace that does not exist**: hard error during resolution naming the missing workspace and the chain of references that reached it.
- **Composition reference that resolves to an empty list**: contributes nothing to the union; this is success, not an error. The effective list may still be non-empty due to other references and explicit additions.
- **`!`-prefixed exclusion of a harness that no reference adds**: silently no-op. The exclusion is applied as a set subtraction; subtracting from a set that did not contain the element changes nothing.
- **Project's marker declares a project-scope harness list that is empty (`harnesses = []`)**: explicit opt-out at project scope. Effective list is empty; workspace and global lists are NOT consulted. Doctor reports harness subsystems as "not applicable."
- **A harness is named in the effective list but is not detected on the local machine**: the integration is still written (the rules-file target and MCP config path are computed from the harness module's contract regardless of whether the harness is installed). The developer can install the harness later; Tome makes no claim about the harness's behaviour. The doctor command informationally notes the not-detected state but does not flag it as broken.
- **A harness is named in the effective list but its MCP config file's parent directory does not exist**: the bind/sync command creates the parent directory using the same atomic-write discipline that protects content writes.
- **Multiple harnesses in the effective list target the same rules file (for example, several harnesses all using AGENTS.md)**: one Tome block is written to that file. Removing one of the harnesses from the effective list leaves the block intact because at least one harness still targets that file; removing the last harness that targets that file removes the block.
- **Multiple harnesses in the effective list target the same MCP config path (rare but possible if two harnesses share a config convention)**: one Tome entry is written. Same removal logic.
- **Existing entry in a harness MCP config named "tome" but written by something else (a developer-authored entry that pre-dates this Tome install)**: refused without the override; doctor reports it as user-owned; override-with-confirm rewrites it. Tome's marker logic is content-based — `command == "tome" && args[0] == "mcp"` is Tome-owned; anything else is user-owned.
- **Developer hand-edits the Tome block inside a rules file**: on next sync, the block contents are rewritten to match the workspace's current rules file (the Tome block is fully owned by Tome). Developer-authored content outside the block is preserved.
- **Developer hand-edits the Tome MCP config entry's `env` field**: preserved across sync. The marker comparison ignores `env`; the rewrite only touches `command` and `args`.
- **Summarisation triggers in a workspace whose `RULES.md` content the developer manually customised**: the regenerated content overwrites the manual edits in the workspace's central rules file. There is no diff prompt and no merge. The manual-edit-then-sync flow remains available — a developer who wants custom content avoids regen-summary and uses sync to push their hand-edited file.
- **Summarisation model produces unparsable or empty output for either prompt**: dedicated non-zero exit; nothing is written; prior cached summaries (if any) remain in place; doctor reports the failure as a summariser subsystem fault.
- **Workspace inside a workspace (or, more precisely, a project marker inside a project marker)**: the closer (more deeply nested) project marker wins. The outer project's binding is unaffected and continues to be visible to commands run outside the inner project's subtree.
- **Schema migration races with another process**: per Phase 3 discipline, forward migrations are taken under the same advisory lockfile that protects ordinary writes; only one process attempts the migration; the other waits, then re-opens at the new schema.
- **Composition cycle through a path of length one (a workspace's settings reference themselves)**: same cycle error; the path is a single hop.
- **`!`-prefixed name that is itself a composition reference, e.g. `![global]`**: undefined behaviour in v1; the system documents that exclusions accept harness names only, and refuses any `!`-prefixed value that is not a plain harness name. (Treat this as a usage error rather than expanding it semantically.)
- **Per-user state directory is read-only (uncommon, but possible in some hardened environments)**: the install fails fast at first write with a dedicated permissions error; no command silently continues against a half-built state.
- **WSL2 project whose marker is created on the WSL2 filesystem (`~/projects/foo`) versus the Windows filesystem (`/mnt/c/...`)**: WSL filesystem only is supported; the Windows filesystem case is explicitly out of scope per the PRD and is documented in `Assumptions`.

## Requirements *(mandatory)*

### Functional Requirements

**Path consolidation and storage layout**

- **FR-300**: The system MUST resolve every Tome-owned on-disk path under a single per-user root directory named `.tome` inside the developer's home directory. The Phase 1 / Phase 2 / Phase 3 references to per-user configuration, data, state, and cache directories (the XDG-style separation) MUST NOT exist anywhere in the code after this phase.
- **FR-301**: The system MUST expose every Tome-owned path through a single typed accessor module. The accessors MUST be the only way other code constructs Tome-owned paths.
- **FR-302**: The system MUST resolve the home directory through a portable mechanism that respects the operating system's notion of the developer's home; the `directories` dependency carried since Phase 1 MUST be removed.
- **FR-303**: The Tome-owned on-disk layout MUST be: a global Tome configuration file at the root; a global harness-settings file at the root; one central index database file at the root; a `catalogs` subdirectory holding shared catalog clones keyed by URL hash; a `models` subdirectory holding the embedder, reranker, and summariser model artefacts; a `logs` subdirectory holding the MCP server's log file; and a `workspaces` subdirectory holding one named subdirectory per workspace (including the privileged `global` workspace).
- **FR-304**: The system MUST NOT fall back to or recognise any Phase 3 location. Developers upgrading from Phase 3 MUST wipe their old per-user state directories by hand; this is a pre-release breaking change and is explicitly within scope of Phase 4.
- **FR-305**: All Tome-owned write operations to files under the new root MUST honour the same atomic-write discipline carried since Phase 1 (write to a sibling temp file on the same filesystem, fsync, atomic rename). On Unix, files containing developer-relevant configuration MUST be created with mode 0600.

**Central database**

- **FR-320**: The system MUST persist all relational state (workspaces, catalogs per workspace, projects bound per workspace, skills, skill embeddings, per-workspace skill enablement, schema metadata) in a single SQLite database file at the central path. Per-workspace database files MUST NOT exist anywhere on disk.
- **FR-321**: The database MUST carry a schema version recorded in a metadata row, as in Phase 2 / Phase 3. The Phase 4 schema MUST be version 2; Phase 3's schema is version 1. The Phase 3 forward-migration framework MUST be exercised by this version bump as its first registered production migration.
- **FR-322**: The Phase 4 schema MUST contain at minimum: a workspaces table (unique workspace name, created-at, last-used-at); a skills table indexed by the triple (catalog, plugin, name) and carrying the skill's description, plugin version at last index, on-disk path, content hash, and indexed-at timestamp — workspace-agnostic, with no enablement column; a vector embeddings virtual table keyed by skill id; a workspace-skills junction table (workspace id, skill id, enabled-at) marking enablement, primary key (workspace id, skill id); a workspace-catalogs table mirroring each workspace's enrolled catalogs and their pinned references, primary key (workspace id, catalog name); a workspace-projects table recording the project paths bound to each workspace, primary key on **project path alone** (the column carrying the path) — enforcing the 1:1 project-to-workspace binding required by FR-342; workspace id is a non-key column of this table.
- **FR-323**: On first database bootstrap, the system MUST seed exactly one workspace row named `global`. This row MUST persist for the lifetime of the install and MUST NOT be deletable.
- **FR-324**: Every query that returns skills scoped to a workspace (CLI query, MCP search, doctor reports, status output) MUST join the workspace-skills junction and MUST return only skills enabled in the resolved workspace.
- **FR-325**: Reads against the central database MUST continue to use the Phase 2 / Phase 3 read-only open path and MUST NOT block on the advisory lockfile. Writes MUST take the advisory lockfile under the same discipline as Phase 2. Readers opened against schema 1 while a writer holds the lock to apply the v1→v2 migration MUST either (a) complete against schema 1 (if the migration's commit has not yet landed under SQLite WAL semantics) or (b) fail with the dedicated schema-version-too-new refusal once the commit lands. A reader MUST NEVER observe a half-migrated schema mid-statement.
- **FR-326**: The migration from schema 1 to schema 2 MUST be wrapped in a single atomic transaction and MUST leave the database at schema 1 if any step fails. Backward migration (schema 2 → schema 1) MUST NOT exist as a code path.
- **FR-327**: The schema-1 → schema-2 migration MUST be **structural only**. It MUST create the Phase 4 tables (workspaces, workspace-skills, workspace-catalogs, workspace-projects) and the seeded `global` row. It MUST NOT attempt to migrate Phase 3 row contents from a real Phase 3 user database — the architecture's pre-release wipe (FR-304) means no Phase 3 user database with developer data will ever be opened by a Phase 4 binary. A v1 database opened by a Phase 4 binary in practice is either a freshly-bootstrapped empty v1 record (impossible after this phase — bootstrap goes directly to v2) or a synthetic test fixture; the migration is exercised end-to-end against the synthetic fixture (SC-116). Data migration from Phase 3 user data is explicitly out of scope.

**Workspace storage model**

- **FR-340**: A workspace MUST be a named, centrally-stored object. Its on-disk presence MUST be a subdirectory at `<root>/workspaces/<name>/` containing two files: a settings file (the workspace's own catalog enrolments, harness declaration, and cached summaries) and a rules file (the long-summary content shown to agents).
- **FR-341**: A project's `.tome/` directory MUST contain at most two files: a configuration file naming the workspace this project is bound to, and a copy of that workspace's rules file. No skills are stored under the project marker; no database is stored under the project marker; the project marker is a binding pointer plus a cached rules copy and nothing else.
- **FR-342**: Project-to-workspace binding MUST be one-to-one. A project MUST NOT be bound to more than one workspace at a time. The one-to-one property MUST be enforced at the database layer by the `workspace_projects` primary key on project path alone (FR-322), not by application-level checks. Rebinding (running the bind command in an already-bound project naming a different workspace) is supported and MUST be implemented as an UPSERT against that primary key; it overwrites the prior binding atomically.
- **FR-343**: A workspace MUST be allowed zero or more bound projects. A workspace with zero bound projects is valid and operates correctly through every lifecycle command except those that operate on bound projects.
- **FR-344**: Discovery of the resolved workspace on every command MUST follow this priority order, deterministically: (1) an explicit `--workspace <name>` command-line flag; (2) a `TOME_WORKSPACE` environment variable carrying a workspace name; (3) a walk from the current working directory toward the filesystem root looking for a project marker, with the first marker found determining the bound workspace; (4) the seeded `global` workspace as the final fallback. Each step MUST be consulted only if the prior step did not produce a workspace. Both override mechanisms take a workspace name (not a path) and select that named workspace from the central registry. A workspace name supplied to either override MUST exist in the central registry; missing-workspace fails with the dedicated code.
- **FR-345**: The Phase 3 `--global` top-level flag MUST NOT exist in Phase 4. Operating against the global workspace MUST be expressed as `--workspace global` or by being outside any project marker.
- **FR-346**: A project marker that names a workspace not present in the central registry MUST cause the resolving command to exit with a dedicated error pointing the developer at the doctor command. A project marker whose configuration file is empty or malformed MUST also exit with a dedicated error.
- **FR-347**: Workspace name rules: a workspace name MUST consist of one or more characters from the set [letters, digits, hyphen, underscore]; MUST NOT begin or end with a hyphen or underscore; MUST be 64 characters or fewer; MUST NOT be `.`, `..`, or empty. A name that violates these rules MUST cause the workspace-creating command to refuse with a dedicated error. In v1 the only **reserved** name is `global` (FR-405); the parse rule above plus the reservation are the only constraints. Future versions may add reserved names; new reservations are breaking changes covered by the conventional commits / changelog discipline.
- **FR-348**: Every Tome-owned settings or configuration file written under the central root or under a project marker MUST be parsed with strict schema rules (unknown keys rejected) — that is: the global settings file, every workspace's settings file, every project marker's configuration file, the global Tome configuration file, the model registry's manifest file. The strictness boundary established in Phase 2 applies to all Tome-owned inputs. Tome MUST NOT silently accept developer-authored auxiliary keys inside any of these files; developers wishing to store project-local annotations adjacent to a project marker MUST place them outside the marker directory (for example, in their own `.local/` siblings).
- **FR-349**: Third-party files Tome writes into (harness MCP configuration files, harness rules files) MUST be parsed leniently (the strictness boundary's lenient side) and MUST be written back preserving every entry, key, comment, and ordering that Tome did not author. For JSON-format harness configs, an order-preserving map (such as `serde_json` with the `preserve_order` feature) MUST be used. For TOML-format harness configs, a comment- and order-preserving editor (such as the `toml_edit` crate) MUST be used; the plain serialise-deserialise round-trip path (which discards comments) MUST NOT be used for these files. The plan MUST select these libraries explicitly and budget them against the dependency justification rule (NFR-102).

**Catalog refcounting rewrite**

- **FR-360**: The system MUST treat the central database's workspace-catalogs table as the sole source of truth for which workspaces enrol which catalogs. The Phase 3 opt-in workspaces-registry file (the `workspaces.txt` opt-in marker) MUST NOT continue to exist in Phase 4.
- **FR-361**: Catalog reference counting MUST enumerate the workspace-catalogs table to determine which catalog URLs have at least one workspace enrolment. The on-disk catalog clone for a URL MUST be deleted only when no workspace-catalogs row references it.
- **FR-362**: `tome catalog add` MUST operate against the resolved workspace: it MUST insert a workspace-catalogs row for the (resolved workspace, catalog name) pair and MUST clone the URL into the shared catalogs subdirectory only if no other workspace already enrolled the same URL; otherwise it MUST reuse the existing clone.
- **FR-363**: `tome catalog remove` MUST operate against the resolved workspace: it MUST delete the workspace-catalogs row for the (resolved workspace, catalog name) pair and MUST delete the on-disk clone only if no other workspace-catalogs row still references the URL. The Phase 2 refusal to remove a catalog with enabled plugins in the same scope MUST continue to apply, with the cascade-on-`--force` semantics inherited from Phase 2.
- **FR-364**: `tome catalog list` MUST report the resolved workspace's enrolled catalogs only; the central registry's other workspaces' enrolments MUST NOT appear in the output.
- **FR-365**: `tome catalog update` MUST refresh every catalog that any workspace currently enrols, reusing the Phase 2 git-pin discipline. The reindex pass triggered by an update MUST cover every workspace's enabled plugins for any updated catalog, not just the resolved workspace's.
- **FR-366**: `tome catalog remove`'s refcount check (FR-361) and on-disk cache deletion MUST be performed atomically with respect to other writers. The check (`is any other workspace-catalogs row still referencing this URL?`) and the conditional delete (`remove the cache directory if not`) MUST both run inside the advisory lockfile (the same lock taken for the row delete itself). Reads against `workspace_catalogs` from other unrelated commands are NOT serialised by this discipline; only writes that may delete or insert rows referencing the same URL are. A `catalog remove` MUST NOT observe an interleaved `catalog add` for the same URL across this window.
- **FR-367**: Concurrent `catalog remove` invocations targeting the same catalog from two workspaces MUST be serialised by the advisory lockfile. The first invocation to acquire the lock removes its row and (if it becomes the last reference) the cache. The second invocation, upon acquiring the lock, observes its row already removed by its own delete, observes any cache already gone if the first deleted it, and exits cleanly with the same "removed" outcome. The system MUST NOT exhibit cache-leak or use-after-cleanup failure modes under this race.

**Plugin enable/disable rewrite**

- **FR-380**: `tome plugin enable` MUST operate against the resolved workspace and MUST: insert or update one skills row per parsed skill keyed by the (catalog, plugin, skill name) triple (workspace-agnostic, re-embedding only when the content hash changes); insert one workspace-skills row per skill (workspace id, skill id, enabled-at), with the insert idempotent on the existing primary key; trigger summary regeneration for the resolved workspace; trigger integration-sync for any bound project (if Tome is operating in a bound project context; otherwise no integration work).
- **FR-381**: `tome plugin disable` MUST operate against the resolved workspace and MUST delete the workspace-skills rows for every skill in the plugin within that workspace. The shared skills row MUST be retained — another workspace may still enable it. Disable MUST trigger summary regeneration and integration-sync.
- **FR-382**: `tome plugin reindex` (and the catalog-update-triggered reindex pass) MUST operate against the resolved workspace and MUST update each affected skills row in place, retain its workspace-skills row when present, and trigger summary regeneration if any skill's content hash changed in the resolved workspace's enabled set.
- **FR-383**: A skills row with no workspace-skills row referring to it MUST be kept on disk indefinitely (its content hash and embedding remain valid for any future enable). The system MUST NOT prune skills rows in v1; future garbage-collection is a Phase 5+ concern.
- **FR-384**: All Phase 2 atomic-write discipline for the index database MUST continue to apply unchanged: every enable/disable is one transaction; an interrupted enable leaves the database at its prior state.
- **FR-385**: When summary regeneration is triggered by a state-mutating operation (plugin enable, plugin disable, plugin reindex with content-hash changes) and the summarisation pipeline subsequently fails (model missing, model checksum mismatch, model output unusable), the underlying skill-state mutation MUST be retained — the workspace-skills row inserts or deletes MUST be committed before the summariser is invoked. The surrounding command MUST exit with the dedicated summariser-failure code (FR-601, code 20); the doctor command MUST report the workspace's cached summary as stale (with the prior summary still cached, if any) and MUST surface the summariser subsystem as broken until the failure is resolved. This is the "fail forward" rule: the developer's intent (enable / disable a plugin) is honoured; the summary regeneration becomes a follow-up task, not a rollback trigger.

**Workspace lifecycle commands**

- **FR-400**: The system MUST provide a `tome workspace init <name>` command that: validates the name against the rule in FR-347; creates the central workspace directory atomically using the multi-file directory-landing discipline (see FR-410); writes the settings file with the workspace's name as the only required field; writes an empty rules file; inserts a workspaces row carrying created-at and last-used-at timestamps. The command MUST refuse if a workspace by that name already exists. An optional flag MUST be available to seed the workspace's catalog list (catalog-name → URL → pinned reference) from the global workspace's catalog list at the moment of creation; the flag MUST NOT copy any enabled-plugin state and MUST NOT copy the harness declaration. If the global workspace has no enrolled catalogs at the moment of creation, the flag is a documented no-op.
- **FR-401**: The system MUST provide a `tome workspace list` command that reports every workspace in the central registry with: name, count of enrolled catalogs, count of enabled plugins (sum of workspace-skills rows across all enabled-plugin groupings), count of indexed skills (same), count of bound projects, last-used timestamp. Structured form MUST be available.
- **FR-402**: The system MUST provide a `tome workspace info [<name>]` command. With no argument, the command MUST report the currently resolved workspace's details. With a name argument, the command MUST report that workspace's details regardless of where the developer is currently inside. The report MUST include the workspace's central directory path, its enrolled catalogs, its enabled plugins, its bound projects, and its cached summary lengths.
- **FR-403**: The system MUST provide a `tome workspace use <name>` command. The command MUST refuse if the current working directory is the developer's home directory or the filesystem root (any other directory is acceptable as a project root; the developer chooses what counts as a project). The command MUST: refuse if `<name>` does not exist in the central registry, with a dedicated error suggesting `tome workspace init <name>`; under the advisory lockfile (the entire bind operation runs as one critical section), create the project marker directory if it does not exist using the multi-file directory-landing discipline (FR-410); write the project's binding configuration file naming the workspace; copy the workspace's central rules file to the project marker; UPSERT the workspace-projects row keyed on project path (per FR-322 and FR-342) — UPSERT semantics make rebinding correct even when an existing row names a different workspace; run the harness-sync logic for the resolved effective harness list automatically. Re-running with the same name MUST be a no-op except for refreshing the rules file copy. Re-running with a different name MUST rebind via the same UPSERT path. If the harness-sync step fails (for example, a harness MCP config file's parent directory is not writable), the binding INSERT/UPSERT MUST remain committed (forward progress); the command MUST exit with the appropriate sync-error code from the new error set; the developer can re-run sync after addressing the cause, and the doctor command reports the harness drift in the meantime.
- **FR-404**: The system MUST provide a `tome workspace rename <old> <new>` command that: validates `<new>` against the name rule; refuses if `<new>` already exists; under the advisory lockfile, renames the central workspace directory using the same atomic-rename-on-the-same-filesystem semantics as FR-410; updates the workspaces row's name column inside the same transaction that updates every bound project's marker file to name the new workspace; updates each bound project's marker config file using the per-file atomic-write discipline; commits the transaction. If any bound project's directory cannot be found on disk, the rename MUST fail with no state changed (the central directory rename has not yet happened at the point of failure; the database transaction is rolled back). Bound project marker updates within the transaction MUST follow a deterministic order (lexicographic by path) so a partial-failure retry is exact; if a marker update fails mid-loop, the transaction MUST be rolled back and the central directory MUST NOT be renamed. (Recovery via "drop missing bound projects from the central registry" is a Phase 5+ concern.)
- **FR-405**: The system MUST provide a `tome workspace remove <name>` command that: refuses without an override flag if any project is bound to the workspace, with a dedicated error naming each bound project's path; with the override flag, executes the cascade in this **explicit order** under the advisory lockfile: (1) tear down integration in every bound project (run the rules-file-block-removal and MCP-entry-removal logic per harness, per bound project); (2) remove each bound project's marker directory; (3) inside one database transaction, delete the workspace-skills rows, then workspace-catalogs rows, then workspace-projects rows, then the workspaces row for this workspace, and commit; (4) remove the workspace's central directory; (5) refcount-clean any catalog clone no longer referenced by any workspace (per FR-366/FR-367). Failure at any step before (3) MUST leave the central database consistent (no rows have been deleted yet). Failure at (4) or (5) MAY leave an orphaned central directory or an orphaned cache directory; both MUST be detectable by the doctor command and repairable by re-running `workspace remove` (the operation is idempotent past step (3)). The `global` workspace MUST never be removable; the command MUST refuse with a dedicated error before doing any work.
- **FR-406**: The system MUST provide a `tome workspace sync [<name>]` command that copies the workspace's central rules file to every bound project's marker. With no name argument, the command MUST sync every workspace. The command MUST be idempotent and MUST NOT regenerate summaries.
- **FR-407**: The system MUST provide a `tome workspace regen-summary <name>` command that: runs the summarisation pipeline against the workspace's current enabled-plugin state; writes both summaries (short and long) to the workspace's settings file; rewrites the workspace's central rules file from the long summary; runs the sync logic for that workspace in the same invocation.
- **FR-410**: Multi-file directory landings (workspace creation in FR-400, workspace rename in FR-404, project marker creation in FR-403) MUST use the atomic-populated-directory discipline established in Phase 3: stage the new directory in a sibling temp dir on the same filesystem (`tempfile::Builder::tempdir_in` against the target's parent), populate every required file in the staged directory, fsync, consume the auto-cleanup guard (`TempDir::keep`), then perform a single `std::fs::rename` of the staged directory to its final path. For replace-existing semantics (used in `--force` paths), the existing target MUST first be renamed aside to a `.old` sibling; if the staged-rename fails, the `.old` sibling MUST be restored before the command exits. A SIGINT mid-population MUST leave only the staged directory on disk (which is `tempfile`-auto-cleanup-eligible on the next run if `keep()` was not yet called); a SIGINT after `keep()` and before `rename` MUST leave the staged directory orphaned with a documented `.tome.tmp.*` prefix, and the doctor command MUST clean such orphans on `--fix` (this is the documented Phase 4 follow-up for the P3 cleanup gap noted in CONCERNS.md TD-016).
- **FR-411**: The `last_used_at` column on the workspaces table MUST be updated on every **write-path** command invocation that resolves to that workspace (every Phase 1 / Phase 2 / Phase 3 / Phase 4 command that takes the advisory lockfile for its primary work). The update MUST be a single SQL UPDATE inside the same transaction as the command's primary write. Pure read-path commands (`query`, `status`, MCP search-side) MUST NOT update `last_used_at`; the timestamp consequently reports the last time the workspace was written to, not the last time it was read. This trade-off accepts a slightly stale read-side timestamp in exchange for avoiding a write on every read.

**Summarisation model and pipeline**

- **FR-420**: The system MUST bundle, via the same download discipline used for the embedder and reranker, a single instruction-tuned local language model for summarisation. The model registry MUST list this model alongside the embedder and reranker, with the same pinned URL and pinned cryptographic checksum discipline.
- **FR-421**: The summarisation model MUST be lazily loaded — only when a summary regeneration is actually required — and unloaded after use to keep idle resident memory close to the Phase 3 baseline. Eager loading on CLI invocations not requiring summarisation MUST NOT occur. The summariser runtime's process-wide initialisation primitive (the runtime's per-process backend / model-format registry, which most LLM inference C-libraries require and which is not safe to initialise more than once per process) MUST be initialised through a process-lifetime singleton (such as `std::sync::OnceLock`) that lives for the lifetime of the binary; "unload after use" refers to dropping the model context / weight memory, not the runtime backend. The plan MUST verify against the chosen summariser binding's documentation that the singleton-and-context-drop model is the supported usage pattern; if the binding requires backend tear-down between uses, the plan MUST revisit the lazy-load contract.
- **FR-422**: The summarisation pipeline MUST take as input the descriptions of every enabled plugin in the resolved workspace plus the names and descriptions of every indexed skill within those plugins. It MUST produce two outputs from two prompts run sequentially: a short summary, in a documented character window targeted at being safe inside an MCP tool description, and a long summary, in its own documented character window targeted at being a brief workspace overview suitable for an agent rules file.
- **FR-423**: Both summaries MUST be cached in the workspace's settings file under a dedicated section that also records a regeneration timestamp. Regeneration MUST be triggered by: plugin enable, plugin disable, plugin reindex with content-hash changes affecting the resolved workspace's enabled set, catalog update affecting enabled plugins in any workspace (per-workspace trigger), and the explicit regen-summary command.
- **FR-424**: When summarisation is required but the summarisation model is missing or fails its integrity check, the surrounding operation MUST exit with a dedicated summariser-failure error and MUST NOT silently complete without a regenerated summary. The doctor command MUST report the summariser subsystem's state alongside the embedder and reranker.
- **FR-425**: The MCP tool description for the search tool MUST include the resolved workspace's short summary, embedded into a fixed scaffold that names the search tool and explains when to call it. The total description length MUST honour a documented length budget per host LLM family (best-effort target with a warning emitted when exceeded — never a hard refusal to start). The MCP server MUST read the cached short summary from the workspace's settings file at startup; if the cached summary is absent (workspace has never had a successful summarisation pass, or settings file has been hand-edited to remove the section), the server MUST fall back to using the fixed scaffold alone (the tool description still names the tool and explains when to call it; the workspace-specific topic enumeration is simply missing). The MCP server MUST NOT invoke the summariser model in-process to fill the cache — the summariser is sync, the MCP server is async, and crossing that boundary on the MCP startup path violates the sync-boundary discipline. Re-populating the cache is a CLI-side responsibility (the developer runs `tome workspace regen-summary` or any plugin-state-changing command).
- **FR-426**: The workspace's central rules file MUST be written from the long summary as its body. Rules-file delivery to bound projects (file copy under the project marker) is covered separately by sync; this requirement only ties the rules file's body to the long summary.
- **FR-427**: The system MUST NOT provide configuration knobs in v1 to swap the summarisation model for a different one, point at an external API, or alter the prompts. If the bundled model is inadequate in practice, replacement is a Phase 5+ concern.

**Layered settings and composition**

- **FR-440**: Each scope MUST be able to declare a harness list: project (in the project marker's configuration file), workspace (in the workspace's settings file), and global (in the global harness-settings file at the central root). The declaration in each scope MUST be optional.
- **FR-441**: The effective harness list for a project MUST be resolved by walking the scopes in priority order: project, then workspace (the one named in the project's marker), then global. The walk MUST stop at the first scope that declares a harness list; further scopes MUST NOT be consulted unless the first declarer's list explicitly references them via composition.
- **FR-442**: A scope that declares an empty harness list (`harnesses = []`) MUST cause the resolution to stop at that scope with an empty effective list. This is the documented opt-out at that scope.
- **FR-443**: Within a harness list, the following entry forms MUST be accepted: a plain harness name (explicit add); a name prefixed with `!` (explicit remove from the union); the literal `[workspace]` (include the bound workspace's list — valid in project scope only); the literal `[workspaces.<name>]` (include the named workspace's list); the literal `[global]` (include the global list).
- **FR-444**: Resolution MUST expand composition references recursively, computing the union of every reachable scope's directly-declared names, then subtract every name preceded by `!` from the union. Order of entries within an array MUST NOT affect the result.
- **FR-445**: Cycle detection MUST use a depth-first walk tracking visited nodes. A cycle MUST cause a dedicated cycle-detected error naming the chain of references that closes the loop.
- **FR-446**: The `[workspace]` reference MUST be valid in project scope only. Its presence in a workspace's or the global settings file MUST cause a dedicated error during resolution.
- **FR-447**: A composition reference to a non-existent workspace MUST cause a dedicated error during resolution. A composition reference to `[global]` is always valid (global is implicit and always present).
- **FR-448**: `!`-prefixed entries MUST be plain harness names. Any `!`-prefixed value that is not a plain harness name (for example, `![global]` or `![workspaces.foo]`) MUST cause the dedicated composition-error code on resolution.
- **FR-449**: Composition references MUST resolve to the referenced scope's **directly-declared** harness list — the list verbatim as it appears in that scope's settings file. They MUST NOT resolve to the referenced scope's *computed effective* list (which would itself walk priority chains and trigger second-order composition expansion at the reference site). Concretely: a project's harness list of `["[workspace]", "!cursor"]` whose bound workspace has no `harnesses` declaration at all resolves to the empty set (with `cursor` then subtracted from nothing). It does NOT fall through to the global declaration via the workspace's would-be priority walk — the workspace's "directly-declared list" is empty, full stop. Composition is a one-level reference, not a re-entrant resolver.
- **FR-450**: All entry values inside the TOML `harnesses` array MUST be **string** values (TOML string syntax, single- or double-quoted). The bracketed forms `[workspace]`, `[workspaces.<name>]`, and `[global]` are string literals containing brackets, not TOML table headers or arrays. Implementations MUST parse the array elements as strings and pattern-match the literal text; treating the bracketed forms as TOML structure MUST be considered an implementation error.

**Harness modules**

- **FR-460**: The system MUST ship exactly five harness modules in Phase 4: `claude-code`, `codex`, `gemini`, `cursor`, and `opencode`. Any harness name in a settings file that is not one of these five MUST cause a dedicated "harness not supported" error during resolution.
- **FR-461**: Each harness module MUST expose a small documented contract: a stable name; a detection check that reports whether the harness's per-user directory exists on the local machine; a rules-file target that takes the project path and returns the file path to write rules content into, following the rule that an existing rules file the harness already reads is preferred to a default, that the developer-conventional `AGENTS.md` is preferred to a harness-specific default, and that for multi-file rules harnesses the returned path is a Tome-specific file in the harness's rules directory; a rules-file strategy (block-in-existing-file or standalone-file); a **rules-file block-body style** (for block-in-existing-file harnesses only: either `AtInclude` — write the body as an `@`-include pointer to the project's marker rules file copy — or `Inline` — write the full rules content inside the block); an MCP config path that takes the project path and returns the harness's MCP config file path (project-scoped if the harness supports per-project MCP config, global fallback otherwise); an MCP config format (the on-disk format Tome must write — JSON, TOML, etc., paired with the order-preserving editor library required by FR-349); and an MCP config key (the standardised name `tome` for naming continuity across harnesses).
- **FR-462**: The per-harness specific path conventions (default rules-file fallbacks, MCP config locations, scope behaviour) are subject to verification at implementation time against each harness's then-current documentation. Phase 4 ships the *integration pattern* through this contract; the specific paths land during implementation and are documented in the data model that accompanies the plan.

**Rules-file integration**

- **FR-480**: For a harness with strategy `BlockInExistingFile`, the system MUST maintain a single delimited block in the harness's chosen rules file. The block MUST be bounded by the exact byte markers `<!-- tome:begin -->` and `<!-- tome:end -->` (each on its own line). When matching, Tome MUST tolerate trailing whitespace on the marker line (regex equivalent: `^<!-- tome:begin -->\s*$` and `^<!-- tome:end -->\s*$`). When writing, Tome MUST emit each marker followed by exactly one `\n` (line-feed) with no trailing whitespace before the line-feed. The block's body MUST be one of two forms per the harness module's declared block-body style (FR-461): `AtInclude` writes an `@`-include pointer to the project marker's local rules file copy (preferred where the harness supports `@`-style file includes in its rules format, because regenerated content propagates without rewriting the rules file); `Inline` writes the full rules-file body verbatim inside the block (fallback for harnesses without `@`-include support).
- **FR-481**: For a harness with strategy `StandaloneFile`, the system MUST write a complete rules file at the path the harness module specifies, with no delimiters. The contents MUST be the full rules-file body inline; standalone-file harnesses MUST NOT use `@`-include.
- **FR-482**: When more than one harness in the effective list targets the same rules-file path (for example, several harnesses all using `AGENTS.md`), the system MUST write one shared block (or one shared standalone file) at that path. The block's contents MUST be identical to what any single one of those harnesses would have written.
- **FR-483**: Removal of the block (or the standalone file) MUST be governed by whether any harness in the effective list still targets that file. If at least one harness still targets the file, the block MUST be retained; if no harness in the effective list still targets the file, the block MUST be removed (or the standalone file deleted).
- **FR-484**: Contents inside the delimited block MUST be considered Tome-owned. Hand-edited content inside the block MUST be overwritten on the next sync. Contents outside the block in the same rules file MUST be preserved verbatim.

**MCP config integration**

- **FR-500**: For each harness in the effective list, the system MUST write one entry into the harness's MCP config file, keyed `"tome"`. The entry's contents MUST express the command path the harness should launch and an arguments array beginning with the literal `mcp` followed by `--workspace` and the workspace name. The format on disk (JSON object, TOML table, etc.) MUST follow the harness module's declared MCP config format. Writing the entry MUST be a **read-modify-write** of the existing file using the lenient-parse + order-preserving-write discipline of FR-349: every entry, key, comment, and ordering Tome did not author MUST be preserved. If the file does not exist, Tome MUST create it with only the Tome entry plus whatever surrounding scaffolding the harness's format requires (for example, a JSON object with `{"mcpServers": {"tome": ...}}` or the TOML equivalent).
- **FR-501**: Marker logic for ownership of the `"tome"` entry MUST be content-based: an existing entry where the command is `tome` and the first argument is `mcp` MUST be considered Tome-owned and freely updatable (for example, when rebinding rewrites the workspace argument); an existing entry under the same name with a different command or different first argument MUST be considered user-owned and MUST NOT be overwritten without an explicit override flag.
- **FR-502**: An attempt to write a Tome entry into an MCP config file that already contains a user-owned entry named `"tome"` MUST exit with a dedicated "harness clash" error quoting the existing entry's command and first argument; the override flag MUST be available to overwrite.
- **FR-503**: When Tome rewrites a Tome-owned entry (for example, on rebind), any developer-added `env` field on that entry MUST be preserved. The marker comparison MUST ignore `env` entirely; the rewrite MUST only touch `command` and `args`.
- **FR-504**: Removal of the Tome entry MUST be governed by whether the harness is in the effective list. Harnesses no longer in the effective list MUST have their Tome entry removed from their MCP config file on the next sync.
- **FR-505**: The MCP config file's parent directory MUST be created when missing, using the same atomic-write discipline that protects content writes.

**Harness command surface**

- **FR-520**: The system MUST provide a `tome harness` command (no subcommand) that lists every supported harness in tabular form, with columns naming the harness, whether the harness's per-user directory is detected on the local machine, the rules-file target the harness would write to in the current project (if any project is resolved), and the MCP config target.
- **FR-521**: The system MUST provide a `tome harness list [<workspace>]` command. With no argument, the command MUST report the effective harness list for the current project, annotating each entry with the scope (project, workspace, global) and the reference chain that brought it into the list. With a workspace-name argument, the command MUST report that workspace's directly-declared list verbatim without expanding composition references.
- **FR-522**: The system MUST provide a `tome harness use <name> [--scope project|workspace|global]` command. The default scope MUST be project. The command MUST append the named harness to the chosen scope's settings file (creating the harnesses key if absent). After the settings file is updated, the command MUST recompute the effective list for the current project and, if the addition changed that list, run the integration logic for the newly-added harness(es). If the addition did not change the effective list, the command MUST print an informational notice naming which scope was edited and reminding the developer that other projects may need an explicit sync.
- **FR-523**: The system MUST provide a `tome harness remove <name> [--scope project|workspace|global]` command. Behaviour MUST mirror `harness use`: the named harness MUST be removed from the chosen scope's settings file; the effective list MUST be recomputed; if the removal changed the effective list, the integration MUST be torn down for the newly-removed harness(es) in the current project. If the removal did not change the effective list, the command MUST print an informational notice.
- **FR-524**: The system MUST provide a `tome harness info <name>` command that reports, for the named harness: a one-line description; whether the harness is detected on the local machine and at which paths; the rules-file target Tome would write to in the current project; the MCP config target; whether Tome currently has integration written in the current project (inferred from the filesystem at command time); and which of the three settings scopes reference this harness (and how, including composition annotations).
- **FR-525**: The system MUST provide a `tome harness sync` command that recomputes the effective harness list for the current project and reconciles the project's actual filesystem integration state to match (per the sync algorithm specified below). The command MUST be **byte-for-byte idempotent**: when the project's current filesystem state already matches the effective list, no rules-file or MCP-config-file content MUST change between two consecutive invocations (the underlying file MUST NOT be re-renamed; that is, the inode's mtime MUST NOT advance for files whose content is unchanged). The command's stdout MUST report "no changes" in human form (and an empty changes array in structured form) when this idempotent path runs.

**Sync algorithm**

- **FR-540**: The sync algorithm MUST compute the effective harness list `L` for the current project via the layered-settings lookup and composition resolution (FR-441 through FR-447).
- **FR-541**: For each harness in `L`, the algorithm MUST consult the harness module's contract to derive the rules-file target, the rules-file strategy, the MCP config path, and the MCP config format.
- **FR-542**: For the union of rules-file targets across `L`, the algorithm MUST ensure each target file exists with the current Tome content (a delimited block under `BlockInExistingFile` strategy, or a complete standalone file under `StandaloneFile` strategy).
- **FR-543**: For every rules-file path Tome could possibly have written to (every supported harness's rules-file target for the current project), the algorithm MUST detect existing Tome-owned content (a delimited block bounded by Tome's markers, or a standalone file matching a Tome-owned path) and MUST remove it if its target is not in the union from FR-542. Hand-edited content outside any Tome block MUST be left untouched.
- **FR-544**: For each harness in `L`, the algorithm MUST ensure the harness's MCP config file contains the Tome-keyed entry pointing at the project's current workspace binding. An existing Tome-owned entry whose contents drift from the expected shape (for example, a stale workspace argument) MUST be updated in place. An existing user-owned entry named `"tome"` MUST cause a dedicated harness-clash error unless the override flag is passed; with the override, the entry MUST be rewritten.
- **FR-545**: For each supported harness *not* in `L`, the algorithm MUST ensure the harness's MCP config file does NOT contain a Tome-owned entry named `"tome"`. If present, the entry MUST be removed. A user-owned entry named `"tome"` in that harness's MCP config MUST be left untouched.
- **FR-546**: The sync algorithm MUST infer all integration state from the filesystem at command time. Tome MUST NOT maintain a sidecar state file or a separate database table tracking which harness in which project is currently integrated. The Tome-block markers and the MCP-entry marker logic are the source of truth.
- **FR-547**: The sync algorithm MUST emit a summary of changes per invocation: added integrations, removed integrations, leave-alones, and (in the structured output form) the underlying decisions per harness.

**Doctor extensions**

- **FR-560**: The doctor command MUST report, in addition to every Phase 3 subsystem, the following Phase 4 subsystems: the project-to-workspace binding (present, well-formed, names an existing workspace); the project's local rules-file copy (present, matches the workspace's central rules file's content); the effective harness list for the current project (per FR-441); per harness in the effective list, the rules-file integration state (block-or-standalone-file present and current) and the MCP config integration state (entry present, Tome-owned, and pointing at the correct workspace); and detection of any other supported harness whose per-user directory is present locally but not configured.
- **FR-561**: For each Phase 4 subsystem, the doctor command MUST classify findings as healthy, drifted (non-fatal repair available), broken (fatal — but the doctor itself never crashes), or "not applicable" (for cases like the harness subsystem when the effective list is empty by design). Visually distinct markers MUST be used per Phase 3 doctor discipline.
- **FR-562**: The doctor command's fix flag MUST repair the following Phase 4 classes: missing or stale project-local rules-file copy (re-copy from the workspace's central file); missing Tome rules-file block in a harness's chosen rules file (re-run harness-sync for that harness); missing Tome MCP entry in a harness's MCP config (re-run harness-sync for that harness). The fix flag MUST NOT, even with an override, repair: a broken binding to a non-existent workspace (requires explicit developer choice — rebind or recreate); a user-owned `"tome"` MCP entry (requires the override flag in combination with the fix flag).
- **FR-563**: The doctor command MUST continue to honour the Phase 3 read-only-by-default discipline (no mutation unless the fix flag is passed) and MUST continue to never perform destructive repairs automatically.
- **FR-564**: From outside any project marker (and without overrides), the doctor command MUST resolve the `global` workspace and MUST report no project binding. The harness subsystems MUST report the global effective list (per the layered lookup that stops at global from the project's perspective; from outside a project, the effective list comes from global directly).

**Errors and exit codes**

- **FR-600**: The closed-error-set principle established in Phase 1 / Phase 2 / Phase 3 MUST continue to hold. There MUST NOT be a generic "other" error variant. Every new Phase 4 failure mode MUST be a named variant with its own dedicated exit code.
- **FR-601**: The system MUST extend its closed error set with the following new variants and exit codes:
  - **Code 13**: workspace not found. Reachable from: an explicit `--workspace <name>` whose name has no row in the central registry; the `TOME_WORKSPACE` env var carrying a missing name; a project marker that names a workspace not in the central registry; a composition reference to a missing workspace (FR-447).
  - **Code 14**: workspace already exists (`tome workspace init` against an existing name).
  - **Code 15**: workspace name invalid (the candidate name violates FR-347, including the reserved-name check for `global` on init).
  - **Code 16**: workspace has bound projects (`tome workspace remove` without the override).
  - **Code 17**: composition error. Reachable from: a cycle in the composition graph (FR-445); `[workspace]` reference appearing in a workspace or global settings file (FR-446); a `!`-prefixed value that is not a plain harness name (FR-448, FR-450 violation). The error message MUST identify which composition-error sub-class triggered it.
  - **Code 18**: harness not supported (a settings file references a harness name not in the five shipped modules).
  - **Code 19**: harness clash (an existing `"tome"` MCP entry is user-owned and the override flag was not passed; FR-502).
  - **Code 20**: summariser failure (model missing at the point summarisation is required, model checksum mismatch at use time, model output unparsable or empty, or summariser load failure). NOTE: a summary that exceeds the documented length window for the short summary is a non-fatal warning per FR-425, NOT a code-20 failure.
- **FR-602**: The following Phase 4 failure modes reuse existing Phase 1 / Phase 2 / Phase 3 variants and exit codes; new variants are NOT introduced for them:
  - **Project marker malformed** (FR-346, second sentence): reuse code 70 (`WorkspaceMalformed`, the Phase 3 variant; semantics widen to cover the Phase 4 marker-as-binding-pointer model).
  - **Workspace rename precondition: bound project directory missing on disk** (FR-404): reuse code 70 (the central registry's recorded binding is malformed relative to the filesystem; same semantic class).
  - **Per-user state directory unwritable / read-only** (Edge Cases, "Per-user state directory is read-only"): reuse code 7 (`Io`).
  - **Bind step succeeds but harness-sync fails inside `tome workspace use`** (FR-403): the binding remains committed; the command exits with the relevant new code from this set (typically code 19 for harness clash) or code 7 (Io). It MUST NOT introduce a "partial bind" code; doctor reports the harness drift.
- **FR-603**: All Phase 1, Phase 2, and Phase 3 exit codes and error variants MUST continue to behave as before. The Phase 3 closed-error set is the baseline; Phase 4 extends it by the eight codes in FR-601 and reuses the variants listed in FR-602.

### Key Entities

- **Tome root directory**: a single per-user directory at `<home>/.tome/` containing every Tome-owned file. Replaces the Phase 1 / Phase 2 / Phase 3 XDG-style scattered layout.
- **Central index database**: the single SQLite database at `<root>/index.db`. Holds every relational table (workspaces, skills, vector embeddings, workspace-skills, workspace-catalogs, workspace-projects, schema metadata). Replaces the Phase 3 per-workspace databases. Carries one schema-version row.
- **Workspace**: a named first-class object stored at `<root>/workspaces/<name>/`. Carries the workspace's own settings file and its own rules file. The `global` workspace is created on first bootstrap and cannot be removed; all other workspaces are user-created.
- **Workspace settings file**: a strict TOML file at `<root>/workspaces/<name>/settings.toml`. Declares the workspace's enrolled catalogs (mirrored into the central database), the workspace's harness declaration, and the workspace's cached summaries (short and long).
- **Workspace rules file**: a Markdown file at `<root>/workspaces/<name>/RULES.md` whose body is the long summary. Copied to every bound project's marker as the source of truth for the agent's rules content.
- **Project marker**: a directory named `.tome/` inside a project's root, holding at most two files: a binding configuration file naming the workspace this project is bound to, and a copy of that workspace's rules file. Absence of a project marker means the developer is operating against the global workspace.
- **Project marker configuration file**: a strict TOML file inside the project marker. Declares the bound workspace name and optionally a project-scope harness declaration.
- **Project marker rules file copy**: a Markdown file inside the project marker. Identical content to the workspace's central rules file at the most recent sync.
- **Effective harness list**: the resolved set of harnesses for a project, computed from the layered scope walk and composition expansion at command time. Not stored; recomputed per command.
- **Harness module**: a Tome code module per supported harness exposing the documented contract (detection, rules-file target, rules-file strategy, MCP config path, MCP config format, MCP config key).
- **Tome-owned MCP entry**: an entry in a harness's MCP config keyed `"tome"` whose `command` is the literal `tome` and whose first argument is the literal `mcp`. Distinguished from a user-owned entry sharing the same name by these content fields.
- **Tome rules-file block**: a delimited range inside a harness's chosen rules file. Bounded by stable Tome markers. Body is either an `@`-include pointer or inline rules content. Fully Tome-owned; hand edits inside it are overwritten on sync. Content outside the block is preserved verbatim.
- **Tome standalone rules file**: a Markdown file Tome owns entirely in harnesses whose rules-file strategy is `StandaloneFile`. Path determined by the harness module's contract.
- **Summarisation model**: an instruction-tuned local language model bundled via the model registry. Downloaded at runtime; checksum-pinned; lazily loaded only when summary regeneration is required; unloaded after each regeneration.
- **Short summary**: a natural-language string in a documented character window suitable for embedding into an MCP tool description. Cached in the workspace's settings file; regenerated whenever the workspace's enabled-plugin state changes.
- **Long summary**: a natural-language string in its own documented character window suitable for the body of an agent rules file. Cached in the same settings file; regenerated alongside the short summary.

## Assumptions

- Phase 1 catalog management, Phase 2 plugin lifecycle, query, models, reindex, status, and version surfaces, and Phase 3 MCP server, workspaces, doctor, and schema-migration framework are all in place. Phase 4 refactors their on-disk shape and storage backend but retains every behavioural contract beyond what FR-300 through FR-580 explicitly redefine.
- The Phase 1 closed `TomeError` enum, atomic-write discipline, advisory-lockfile discipline, and credential-scrubbing discipline carry forward unchanged. Phase 4 extends them; it does not relax them.
- The Phase 3 forward-migration framework lands with zero registered migrations; Phase 4 registers its first production migration. The framework's atomic-rollback discipline is unchanged.
- The summarisation model is bundled and downloaded at runtime per the same model-registry discipline as the embedder and reranker. The specific model chosen is Qwen2.5-0.5B-Instruct, GGUF INT4 quantisation, served via `llama-cpp-2` Rust bindings, per the PRD's resolved decisions. The model artefact's on-disk footprint is approximately 400 MB; this is the third entry in `<root>/models/`.
- The MCP server's stdio transport and the two tools (`search_skills`, `get_skill`) are unchanged from Phase 3. The search tool's description string is extended to include the resolved workspace's short summary, embedded into a fixed scaffold, per FR-425.
- Workspaces are an explicit opt-in concept since Phase 3, and Phase 4 preserves that property: developers who never run `tome workspace init` or `tome workspace use` continue to operate against the seeded `global` workspace. The Phase 3 explicit `--global` flag is removed; operating "against global" becomes `--workspace global` (or, more commonly, being outside any project marker).
- Per-harness rules-file paths, MCP config locations, and per-project versus global scope behaviour for each of the five supported harnesses are accurate as of the PRD's drafting date but are verified against each harness's then-current documentation at implementation time. Phase 4 ships the *integration pattern* and the *harness-module contract*; specific paths land during implementation and are documented in the data model that accompanies the plan.
- `@`-style file include support varies per harness's rules-file format. The harness module's contract carries a per-harness decision for whether the rules-file block body is an `@`-include or inline; verifying that decision against each harness's docs is an implementation-time task.
- WSL2 against the Windows-mounted filesystem (`/mnt/c/...`) is out of scope; the Windows-native filesystem on a WSL2 install introduces file-permission semantics that interact badly with the atomic-write and 0600-mode discipline. The supported case is the WSL2 native filesystem (`~/...` inside WSL2). Native Windows and WSL1 remain out of scope from prior phases.
- The summarisation model's two prompts (one for the short summary, one for the long summary) are owned by Tome and are NOT user-configurable in v1. Phase 5+ may add prompt-customisation surfaces if dogfooding demonstrates need.
- No automated migration of pre-Phase-4 user data is provided. The PRD explicitly accepts that pre-release users wipe their old data directories by hand. The doctor command does not detect or surface pre-Phase-4 layouts.
- The current constitution (v1.2.0) carries an Operational Constraint named "Paths" that states "XDG-aware via `directories`. Never hardcode `~/.tome`." Phase 4's path-consolidation requirements (FR-300 through FR-303) directly contradict that constraint. Per the constitution's governance rule ("Where it conflicts with PRD detail, the constitution wins on *how* and the PRD wins on *what*"), the PRD's architectural correction takes precedence on *what* the on-disk layout should be — but the operational constraint itself MUST be amended before Phase 4 implementation lands. The plan's first deliverable MUST include a constitution amendment (v1.3.0) that rewrites the Paths operational constraint to: "Tome-owned paths resolve under `<home>/.tome/`. The home directory is resolved via a portable mechanism. Cache directories are content-addressed (sha256 of source URL) to prevent collisions." The amendment requires a PR, a brief rationale, green CI, and the `Last Amended` date bumped (per Governance §Amendments).

## Out of Scope (Phase 4)

- Translation of commands, agents, or hooks across harnesses. The cross-harness integration in Phase 4 is configuration only: rules-file blocks and MCP config entries. No SKILL files, command files, agent files, or hook scripts are copied or transformed.
- Additional harnesses beyond the five listed (Claude Code, Codex CLI, Gemini CLI, Cursor, OpenCode). Adding Cline, Goose, Aider, or others is a Phase 5+ concern; it requires only a new harness module against the existing contract.

- Multi-tenant or shared MCP servers serving more than one client.
- Authentication, authorization, or remote access for the MCP server.
- Escape-hatch configuration knobs for the summarisation model: no model swap, no external API endpoint, no per-workspace or per-project prompt customisation.
- Cross-machine workspace synchronisation tooling. The architecture allows it (a workspace directory under `<root>/workspaces/<name>/` can be Dropbox'd or Git-tracked by the developer at their own discretion) but Tome ships no tooling, no conflict resolution, and no on-disk discipline to support such usage in v1.
- Automated migration of pre-Phase-4 on-disk data. Pre-release users wipe their old per-user data directories by hand.
- Native Windows, WSL1, or WSL2 against the Windows-mounted filesystem. Supported environments are macOS, Linux, and WSL2 against the WSL native filesystem.
- Garbage collection of unreferenced skills rows. A skills row whose every workspace-skills row has been deleted is retained on disk indefinitely in Phase 4; pruning is a Phase 5+ concern.
- A workspace-level command to drop bound projects whose directories have been deleted from disk. The Phase 4 lifecycle treats this as a precondition for `tome workspace rename`; recovery is a manual edit of the central database in v1, with a developer-facing command shipping in Phase 5+ if needed.
- Plugin authoring tools (new, lint, validate, publish).
- Per-harness MCP entry annotations beyond plain `command` / `args` / `env` (no caching hints, no permission scopes, no resource subscriptions).
- Pre-emptive validation of a harness's own MCP configuration file structure. Tome reads what the harness already declares as the format and writes its entry into the keyed slot; it does not lint the harness's other entries.
