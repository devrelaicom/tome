## What Worked Well

- **`llama-cpp-2` integration landed in one slice** without API research detours. US4.a's brief flagged llama-cpp-2 as "unknown territory"; the implementing agent worked through the API in <30 min and surfaced 4 surprises that informed US4.b (encoding_rs not re-exported, `sample(&ctx, -1)` magic constant, no KV-cache reset hook, `add_sequence`/`logits_all = false` semantics). Documenting surprises in the PR body propagated forward.
- **Salvageable partial dispatch (US4.d-1)** — the agent reported "still running" but the WIP turned out to be complete (4 blockers + 9 majors applied) and passed all gates. Same recovery pattern as US3.a + US3.b + US3.d-1: inspect with `git status`, run `cargo test/clippy/fmt`, commit if green.
- **Parallel reviewer pass (4 agents)** caught the same critical issue all four ways: production registry shipped the placeholder SHA-256 (C-B1 contract / S-M3 security / R-B1 rust / T-M-related test). The fix was bounded: download real model + compute hash + record + add regression test. The reviewer pass paid for itself — without it, US4 would have shipped non-functional.
- **`SummariserOverrideGuard` test fixture** (US4.b) — process-global slot mirrors the `HARNESS_MODULES_OVERRIDE` + `MIGRATIONS_OVERRIDE` patterns. RAII guard clears on Drop. Test-only by convention; production paths construct `LlamaSummariser::new()` directly.
- **FR-385 forward-progress** correctly implemented across all 5 trigger sites (enable/disable/reindex/catalog-update/catalog-remove): mutation tx commits FIRST, summariser invoked OUTSIDE the tx, failure exits 24 with mutation retained. Doctor (US5) will detect the stale-summary state.
- **Cached `LlamaModel` + verify-once-at-`new()`** (US4.d-1 S-M4 fix): expensive 400 MB SHA verify + load happens once; per-trigger `summarise()` reuses the cached model and only creates a fresh context. Eliminates the per-trigger DoS + small TOCTOU window.

## What Didn't Work

- **F6 → US4.a registry placeholder oversight** (the C-B1 root cause): the F6 skeleton shipped a placeholder SHA-256 with a comment "US4.a flips this to the real digest". US4.a's brief said "audit and wire `tome models` extension to list the summariser as the third entry"; the agent did, but never actually computed the real SHA-256. Symptom: `tome workspace regen-summary` returned `ModelMissing` on every run because the SHA verify fired against the all-zero placeholder. The 4-reviewer pass caught it; without that, US4 would have shipped silently broken. **Lesson**: when a skeleton ships a placeholder constant with a "must flip later" comment, the slice that's supposed to flip it MUST include the manual step explicitly in its brief and the resulting commit MUST have a regression test pinning the non-placeholder state. The US4.d-1 fix added `tests/summariser_registry_no_placeholder.rs` (3 tests) to close this for good.
- **Exit-code 20 vs 24 doc drift** (C-B2): early Phase 4 drafts said `SummariserFailure → 20`; F3 moved it to 24 to avoid Phase 2 PluginNotFound collision; docs + docstrings in three files didn't get the memo. Reader-trap. Fixed in US4.d-1 by grep + scrub. **Lesson**: when shifting an exit code mid-design, grep the entire repo (`src/`, `specs/`, `tests/`, `docs/`) and not just the type definition.
- **`LONG_MAX_CHARS` defined in two places with different values** (C-B3 / R-B1): `summarise/prompts.rs` had 2400; `workspace/regen_summary.rs` had 2500. Long summaries in 2401–2500 warned ONCE; 2501+ warned TWICE. Two reviewers caught it. Fixed by consolidating both into `src/summarise/mod.rs` as the single source. **Lesson**: any constant naming a max/min/limit that's consumed in two places is a future bug; promote to `mod.rs` of the owning module at the first cross-module consumer.
- **`SummariserOverrideGuard` was dead code at end of US4.b** (T-B1): the override slot existed; the trigger callers consulted it; but no test actually installed an override + invoked the production trigger code path. Every test called `regenerate_for_trigger_with_summariser` directly, bypassing the override. The 4-agent test reviewer caught this; the fix added `tests/summariser_triggers_end_to_end.rs` (2 tests). **Lesson**: when adding a test-injection seam, the SAME PR must include at least one test that exercises the seam through the production code path. Don't ship a seam that the production path consults if no test verifies the consultation works.

## Workarounds & Solutions

**Real SHA-256 for Qwen GGUF** (US4.d-1 C-B1 fix): downloaded the model file from the canonical Hugging Face URL via curl, computed SHA-256 via `shasum -a 256`. Result: `74a4da8c9fdbcd15bd1f6d01d621410d31c6fc00986f5eb687824e7b93d7a9db` (491,400,032 bytes). Recorded in `src/embedding/registry.rs` AND `src/summarise/registry.rs`. Regression test `tests/summariser_registry_no_placeholder.rs` pins the constant. Generalisable: when a registry entry needs a real digest, document the curl+shasum recipe in the registry doc comment so future contributors can reproduce.

**`tracing_subscriber` in-test capture layer** (US4.d-1 T-M5 fix): for tests that need to verify `tracing::warn!` actually fires, installed a bespoke layer with a thread-local channel. Avoided the `tracing_test` dep + `callsite::interest` cache hazard from `with_default`. Pattern: lightweight test-only subscriber for warn-capture assertions. Reusable for any future test that needs to assert a specific `tracing` event was emitted.

**Cached `LlamaModel` + per-call `LlamaContext`** (US4.d-1 S-M4 fix): `LlamaSummariser` stores `model: LlamaModel`; the SHA-256 verify + model load happens once in `new()`. `summarise()` creates a fresh `LlamaContext` per pass (llama-cpp-2 has no KV-cache reset hook). Verified upstream `unsafe impl Send + Sync` for `LlamaModel` satisfies the `Summariser: Send + Sync` bound. Generalisable: expensive immutable resources cache at constructor time.

**`PoisonError::into_inner` for OnceLock-wrapped mutex** (US4.d-1 R-M7 fix): `summarise::backend()` was `?`-bubbling poison via `mutex.lock()?`. A single poisoning event would brick the process for the OS-process lifetime. Fix: `mutex.lock().unwrap_or_else(|e| e.into_inner())`. Generalisable: poisoning a once-init resource shouldn't be a one-way door.

**Pragmatic carve-out: `ModelMissing` silent no-op in trigger callers** (US4.b decision + US4.d-1 C-M2 contract amendment): the bundled LLM is on-demand (FR-420). If a user hasn't downloaded the summariser model, every `tome plugin enable` shouldn't hard-fail with exit 24 — the implicit trigger is best-effort. The explicit `tome workspace regen-summary` surface STILL hard-fails (it's the user-named feature). Pattern: when a feature is on-demand, the implicit trigger paths are best-effort; the explicit invocation surface (the command that names the feature) MUST hard-fail to surface the missing dependency.

**`tome catalog remove --force` cascade now invokes the summariser trigger** (US4.d-1 R-M6 fix): the cascade-disable path was the ONE workspace_skills mutation site that didn't trigger summary regeneration. Effect: workspace's cached summary still mentioned skills that no longer exist. Fix: after cascade-disable commits, iterate affected workspaces + invoke `regenerate_for_trigger` per workspace.

## Patterns & Code

- **Process-global override slot + RAII guard + Drop-cleans-slot** (`SummariserOverrideGuard`, `HarnessModulesGuard`, `MigrationsGuard`) — established pattern for test-injection of trait objects into production-only-consults-the-slot code paths.
- **OnceLock-wrapped resource with poison recovery** (`backend()` US4.d-1 R-M7) — replace `?`-bubble with `into_inner()` recovery.
- **Cache expensive immutable resources at constructor time** (`LlamaSummariser` US4.d-1 S-M4).
- **Single source of truth for cross-module constants** (US4.d-1 C-B3 fix consolidated `SHORT_MAX_CHARS`/`LONG_MAX_CHARS` into `src/summarise/mod.rs`).
- **Pragmatic implicit-vs-explicit trigger carve-out** (US4.d-1 C-M2): on-demand features' implicit triggers are best-effort; explicit user-named surfaces hard-fail.
- **Forward-progress invariant (FR-385) wired across 5 trigger sites**: mutation tx commits BEFORE summariser invocation; failure exits 24 with mutation retained; doctor detects stale state.
- **Bespoke `tracing_subscriber` layer for warn-capture tests** (US4.d-1 T-M5) — avoids `tracing_test` dep + `callsite::interest` cache hazard.
- **Regression test for placeholder-constant invariants** (US4.d-1 C-B1): `tests/summariser_registry_no_placeholder.rs` (3 tests) asserts the SHA-256 is never the all-zero placeholder. Generalisable to any "must never be the placeholder value" invariant.

## For Next Time

- **Skeleton-to-production wiring needs an explicit checklist**: each Foundational slice that ships a placeholder constant should also ship a "must-flip" tracking issue. When the consuming slice opens, its FIRST commit should grep for "TODO: flip" / "PLACEHOLDER" / "all-zero" in its scope and either flip them or document why not. Same lesson as US3's F8 → US3.a `StubScope` audit.
- **Reviewer pass for any slice that consumes a Foundational skeleton must explicitly verify the wiring** — not just "does the skeleton's API work" but "does production use the real value, not the placeholder". US4 caught this via 4-agent pass; we shouldn't rely on the reviewer pass to catch foundational-wiring gaps. Add to the slice-opening checklist.
- **CI ubuntu flake observation continues** (PR #97 ubuntu stable hit `collect2: fatal error: ld terminated with signal 7 [Bus error]` once; retry passed). 2-in-N pattern. Worth a small follow-up issue for "ubuntu runner CI hardening" — possibly `ubuntu-24.04-large` for both jobs, or split `cargo test` to reduce per-link memory load (the linker is OOM-ing on the heavy llama-cpp-sys-2 + onnxruntime co-link).
- **Length-window constants should live in `src/<module>/mod.rs` as `pub const`** at the FIRST cross-module consumer; don't redefine in `prompts.rs` and `regen_summary.rs`. US4.d-1 collapsed both to `src/summarise/mod.rs`. Future: when adding a numeric constraint constant, audit cross-module consumption + promote pre-emptively.
- **End-to-end test for any new test-injection seam** in the SAME PR that introduces the seam. US4.b shipped `SummariserOverrideGuard` without an end-to-end test that uses it from the production code path; US4.d-1 had to backfill. Pattern: "if you add a `XYZ_OVERRIDE` slot, the same PR includes at least one `tests/*_end_to_end.rs` test that installs an override + invokes the production code path that consults the slot".
- **Defer-list consolidation finally** (third US in a row): US1 + US2 + US3 + US4 each defer ~8 items. Build a single `specs/004-phase-4-refactor-harnesses/followups.md` at the start of US5 so the Polish phase has one place to read.
