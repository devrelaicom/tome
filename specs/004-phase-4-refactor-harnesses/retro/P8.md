## What Worked Well

- **Phase-wide 4-reviewer pass** surfaced 3 blockers + 43 majors + 50 minors/nits in a single coordinated parallel run. All four reviewers wrote to `/tmp/tome-phase4-polish-*.md`, then the parent consolidated to `findings.md` + `disposition.md`. The phase-level view caught issues per-US reviewers missed (e.g. R-M7 SQL repetition across 9 sites; T-M5 install_synthetic drop-order discipline duplicated across 4 files; S-M1 accumulated 26+ `unbounded read_to_string` sites).
- **Disposition.md as the routing document** worked. Every major finding got an explicit row mapping to a Polish PR or to a deferred-to-follow-up issue. Polish then became mechanical execution per row — no surprises, no scope creep, no item dropped on the floor.
- **Sub-agent dispatch for high-LOC refactors** (PR-C bigger items, PR-D test scaffolding, PR-E security hardening). Each sub-agent received a fully-scoped brief with file paths + exact patterns. Reports back as a structured commit list + final test count. Parent merged the branch then opened the PR. Reusable for any future Polish-phase work.
- **Quick-win commits first** (PR-C 4 quick wins: R-M3, R-M8, R-M12, C-M9) before dispatching the sub-agent for the 5 bigger refactors. Kept velocity high; parent kept stable checkpoint while sub-agent worked.
- **CI green throughout** except one `ld signal 7` ubuntu transient on PR-F (per handover: retry once, passed second time). No real CI failures across 6 polish PRs.
- **Branch chaining** (PR-A → PR-B → PR-C off each other, then rebase onto main when prior PR merged) — fast feedback loop. Force-push-with-lease handled the rebase cleanly.

## What Didn't Work

- **C-M3 (SyncOutcome shape) deferred** mid-PR-C. The disposition called for "align impl to contract" but the impl shape `{added, updated, removed, leave_alones, decisions}` had shipped through 5 user stories. Rolling that back would be net-negative; aligning the contract to the impl is a 5-line edit. Sub-agent flagged this at handover; parent chose to defer rather than do the risky direction. **Lesson**: when a wire-shape divergence is caught at Polish, the burden-of-proof flips. Impl is stable; contract is the document. If impl is reasonable, update the contract.
- **`git stash + checkout main + pull --ff-only` ceremony** ran 3 times because of an untracked file in the agent-foundry directory. Could have been avoided by gitignoring it or using `git restore --staged` properly. Minor friction; documented for next time.
- **Force-push round-trip latency** felt slow during PR-E rebase. The push completed but `git ls-remote` showed the old SHA for ~10 seconds. Polled until it caught up. Not a process problem but worth being aware of when chaining time-sensitive rebases.
- **PR-D sub-agent ran ~22 min** (the long tail of 4 test bundles). Sub-agents on test scaffolding are slower than on production-code refactors because of the more-test → more-iteration loop. **Lesson**: dispatch test work for the longer-running PRs even if they look smaller in LOC.

## Workarounds & Solutions

- **Sub-agent brief budget cap of ~6000 tokens** for the final summary worked well — kept the parent's context consumption bounded. Future Polish phases should reuse this convention.
- **Per-PR test count progression** (916 → 919 → 919 → 943 → 954 → 954 → 954 → 954) is a clean signal of work landed. Recording it in each commit message + PR body made the trajectory auditable.
- **Disposition rows include the agent-discoverable artifact** ("Apply: rename …", "Apply: add … to tests/X.rs"). Sub-agents could read disposition.md directly to understand scope; no follow-up clarification round-trips.
- **`HomeGuard` + `HOME_MUTEX` consolidation** (T-M8): the 9 test files that already used the pattern were unaffected; `paths_phase2.rs` + `paths_phase3.rs` migrated cleanly. 2 `unsafe` blocks + 40 lines removed.
- **`util::bounded_read_to_string` per-class caps** (S-M1): the 26 callsite migration was the biggest single Polish change. Per-class caps (`TOME_CONFIG_MAX = 1 MiB`, `PLUGIN_MANIFEST_MAX = 256 KiB`, `HARNESS_MCP_MAX = 1 MiB`, `HARNESS_RULES_MAX = 4 MiB`) keep the policy named at the call site.

## Packages & Dependencies

- No new dependencies in Polish phase. PR-E's `util::bounded_read_to_string` uses only `std::fs::metadata` + `std::fs::File::open` + `Read::read_to_string`.
- The constitution v1.2.0 50 MiB binary cap held (final size 26.32 MiB on macOS arm64; recorded in `RELEASE-BINARY-SIZE.md` per T419).

## Patterns & Code

- **Per-class size caps + helper-on-every-read** (PR-E S-M1): `util::bounded_read_to_string(path, max_bytes)` accepted by every Tome-owned read site. Constants live in `util::io` so the policy is named once, applied many times. Reusable pattern for any future "every read needs the same gate" surface.
- **Symlink refusal at every entry point** (PR-E S-M2, S-M6): `symlink_metadata().is_symlink()` check before any `std::fs::rename` / `remove_dir_all` / `OpenOptions::write`. The pattern composes across atomic-write helpers (catalog::store::write_atomic, harness::rules_file::atomic_write, harness::mcp_config::atomic_write), the directory landing helper (`util::atomic_dir`), and the orphan sweep (`doctor::orphan_cleanup`). Each carries the same 6-line guard; consolidation owed (R-M1) is deferred to v0.5.0.
- **`home_root()` validation chain** (PR-E S-M7): `env::var_os` → `is_absolute` check → `canonicalize` → error on relative or missing. Two callsites (paths.rs + commands/harness/mod.rs) carry the same logic. **Lesson**: when an environment-derived value participates in path construction, validate at the boundary.
- **Typed `Subsystem` enum with byte-stable wire** carried forward into Polish (PR-C C-M9 data-model alignment). The data-model.md spec updates to match impl when impl is the more ergonomic shape.
- **Coalesce contract-doc fixes** (PR-B C-B1): exit code 20 → 24 swept through `spec.md` × 3 sites + 3 contract docs. Quick to apply, but the brittleness (3 contract docs disagreeing for ~2 weeks) is worth a process note: when an exit code reconciles after a closed-set discovery, sweep ALL contract documents in the same commit, not just the closest one.
- **InitOutcome field rename + add `id`** (PR-B C-B2): the missing `id` field was load-bearing for downstream tooling. Catching this at Polish caught it before v0.4.0 shipped. **Lesson**: byte-stable JSON pin tests (`tests/workspace_init_json_shape.rs`) should pin every field name + order from day-one of the data type, not later.

## For Next Time

- **Pre-emptive contract-vs-impl wire-shape audit** as part of every user story closeout, not just Polish. PR-D added 4 new JSON wire-shape pins (EffectiveEntry, AsWrittenOutcome, SyncOutcome, WorkspaceCatalogEntry); if those had landed in the original US slices, the C-B2 + C-M3 + C-M9 drifts would have been caught at land-time. **Concrete recommendation**: every `pub struct *Outcome` or `pub struct *Report` ships with a `tests/<name>_json_shape.rs` byte-stable pin in the SAME PR that introduces the type.
- **Mid-phase mapper rotation** (recommendation from US3 retro carried forward): `/sdd:map incremental` once per Foundational slice + once per US closeout = 6 runs in Phase 4. Worked well at catching code-doc drift. Polish phase ran one final pass (PR-G); a Phase 5+ mid-phase rotation would catch the consolidation duplications earlier (the 9-site SQL repeat in R-M7, the 4-site install_synthetic in T-M5).
- **`util::bounded_read_to_string` is a v0.5.0 candidate for further hardening** — currently maps over-cap to `Io(InvalidInput)`. A dedicated `TomeError::FileTooLarge { path, size, cap }` variant would give a cleaner closed-set landing.
- **Force-push remote-state polling** could use a 5-second hard floor before retry to avoid the busy-poll feel observed during PR-E rebase. Not a real performance problem but cleaner UX.
- **Polish phase budget**: 6 production PRs + 1 docs-only closeout PR + 1 reviewer findings PR = 8 PRs total, matching Phase 3's Polish phase exactly. Future phases can budget this size with confidence.
- **Deferred items consolidation**: ~30 items deferred from Polish to follow-up. `disposition.md` § "Deferred to follow-up issue" is the canonical list. Worth tracking as a v0.5.0 backlog in a single GitHub issue once Phase 4 ships.
- **JSON wire-shape pin discipline**: every `#[derive(Serialize)]` struct that lands in a CLI `--json` envelope needs a byte-stable pin test from day one. The Phase 3 pattern (`workspace_info_json_shape.rs`) was right; Phase 4 should have followed it for every new outcome type instead of catching it at Polish.
