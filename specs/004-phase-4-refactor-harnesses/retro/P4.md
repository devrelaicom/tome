## What Worked Well

- **Brief-cap discipline held**: each of US2.a-1, US2.a-2, US2.b, US2.c, US2.d-1 dispatched as a single ≤ 8 KB brief that the agent completed without timing out (~10-30 min per agent). Total US2 surface across 6 PRs ≈ 19 production-source files + 16 test files; no agent had to be re-dispatched mid-flight.
- **Parallel reviewer pass (4 agents)** mirrored the US1.d-2a pattern. Total wall-clock for the review pass was the slowest single agent (~7 min for the test reviewer); findings consolidated into `review/us2-findings.md` + `us2-disposition.md` in one main-thread pass. 4 blockers + 23 majors surfaced; 4/4 blockers + 11/23 majors applied in US2.d-1.
- **Single-source-of-truth atomic-write helper (US2.d-1 S2-M1+M2 fix)**: lifting mode preservation + symlink refusal into `catalog::store::write_atomic` eliminated the divergence between harness writers (US1.d-2a hardening) and workspace writers (US2 callers). Net effect: every file write outside `src/harness/` now inherits the hardening for free.
- **Per-project effective-list narrowing (US2.d-1 C-B1 fix)**: `workspace remove`'s cascade now consults `settings::resolver::resolve_effective_list` per project, matching the contract step-1 narrative. Forced the test fixture in `workspace_remove_cascade.rs` to declare `harnesses = ["claude-code"]` in the marker — a useful documentation side-effect.
- **`toml_edit::DocumentMut` for marker preservation (US2.d-1 T-B1 fix)**: `rename` no longer clobbers the optional `harnesses` field in bound-project markers. Surgical edit pattern carries forward from US1's `harness::mcp_config` use.
- **JSON wire-shape pins for every Outcome (US2.d-1 T-M1 fix)**: 4 new `tests/workspace_*_json_shape.rs` files prevent `#[serde(rename)]` drift across `InitOutcome` / `RenameOutcome` / `RegenSummaryOutcome` / `RemoveOutcome`. Pattern continues from US1's `tests/workspace_use_json_shape.rs`.

## What Didn't Work

- **C-M1 (init's empty `[summaries]` table)**: contract called for an empty TOML table; the implementing agent discovered that `CachedSummaries` has required fields under strict deserialisation, so an empty `[summaries]` would make every test that reads the workspace settings fail. Deviation: emit as a one-line comment `# [summaries]  -- populated by 'tome workspace regen-summary'`. The contract's discoverability intent is preserved; the actual table appears post-regen. **Lesson**: when a contract specifies a TOML structure, cross-check the deserialisation strictness on the receiving struct BEFORE writing the contract sample. The agent caught this mid-flight, but the disposition file's "Apply" verdict was wrong on first pass.
- **C-B2 / R-M2 doc-vs-code drift**: `rename`'s module doc claimed marker rewrites happen inside the DB transaction, but the code did them BEFORE the tx opens. The disposition's "fix" was to open the tx earlier — but FS writes can't actually roll back inside a SQLite tx. The pragmatic resolution was: open the tx earlier (so SQL UPDATE failure rolls back the workspace name change), keep the marker rewrites before (cannot rollback), document the partial-failure honestly, and route the orphan recovery through doctor `Binding` (US5). The contract's "Inside ONE DB transaction" language is aspirational; the reality is "the SQL UPDATE is inside the tx; the FS writes are not." Worth amending the contract for clarity in a future polish PR.
- **Test fragility from C-B1 ripple**: the existing `cascade_step1_tears_down_real_harness_mcp_entry` test only passed before the C-B1 fix because the buggy code iterated every harness unconditionally. After the fix, the test fixture had to declare `harnesses = ["claude-code"]` in the marker. The implementing agent caught this and patched the seed helper; the explanatory comment in `seed_bound_project` documents why.

## Workarounds & Solutions

**`toml_edit::Datetime` for `generated_at` (US2.d-1 C-M5)**: `toml_edit::value(...)` accepts `&str`, `bool`, `i64`, `f64`, `Datetime`. The contract specifies an unquoted TOML datetime literal; the implementation initially wrote it as a basic string. Fix: parse the RFC 3339 string via `toml_edit::Datetime::from_str(s)?` then `toml_edit::value(datetime)`. Now the on-disk artefact matches the contract example byte-for-byte. Generalisable: when a contract specifies a TOML scalar type, route through the matching `toml_edit::value` constructor.

**Settings-toml preservation via `toml_edit::DocumentMut`** (carried from `harness::mcp_config`): `workspace::regen_summary::update_settings_summaries` reads existing settings.toml, parses via `DocumentMut::from_str`, locates or inserts the `[summaries]` table, writes back. Preserves `[[catalogs]]` arrays + `harnesses` field + comments + key order. Same pattern now applied in `workspace::rename::rewrite_marker` for the project-marker rewrite. Reusable for any future Tome-owned TOML rewrite that must preserve sibling fields.

**Single hardened atomic-write surface** (US2.d-1 S2-M1+M2): `catalog::store::write_atomic` now carries the mode-preservation + symlink-refusal logic. The two harness atomic-write helpers (`harness::rules_file::atomic_write` + `harness::mcp_config::atomic_write_json/toml`) remain as their own copies but the pattern is now in 3 places, not 2 — and the cleanup PR (P5 retro?) can dedupe into a single `util::atomic_write` if it becomes the rule of three. For now: each location's docstring cross-references the others.

**StubScope for per-project effective list (US2.d-1 C-B1)**: production `ScopeProvider` (DB-backed) lands in US3.a; US2.b's `workspace remove` cascade uses `StubScope::new()` (empty registry → `[workspaces.<name>]` refs fail). Same pattern as `harness::sync::sync_project` from US1.b-3. Documented inline. When US3.a lands, swap in the production provider here too.

**Concurrent-remove-vs-use exit-code asymmetry**: the lockfile is non-blocking (`try_lock`), so concurrent `remove` + `use` for the same workspace can produce EITHER exit 13 (`WorkspaceNotFound` for the loser if remove won) OR exit 50 (`IndexBusy` for the loser if both raced before the winner committed). The test accepts either outcome. This matches the contract's non-blocking lock semantics but is a small surprise for callers expecting deterministic exit codes; documented in the test.

## Patterns & Code

- **Silent-compute + emit-wrapper** across all 7 workspace subcommands: every `commands::workspace::{init,list,info,rename,regen_summary,remove,sync}::run` is a thin wrapper that calls the corresponding `workspace::{init,info,rename,regen_summary,remove,sync}::*` library function and emits human/JSON per `Mode`. Pattern carries forward from US1.
- **5-step cascade ordering** (`workspace::remove`): explicit numbered steps per FR-405 with documented per-step failure semantics. Steps 1-2 best-effort (logged + continue); Step 3 atomic DB tx (rolls back on failure); Steps 4-5 orphan-recoverable (doctor `--fix` in US5).
- **`HashSet<String>` URL dedup vs `SELECT DISTINCT url`**: `remove` Step 5 deduplicates URLs via `HashSet` after the SELECT; reviewer R-M1 noted `SELECT DISTINCT` would be cleaner. Deferred — acceptable for the typical (single-digit) URL count per workspace.
- **`toml_edit::DocumentMut` for surgical field updates** that must preserve sibling fields: settings.toml `[summaries]` patch in `regen_summary`, marker `workspace = "<new>"` rewrite in `rename`. Both use the same shape: read → `from_str` → `doc["key"] = toml_edit::value(...)` → `to_string` → atomic-write.
- **Single-DB-transaction wrapping**: `init`'s INSERT + `--inherit-global` catalog rows; `rename`'s SQL UPDATE; `remove`'s Step 3 (skill/catalog/project/workspace row delete); `regen_summary`'s last_used_at bump. Pattern: any compound DB write opens `conn.transaction()` even if only one SQL statement runs inside (for forward compatibility + consistent rollback semantics).
- **Catalog-cache refcount cleanup** (`workspace remove` Step 5): for each URL the removed workspace held, query remaining `workspace_catalogs` references; `remove_dir_all` only when refcount → 0. Mirrors Phase 3 / US3.b's `catalog::store::reference_count` pattern from the `catalog remove` cascade.
- **Per-project effective list narrowing in `workspace remove`**: consult `settings::resolver::resolve_effective_list` per bound project before harness teardown. Matches the contract's per-project narrative + closes the "remove probes every harness directory on every bound project" footgun.
- **Hardened atomic-write inheritance**: every file writer outside `src/harness/` routes through `catalog::store::write_atomic`. Mode preservation + symlink refusal land at the single source; no future writer needs to remember to chmod or refuse symlinks.
