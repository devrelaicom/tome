Distinguishes "workspace exists but no settings.toml" (Ok(None)) from "unknown workspace" (Err(UnknownWorkspace) → exit 13) from "IO/parse error" (new mapping → WorkspaceMalformed exit 70).

## Workarounds & Solutions

**Tuple-drop discipline for RAII guards** (PR #90 + US3.d-1 T-B1):
- Rust drops struct/tuple fields in declared order, NOT reverse. When two RAII guards must be released in a specific order (e.g. clear shared state THEN release the mutex protecting that state), declare the "clear state" guard FIRST and the "release lock" guard SECOND.
- Applies to: `tests/common/mod.rs::HomeGuard { _previous, _lock }` (US3.d-1) and the per-test `install_synthetic() -> (HarnessModulesGuard, MutexGuard)` helpers in `tests/settings_{priority,composition,workspace_ref_outside_project,bad_exclusion}.rs` (PR #90).
- Generalisable: any future RAII helper that owns multiple guards should document its drop order with this rule.

**`CentralDbScopeProvider` distinguishes three workspace states** (US3.d-1 C-B1/C-B2/S-M4/R-M1):
- "Workspace exists with settings.toml": `Ok(Some(harnesses))` — composition resolves to declared list.
- "Workspace exists, no settings.toml": `Ok(None)` — legal; composition resolves to empty list at that scope.
- "Workspace not in central DB": `Err(UnknownWorkspace)` → exit 13.
- "IO or parse error reading settings.toml": new `CompositionErrorKind::SettingsReadFailure { name, source }` → exit 70 (`WorkspaceMalformed`).
- Reusable pattern: any future provider that reads workspace-scoped on-disk state should distinguish these four cases explicitly. Don't conflate IO errors with absence.

**Pre-entry validation vs end-of-resolution** (US3.d-1 C-M4): when validating user input against a closed set (like `SUPPORTED_HARNESSES`), validate at PARSE time, not at END time. `["fake", "!fake"]` should error on the inclusion immediately; deferring to end-of-resolution allows malformed exclusions to silently cancel out the error. Generalisable: validate as close to the input boundary as possible.

**`tome harness use|remove` lock-around-rmw** (US3.d-1 C-M5/R-M2/S-M2): acquire `index.lock` advisory lockfile BEFORE reading the settings file; hold until after atomic-write. Closes a classic R-M-W race where two concurrent invocations both read the starting state, each adds its harness, the second write clobbers the first. Generalisable: any command that modifies a Tome-owned config file via R-M-W must hold the lock.

**`HarnessModule` validation via `with_effective_modules`, not `lookup`** (US3.c-2 deviation): in commands that need to validate a harness name AND honour test-installed overrides, use `with_effective_modules(|mods| mods.iter().any(|m| m.name() == name))` rather than `lookup(name)`. `lookup`'s `&'static` return type precludes test override visibility; production paths are equivalent.

## Patterns & Code

- **Silent-compute (`assemble_*`) + emit-wrapper (`run`) split** for every CLI subcommand. Required for testability (US3.d-1 T-B2 was the forcing function). Pattern: `pub fn assemble_*(args, scope, paths) -> Result<Outcome, TomeError>`; `pub fn run(args, scope, paths, mode) -> Result<(), TomeError>` that calls `assemble_*` then emits. Tests use `assemble_*`; CLI dispatcher uses `run`.
- **`HomeGuard` + `HOME_MUTEX`** in `tests/common/mod.rs` for tests mutating `std::env::set_var("HOME", ...)`. RAII guard restores previous HOME on drop; field declaration order `_previous, _lock` so HOME restores BEFORE the mutex releases.
- **`HarnessModulesGuard` tuple-drop discipline** (PR #90 + US3.d-1): in `install_synthetic() -> (HarnessModulesGuard, MutexGuard)`, the guard is FIRST in the tuple so it drops FIRST, clearing the override BEFORE another waiting thread acquires the mutex.
- **Per-entry validation in resolve_list** (C-M4 fix): user-input validation runs at parse time. Don't defer validation that could be silently cancelled by other input.
- **`CentralDbScopeProvider`** as the canonical production provider for composition resolution; `StubScope` remains the test fixture.
- **Settings.toml R-M-W under advisory lock** (`tome harness use|remove`). Generalises: any Tome-owned config file modified via R-M-W holds the lock.
- **`settings::edit` module** wraps `toml_edit::DocumentMut` for surgical edits with comment + order preservation. Routes through hardened `catalog::store::write_atomic`.
- **Cycle path renders walk-order chain** (US3.a fix): DFS state has parallel `visited: HashSet` + `stack: Vec` structures; cycles render the active DFS chain (e.g. `["A", "B", "A"]`), not the alphabetically-sorted visited set.
