## Workarounds & Solutions

**F3 — variant name + exit-code collisions** (commit `0f10079`):

1. *Variant name collision*: Phase 3's `WorkspaceNotFound { path: PathBuf }` (exit 71, the marker-walk error) shared the `WorkspaceNotFound` name with the Phase 4 spec's new `WorkspaceNotFound { name: String }` (exit 13). Resolved by renaming the Phase 3 variant to `WorkspaceMarkerMissing` with a doc-comment flagging it as transitional. F10 will delete it when the resolver is rewritten.

2. *Exit-code collision*: early draft `contracts/exit-codes-p4.md` specced `SummariserFailure` at code 20, but Phase 2's `PluginNotFound` already owns 20. Closed-set rule II (pairwise-unique codes) forced a move. `SummariserFailure` placed at code 24 (next free slot above the Phase 2 plugin-lifecycle range 20–23). Contract + data-model reconciled in the same PR (commit `632a281`).

**F5 — audit findings** (no code change):

- *`serde_json/preserve_order`*: enabled project-wide in PR #59. Audit of `src/` shows no codepath relies on alphabetical `Value::Object` ordering: `output.rs` / `embedding/download.rs` / `plugin/lifecycle.rs` / `commands/models/mod.rs` / `commands/plugin/mod.rs` serialise typed `Serialize` structs (struct-field declaration order, independent of map backing); `plugin/components.rs` `from_slice::<Value>` only structurally detects fields without writeback; `mcp/log.rs` uses typed `serde_json::Map` directly. Switching from `BTreeMap` to `IndexMap` is therefore behaviourally neutral on existing code. F7's harness MCP config read-modify-write paths are the first consumers of the order-preservation guarantee.
- *`toml_edit`*: zero imports under `src/` at HEAD. F7's `harness/mcp_config.rs` is the first consumer. Tome-owned TOML (`config.toml`, `settings.toml`, manifests) continues to use the `toml` crate via `serde`.
