## What Worked Well

- **Brief-cap discipline (the F11b lesson)**: every dispatch stayed at ≤ 8 KB. The largest slice (US1.b sync) split into three sub-PRs (US1.b-1 rules_file primitives, US1.b-2 mcp_config primitives, US1.b-3 sync orchestrator); each agent ran to completion in 10–20 minutes without timing out. Total US1 surface shipped across 8 PRs ≈ 27 production-source files + 19 test files, none of which hit an agent budget ceiling.
- **Parallel reviewer pass** (T165): 4 agents (contract / Rust-lens / test / security) ran concurrently against the same merged surface and wrote per-reviewer findings to `/tmp/`. Total wall-clock for the review was the slowest single agent (~9 min) rather than 4× sequential. Findings consolidated into `review/us1-findings.md` + `us1-disposition.md` in one main-thread pass. Phase 3's review used the same pattern at the phase-close; carrying it down to per-user-story closeout per research §R-18 paid off — caught one data-correctness blocker (R-B1, lossy non-UTF8 path encoding) before US2 stacked on it.
- **HARNESS_MODULES_OVERRIDE injection** + RAII `HarnessModulesGuard` (US1.b-3): synthetic-registry tests against `StubHarness` exercised the full dispatch pipeline without touching the 5 real harness modules. Per-test-file `OVERRIDE_MUTEX` mitigated intra-binary contention; cargo's per-binary scheduling kept inter-file isolation safe. Pattern carries forward to US3.c when the four-other-harnesses are verified individually.
- **Library-API vs CLI binary test boundary held**: heavy-state paths (workspace seeding, full `bind_project` flow, sync orchestrator) used the library API with `StubEmbedder` / `StubHarness`; light/exit-code paths used `tests/exit_codes_e2e.rs` via the CLI binary. The split kept full-suite runtime under 90 seconds locally and avoided real ONNX model loads in CI.
- **Contract amendments unblocking reviewer findings**: T-B1 surfaced as a contract ambiguity (env preservation on `--force` with user-owned entries). Rather than re-litigate which side was right, we amended `mcp-config-integration.md` § Ownership marker with a clarifying paragraph and kept the safer test stance. Faster than a code-vs-contract argument; matches Phase 3 PR-H's "downstream contract drift fixed at the docstring layer" pattern.

## What Didn't Work

- **CI flakes on ubuntu-latest / msrv twice in a row** (PR #79): once with `collect2: fatal error: ld terminated with signal 7 [Bus error]` (runner OOM), once with `System.IO.IOException: No space left on device`. Second retry passed. Two consecutive retries before getting clean green added ~10 minutes per PR and forced manual `gh run rerun --failed` invocations. Pattern: ubuntu MSRV is the most resource-constrained job in the matrix because it's the heaviest build (no incremental cache vs stable) on the smallest runner. Plus retry tax adds ~3 minutes per attempt. Worth flagging as a CI-budget concern for the Polish phase if it persists.
- **Initial dispatch didn't anticipate stale-docs cleanup**: R-M6 (stale `--global` references) and R-M3 (overclaimed JSON formatting preservation) were both docstring-only fixes that should have landed at F10 deletion time. They slipped through. **Lesson**: when deleting a flag/feature, grep for ALL its references in `src/`, `specs/`, `docs/`, and `README.md` — not just the type definition. F10 closed the flag but left the docstring breadcrumbs.
- **C-M1 multi-harness mixed-style edge case** (claude-code + opencode targeting the same `AGENTS.md` with disagreeing body styles) was deferred from US1.d-2a because the test matrix doesn't exercise it. The deferral is honest — but it means a real regression in that edge case wouldn't fire until US3.c when all 5 harnesses are wired into integration tests. Tracking as US3.c precondition rather than US1 hangover.

## Workarounds & Solutions

**R-B1 — Linux-only test for non-UTF8 path refusal** (PR #80): macOS APFS rejects non-UTF8 directory names at the `mkdir(2)` syscall layer with "Illegal byte sequence", so a `tempdir + std::os::unix::ffi::OsStringExt::from_vec(b"\xff\xfe".to_vec())` test cannot construct the fixture on macOS. Gate with `#[cfg(target_os = "linux")]` instead of the obvious `#[cfg(unix)]`. The production check itself is platform-independent; only the test fixture is constrained.

**S-M3 — mode preservation across `NamedTempFile::persist`** (PR #80): `NamedTempFile` defaults to mode 0600 on Unix. When we rewrite an existing 0644 `AGENTS.md`, the rename silently lowers permissions. Fix: capture `target.symlink_metadata().map(|m| m.permissions().mode())` BEFORE the write, then on Unix `std::fs::set_permissions(tempfile.path(), Permissions::from_mode(mode))` BEFORE `persist`. ~10 LOC fix in both `rules_file.rs` and `mcp_config.rs`. Pattern reusable for any future atomic-rewrite of developer-authored files. Add a regression test under `tests/security_hardening.rs::preserve_file_mode_on_*_rewrite`.

**T-B2 — `tests/sync_idempotence.rs` referenced but missing** (PR #80): three contracts referenced a holistic cross-harness FR-525 idempotence test that didn't exist. Single-harness idempotence was scattered across 4 files. Fix: created `tests/sync_idempotence.rs` exercising claude-code + StubHarness simultaneously via `HARNESS_MODULES_OVERRIDE`, capturing mtimes for all 4 outputs, asserting unchanged after re-sync. Lesson: when a contract references a test file by name, the test file must exist OR the contract reference must be retracted; CI doesn't catch the broken cross-ref but a reviewer pass does.

**Sync orchestrator's StubScope** (PR #77): the composition resolver needs a `ScopeProvider` trait impl. The production version (DB-backed) lands in US3.a, so US1.b-3 shipped `StubScope::new()` (empty registry, returns `UnknownWorkspace` for any `[workspaces.<name>]` reference). Tests using project-marker `harnesses` declarations work; tests using `[workspaces.<name>]` cross-refs will fail with exit 17 until US3.a. Documented inline in `sync_project`'s doc comment. Generalisable pattern: when a downstream feature needs a trait impl that hasn't been built yet, ship a `StubScope`-shaped placeholder with a clear "production lands in US-X" comment + a documented failure mode for the unsupported cases.

**HarnessClash forward-progress vs lock release ordering** (PR #77): the bind algorithm commits the DB write (workspace_projects UPSERT + last_used_at), releases the advisory lock, THEN runs the sync orchestrator (Phase B per contract). A HarnessClash error inside Phase B propagates as the command's exit code but the DB state remains committed (FR-403). The first iteration assumed the sync's harness-clash error rolled back the binding; corrected by re-reading the contract before T144 wiring. Test `binding_commits_even_when_harness_clash_returns_exit_19` pins the invariant.

## Patterns & Code

- **Atomic-populated-directory** (`src/util/atomic_dir.rs` from F4): the canonical mechanism for landing `<project>/.tome/` and other multi-file directory writes. `land_directory_with_replace` adds the rename-aside-then-rollback semantic; US1.a uses it for the binding marker (with `--force` paths that overwrite existing `.tome/`). Reusable in US2 (`workspace add`) and US3 (`harness sync`'s rules-file directory creation).
- **`refuse_symlink` per-entrypoint**: every file write path in `rules_file.rs` and `mcp_config.rs` calls `refuse_symlink(target)` before any read. Phase 3 PR-F's hardening discipline carried forward — but with a TOCTOU window across multiple reads (S-M2, deferred). Pattern: never `O_FOLLOW` user-controllable paths; the rename(2) no-follow semantic mostly bounds the damage even when the window opens.
- **Idempotence-by-mtime testing**: across `rules_file_*`, `mcp_config_*`, `harness_sync_stub`, `workspace_use_claude_code_e2e`, `sync_idempotence`: capture mtime, sleep 1500ms, re-run, assert mtime unchanged. The `MTIME_TICK = 1500ms` constant is documented as a macOS APFS-stability hack but consistent across all idempotence tests. Generalisable pattern.
- **Test stub + `HARNESS_MODULES_OVERRIDE` injection** (US1.b-3): production code uses `with_effective_modules(|mods| ...)` which consults the override slot first. Tests install `Vec<Box<dyn HarnessModule>>` via `HarnessModulesGuard::install(...)`; the guard's `Drop` clears the slot. Pattern is the harness-module mirror of F11c-2's `MIGRATIONS_OVERRIDE` thread-local for the schema-migration framework — same shape, RwLock+Vec for harnesses vs thread_local+Vec for migrations because harness modules need `Send + Sync`.
- **`sync::SyncOutcome` per-harness records** (FR-547): `Vec<SyncChange>` + `Vec<HarnessDecision>` give the JSON envelope enough resolution that consumers can list every touched file path + every dispatch decision. Wire-shape pinned by `tests/workspace_use_json_shape.rs`.
- **Reviewer findings → disposition → applied + deferred** (US1.d-2a): the consolidated `review/us1-findings.md` lists every finding by ID; `review/us1-disposition.md` maps each to "apply now" vs "defer to follow-up issue" vs "defer to US5 polish". Phase 3 PR-A established this pattern at the phase-close; US1 demonstrated it at the per-user-story scale. Cost: ~30 min main-thread time to consolidate + decide; benefit: an auditable record of "which findings we shipped, which we deferred, why".
- **Helper-promotion lag** (R-M1, S-M3): both fixes were ~10 LOC changes touching multiple files. Resisted the urge to extract a `util::atomic_rewrite_preserving_mode` helper now — rule of three says wait until US2/US3 land a third caller. Worth revisiting in the Polish phase.
- **`#[ignore = "<unhide target>"]` on the SIGINT atomicity test**: documented why SIGINT-mid-populate testing was skipped (cargo per-process signal handler races other tests in the binary) with the explicit unhide target "US5 doctor --fix orphan cleanup". Per CLAUDE.md convention §6.

## For Next Time

- **Run `/sdd:map incremental` at each user-story closeout, not only at phase close.** Phase 4 / F11 closeout did this; US1 closeout did this. Cheap (~3 min for 4 parallel mappers producing 8 docs) and keeps the `.sdd/codebase/` view in sync with merged source. Recommended for US2–US5 closeouts.
- **CI flakes on ubuntu MSRV are now a 2-in-3 pattern**, not 1-in-10. The handover doc characterised this as transient bus error; we observed both bus error AND "no space left on device" within the same PR. If the pattern continues into US2+, time to investigate the MSRV runner specifically (Cargo.lock has `rust-version = 1.93`; the MSRV build is heavier than stable). Possible mitigations: bump runner from `ubuntu-latest` to `ubuntu-24.04-large` for MSRV only; add a build-time profile that strips debug info more aggressively for MSRV; or drop MSRV CI to a weekly schedule rather than per-PR.
- **Reviewer brief drift between phases**: the 4-reviewer pass in US1 used ad-hoc per-scope briefs. Phase 3 used the same pattern but the briefs were more standardised (each focused on one document like `contracts/X.md`). Recommendation: build a small `review/<US>-brief.md` template that captures the per-reviewer scope + finding-classification rules + expected output file path. Then US2–US5 just instantiate the template. Saves ~5 min per US.
- **Pre-emptive sub-slice planning** when the brief would exceed 8 KB. F11b was the first lesson; US1.b was the first proof. Both went well; cement as the default approach for any slice with > 10 sub-task identifiers in tasks.md.
- **Test gap follow-up issue triaged at disposition time**: US1.d-2a deferred 8 test gaps (T-M2, T-M3, T-M5, T-M6, T-M8, T-M9, T-M10, T-M11). Track these in a single tracking issue, not 8 separate ones. The "single follow-up issue for a class of related gaps" pattern keeps GitHub issue noise low.
- **Sync orchestrator coverage matrix expansion** for US3.c: when the 4 other harnesses (codex, cursor, gemini, opencode) are verified individually, add a `tests/workspace_use_full_matrix_e2e.rs` that drives the real `SUPPORTED_HARNESSES` registry through a bind+sync with all 5 in the effective list. Will exercise C-M1's deferred edge case AND verify shared-rules-file dedup works across the real harness paths.
