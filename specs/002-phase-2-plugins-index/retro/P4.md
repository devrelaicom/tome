## Slice 1 — implementation (`run_interactive` + bare `tome plugin` dispatch)

### Test-driving strategy decision (recorded up-front per handover guidance)

`inquire` does not expose a public test backend. The two reasonable paths for T101:

1. **pty harness** (e.g. `expectrl`, `rexpect`, or a hand-rolled pty + scripted reads/writes): spawn `tome plugin` as a child process attached to a real pty, send arrow-key + Enter byte sequences, read the rendered prompts. Treats `inquire` as a black box. Heavier in test-only deps but keeps `run_interactive` as a plain straight-line call into `presentation::prompt`.
2. **Injectable I/O abstraction**: refactor `presentation::prompt` (or `run_interactive`) to take a trait object (`Selector`/`Confirmer`). Production passes the `inquire`-backed impl; tests pass a scripted impl. Lighter on test deps but threads testability concerns through the production design.

**Decision for slice 1: option 1 (pty harness in slice 2).** `run_interactive` stays as plain `presentation::prompt::{select,confirm,require_terminal}` calls. Reasons:

- Existing `presentation::prompt` is already a thin function-level wrapper, no trait. Refactoring it just for testability would expand the surface across every call site (`enable`'s model-download confirm, `disable`'s confirm, the interactive selector — every prompt would need to thread the abstraction through).
- The pty harness in slice 2 also covers SIGINT semantics, terminal-size handling, and multi-step state in a way an injected scripted impl can't easily simulate. It's closer to what users actually experience.
- If the pty harness turns out to be unworkable (test flake, platform issues), slice 2 can pre-emptively split into 2a (refactor for injection) + 2b (tests). Picking the harness now, with the option to revisit at the slice boundary, beats refactoring twice.

### Implementation notes

- **Bare `tome plugin` via `Option<PluginCommand>`.** clap derive does not let a `Subcommand` enum be made directly optional, but `Args` wrapping a subcommand can hold an `Option<PluginCommand>`. So `Command::Plugin(PluginArgs)` where `PluginArgs { #[command(subcommand)] command: Option<PluginCommand> }` is the smallest change that gives us `tome plugin` as a no-arg dispatch site. `main.rs` matches the `Option` and routes to either `commands::plugin::run` or `commands::plugin::run_interactive`.
- **Three loop levels mirror the menu hierarchy** (`catalog_loop` → `plugin_loop` → `view_loop`). Each loop's return type encodes the two distinct "exit" signals it can produce:
  - `Result<(), TomeError>` for the top-level `catalog_loop` (because the very top is what the public API returns).
  - `Result<(), LoopExit>` for the inner two, where `LoopExit::Quit` means "the user picked Quit / pressed Esc / pressed Ctrl-C — unwind every level cleanly" and `LoopExit::Err(TomeError)` means "a real error happened, propagate". `From<TomeError> for LoopExit` lets `?` work on calls into `lifecycle::*` and `enable::run` directly.
- **Action menu only shows the relevant verb.** `build_action_menu(status)` returns `Enable + Back` for Disabled, `Disable + Back` for Enabled, just `Back` for Unindexable. Avoids surfacing actions that would always fail.
- **`render_plugin_view` is duplicated from `commands::plugin::show::run` rather than refactored out.** Both functions are short and the duplication is honest; promoting a shared `render_for(record, agg)` helper makes more sense once US3's disable subcommand lands and there's a third caller.
- **Enable is delegated to `enable::run`** with a synthesised `PluginEnableArgs { id, yes: false }`. Gives us the model-download prompt, banner, spinner, warnings, and final summary line for free. Mode is hard-wired to `Mode::Human` — the bare interactive form already rejected `--json`.
- **Disable inlines `lifecycle::disable` + confirm prompt.** No `commands::plugin::disable::run` exists yet (US3 / T113). Interactive disable is "confirm + force-equivalent" per contract — exactly `lifecycle::disable` with no second prompt.

### Gotchas

- **`presentation::prompt::confirm` returns `TomeError::Interrupted` on Esc / Ctrl-C, not a distinct "cancelled" variant.** Same for `select`. We trap both `Interrupted` and `NotATerminal` at the boundary and translate to `LoopExit::Quit`. `NotATerminal` shouldn't actually trigger inside the loops (we required a TTY at entry), but trapping it defensively keeps the code total.
- **`NotATerminal` message duplication.** The contract specifies a specific guidance message ("Try `tome plugin list`…"). The `TomeError::NotATerminal` Display is generic. To get the specific guidance, the entry-point writes the contract's line directly to stderr, then returns the error — the harness then appends its generic line. Two stderr lines instead of one is the minor cost; the alternative would be a dedicated error variant or threaded message through the error type, neither of which is worth the churn for one call site.
- **Quit semantics around enable's "user declined download".** When the interactive flow calls `enable::run` and the user declines the model-download prompt, the existing code returns `TomeError::Interrupted` (exit 8). That propagates through `LoopExit::Err` and ends the whole interactive flow with exit 8 — not exit 0. This matches handover gotcha #12's "user-declined-download → Interrupted" convention. Whether that's the desired interactive UX (vs. "decline, bounce back to view") is open; it's the same exit code as Ctrl-C-during-embed, so it's at least consistent. Worth revisiting if integration tests pin it the other way.

### What worked well

- `presentation::prompt::{select, confirm, require_terminal}` was exactly the right granularity — no need to peek through to `inquire` for the interactive flow.
- The pre-existing `commands::plugin::mod.rs` `pub(crate)` helper set (`aggregate_for_plugin`, `open_index_for_read`, `read_catalog_manifest`, `resolve_plugin_dir`, `human_relative`) covered every read path the interactive flow needed. No additional shared code had to be promoted in this slice.
- The smoke check `tome plugin --help` confirmed clap's wrapper-arg pattern was rendered correctly (commands listed, `[COMMAND]` shown as optional, our long_about prose surfaced).

### For next time (slice 2)

- T101: pty harness will spawn the `tome` binary attached to a real pty, send arrow-down / Enter / `y` / `n` byte sequences, and read prompt output. Likely a single dev-dependency on `expectrl` or `rexpect`; verify it compiles on `macos-latest` and `ubuntu-latest` (the latter for CI). If portability is fragile, hand-rolling via `nix`'s `openpty` is the fallback.
- T102: non-TTY refusal is a normal `Command::new(...).output()` (no pty). Assert exit code 54 and that stderr contains the specific pointer message.
- Both tests should NOT pre-enable plugins via `lifecycle::enable` (which needs the stub embedder) — the interactive flow is read-only at the menu level, and the disable path can be tested independently by pre-enabling once and then driving the disable action.
