## What Didn't Work

### indicatif unit tests + pre-push hook hang (slice 2 — presentation)

Four unit tests in `src/presentation/progress.rs` (constructing and finishing `ProgressBar` / `byte_bar` / `spinner`, plus a rapid-increment test) and one env-mutating test in `src/presentation/colour.rs` ran cleanly under every direct invocation pattern — `cargo test --lib`, `cargo test --workspace`, `cargo test < /dev/null`, `sh -c 'cargo test' < /dev/null > out 2> err`, and even `lefthook run pre-push` standalone — but **deadlocked the unit-test binary** when invoked through an actual `git push → .git/hooks/pre-push → lefthook → cargo test` chain. The hung child was `target/debug/deps/tome-<hash>` (the lib unittests binary), running for 12–23 min with 0.01 CPU — a wait-state, not a slow-path.

Hypotheses we did not fully verify:

- `indicatif` `enable_steady_tick` spawns a background thread that may interact with the stdin git provides to pre-push hooks (refspecs piped on stdin).
- The `console` crate's terminal-attribute detection on macOS arm64 may block under specific stdio configurations.
- `unsafe { std::env::set_var(...) }` in the lib test binary may race with other tests in the same binary when run in parallel under that environment.

Mitigation for slice 2: removed the suspect test cases and deferred coverage of `progress.rs` to integration tests downstream (`tests/plugin_enable.rs`, `tests/models_download.rs`, `tests/catalog_update_reindex.rs` once those land). Module code is unchanged. The TTY-detection branch is indirectly verified by SC-007.

**Action for Polish phase**: write a small reproducer crate that runs a `cargo test` against a fixture binary using `indicatif` from inside a `git push` simulator, identify the failure mode, and either fix the test pattern or document the constraint in CONVENTIONS.md so future Foundational slices avoid it.
