## What Didn't Work

### lefthook pre-push hook hanging on `git push` under the dev environment

During slice 2 (presentation primitives) every `git push` ran the lefthook pre-push hook which invoked `cargo test --workspace` — and that workspace test binary hung indefinitely (0.01 CPU after 13+ min) every time. The same `cargo test --workspace` invocation outside the hook ran clean in 3.26 seconds. The hang reproduced even after removing the tests we initially suspected (indicatif draw-target tests and an env-mutating colour test), proving the diagnosis "specific test cases" wrong.

Process tree at the hang showed a `tokf` output-filter wrapper in the parent chain — the dev environment wraps shell commands to filter their output. We hypothesise (without confirming) that the wrapper's pipe buffer fills before the lefthook output sink drains it, blocking the test binary on stderr write. Other hypotheses considered: indicatif's `enable_steady_tick` background thread interacting with git's pre-push stdin (git pipes refspecs there), and `console`-crate terminal-attribute queries on macOS arm64 under closed-stdin conditions.

## Workarounds & Solutions

### Migrated off lefthook to native `core.hooksPath` git hooks

After confirming local `cargo test --workspace` was healthy (3.26s, all green) and the hang was exclusively environmental, we removed lefthook entirely in favour of three small versioned shell scripts under `.githooks/`:

- `.githooks/pre-commit` — runs `cargo fmt --check`, `typos`, `cargo clippy --all-targets --all-features -- -D warnings` sequentially.
- `.githooks/commit-msg` — `cog verify --file "$1"`.
- `.githooks/pre-push` — drains the refspec stdin git provides, then runs `cargo test --workspace`. The stdin drain is the key change vs lefthook's invocation pattern; it isolates `cargo test`'s stdio inheritance from the pipe git keeps open during a push.

Bootstrap is `git config core.hooksPath .githooks` (one-time, per clone). No external hooks manager. CONSTITUTION.md bumped to v1.1.0 (workflow change is contributor-facing). CONTRIBUTING.md, CLAUDE.md, the `.sdd/codebase/*` docs, and CHANGELOG.md updated accordingly.

The new pre-push hook also runs through the `tokf` wrapper but with the explicit stdin drain it no longer hangs. The constitution gates (principle X — fmt + clippy + typos + conventional commits + cargo test) all still run on every push.

## Workarounds & Solutions

<!-- Bug fixes, non-obvious solutions, clever workarounds -->

### `File::try_lock` returns `Result<(), TryLockError>`, not an `io::Error`

The stable signature stabilised in Rust 1.89 surfaces contention through a
typed `TryLockError::WouldBlock` variant rather than an `io::Error` with
`ErrorKind::WouldBlock`. The first attempt at `src/index/lock.rs` matched on
`io::Error` and only compiled after switching to:

```rust
match file.try_lock() {
    Ok(()) => Ok(LockGuard { … }),
    Err(TryLockError::WouldBlock) => Err(TomeError::IndexBusy),
    Err(TryLockError::Error(e)) => Err(TomeError::Io(e)),
}
```

### `sqlite3_auto_extension` already has the full C signature

libsqlite3-sys exposes the full `(*mut sqlite3, *mut *mut c_char,
*const sqlite3_api_routines) -> c_int` shape — the older "narrow to
zero-arg + `mem::transmute`" pattern is no longer needed. The vec_ext
registrar just passes `Some(sqlite3_vec_init)` directly.

### Atomic-rename cleanup must wrap the full post-stream pipeline

The first cut of `download_model` only ran the partial-dir cleanup when
`stream_to_partial` itself returned `Err`. Checksum mismatch — which is
detected *after* `stream_to_partial` succeeds — therefore leaked the
`.partial/` directory. Fix: wrap the entire pipeline (stream → verify →
rename → manifest) in a closure and run cleanup against any error from it.
Caught by `tests/model_download.rs::checksum_mismatch_aborts_and_cleans_partial_dir`
on the first run.

### `Box::leak` for `&'static str` test fixtures

`ModelEntry` carries `&'static str` fields because the production registry
is compile-time `const`. Tests need to construct entries pointing at a
runtime-allocated URL (the local `TcpListener` port). `Box::leak(s.into_boxed_str())`
gives a `&'static str` whose lifetime is the test binary's process — fine
because `cargo test` runs each integration-test binary fresh.

## Packages & Dependencies

<!-- Useful packages discovered, packages to avoid, version issues -->

### fastembed v4 has two distinct init-options types

`TextEmbedding::try_new_from_user_defined` wants `InitOptionsUserDefined`;
`TextRerank::try_new_from_user_defined` wants `RerankInitOptionsUserDefined`.
Same shape, different type; mixing them up is a compile error not a panic,
which is the right safety boundary.

### ort is transitive-only through fastembed

We do not link `ort` directly. The first attempt at `src/embedding/runtime.rs`
tried to call `ort::init()` and failed at compile time. The module is now a
no-op placeholder — fastembed manages ORT lifecycle internally — kept around
as the future hook if a direct dep is ever needed.

### `std::fs::File::try_lock` (Rust 1.89+) is sufficient for the index lock

No need for `fs2`. Per-fd locks (F_OFD_SETLK on Linux, flock on macOS/BSD,
LockFileEx on Windows) mean two `File::open` handles in one process compete
properly, which keeps `tests/index_lock.rs` simple (no child-process
spawning).

## Patterns & Code

<!-- Reusable patterns, helpful code snippets, architectural insights -->

### Closure-injected embedder in `enable_plugin_atomic`

To avoid coupling `src/index/skills.rs` to the embedding crate (and the
slice ordering it implied), `enable_plugin_atomic` takes
`F: FnMut(&str) -> Result<Vec<f32>, TomeError>` rather than a trait object.
A real `Embedder` impl wraps to a closure trivially. This kept slice 4b
independent of slice 5.

### Hand-rolled `TcpListener` HTTP fixture beats a dev-dep

`tests/model_download.rs` runs a one-shot HTTP server in ~15 lines: bind
`127.0.0.1:0`, spawn a thread, accept one connection, sink the request
bytes, write the canned response, close. No `httptest` / `wiremock`
dev-dependency. Reusable for any future test that needs a controlled
HTTP responder.

### Strictness boundary is enforced by attribute, not module name

`#[serde(deny_unknown_fields)]` lives on Tome-owned types (`Config`,
`CatalogManifest`, `ModelManifest`, `IndexMeta` keys). Third-party types
(`PluginManifest`, `SkillFrontmatter`) omit it. The boundary is type-by-type
rather than module-by-module — both kinds happily coexist under
`src/plugin/` and `src/embedding/`.

### Slice splitting after the fact is cheap; before the fact is cheaper

Slice 4 was pre-emptively split into 4a (bootstrap pipeline) + 4b (lock /
meta / integrity / skills / query) on the hand-over's recommendation. Each
half landed cleanly with a focused PR. Slice 5 deferred T057 to the
phase-close slice; that worked but cost an extra cycle of context switching.
Pre-emptive split when the spec already exposes a seam beats post-hoc.

## For Next Time

<!-- Improvement suggestions, lessons learned, things to try differently -->

### Move pre-emptive splits into `/sdd:plan` output

The hand-over identified slice 4 as too large *before* implementation
started, by inspection of `data-model.md` and `contracts/index-schema.sql`.
The split shape (`schema + db + migrations + vec_ext + bootstrap-test`
versus `lock + meta + integrity + skills + query + lock-test`) was obvious
from the dependency graph. The next `/sdd:plan` pass should encode this in
the task list itself rather than leaving it to the implementing session to
notice.

### Registry placeholders carry real risk

`MODEL_REGISTRY` ships with all-zero `sha256` + `0` `size_bytes`. The
download guard (`has_placeholder_checksum`) refuses to install against
those values, so production code is safe. But a future CI run that fetches
the real upstream artefacts will be the *first* moment those values get
verified — and may surface upstream-moved-the-artefact problems. Pin the
real values in the first US1 slice; don't defer further.

### Codebase docs (`.sdd/codebase/*`) drifted

Only STACK.md and STRUCTURE.md were refreshed this phase. CONVENTIONS.md,
TESTING.md, ARCHITECTURE.md, SECURITY.md, CONCERNS.md, and INTEGRATIONS.md
are still Phase-1 in tone and need an incremental refresh — likely a single
focused slice once US1 lands so the docs reflect actual user-facing
surface, not just the foundational plumbing.
