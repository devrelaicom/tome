# Phase 6 Retrospective: User Story 4 — Explicit model management commands

## What Worked Well

- **Mirror-the-`plugin/`-module layout for `models/`.** Reusing the per-subcommand-file convention (`mod.rs` dispatches; each subcommand owns a `pub fn run(args, mode) -> Result<(), TomeError>`) made slice 1 mechanical: copy the structure, swap the verbs, point at `embedding::download` instead of `plugin::lifecycle`. No new architectural debates, no naming bikesheds.
- **Slice 1 = production code, slice 2 = tests** held tightly. Slice 1 shipped at +517 production lines + 40 lines `cli.rs` + 1 line `main.rs` + 21 lines `embedding/download.rs`; slice 2 shipped at +387 test lines and **zero** production changes. The boundary was respected end-to-end — no "while I'm in there" mutations.
- **Sparse-file fixtures.** `File::set_len(280_000_000)` produces a zero-filled artefact that takes ~no disk space, costs ~zero IO, and intentionally disagrees with the registry-pinned SHA-256 (so `--verify` paths flip to `checksum_mismatched` naturally). The 280 MB reranker fixture is now essentially free in CI.
- **Library-API vs CLI-binary boundary held.** `tome models download`'s real network path is library-level only (`tests/model_download.rs` with the hand-rolled `TcpListener` HTTP fixture from Phase 2 slice 7). The CLI tests cover only the *skip path* (`fabricate_all_installed_models`), exit-2 usage error, exit-30 missing-model error, and exit-54 non-TTY refusal — i.e. the paths that don't require an embedder or network.
- **Banner-skip-in-JSON-mode** is now a firm precedent across 4 commands (`enable`, `disable`, `models::download`, `models::remove`). NDJSON consumers see only structured records on stdout. No special-case logic in callers — each `run` function just gates `presentation::banner` on `OutputMode != Json`.

## What Didn't Work

- **The retro template stayed empty through slice 1 and slice 2.** Both PRs landed before the retro got more than up-front context. The closeout pass is where it actually got filled in — fine, but the better discipline is a "drop two bullets into P6.md before merging the slice" rule. That's a transferable lesson for Phase 7+.
- **The byte-progress refactor of `download_model` slipped further.** Phase 6 shipped indeterminate `indicatif` spinners across both `plugin enable` and `models download`. With two consumers, the refactor is now strictly past the rule-of-three threshold and lives in CONCERNS.md as TD-010.

## Workarounds & Solutions

- **`output::write_json` `?Sized` relax** (PR #22, `src/output.rs` 1-line change): `models::list::emit_json` wants to serialise a `&[ListRecord]` slice. The original `T: Serialize` bound forced an intermediate `Vec` allocation; relaxing to `T: Serialize + ?Sized` makes slice serialisation work directly. Low-risk widening — `serde_json::to_writer` already accepts `?Sized` upstream. Apply the same relax when any future caller needs to JSON-emit a slice.
- **All-zero sparse contents force a SHA-256 mismatch deterministically.** This is the linchpin of the `--verify` flip test: the manifest claims a real SHA, the artefact is 280 MB of zeros, the rehash disagrees, state flips to `checksum_mismatched`. No need for tampering after fixture-build; the fixture itself is the tampered state.
- **Sparse file + multi-file artefact set.** Some model entries have multiple files (e.g. `tokenizer.json`, `config.json`). `fabricate_installed_model` sets the primary file (i = 0) to `entry.size_bytes` and every auxiliary to 1 byte. 1 byte satisfies "exists and non-empty" for `models::cheap_state` without wasting space.

## Packages & Dependencies

- **No new direct dependencies.** Slice 1 only wires existing `embedding::download`, `embedding::registry`, `indicatif`, `comfy-table`, `inquire`. Slice 2 only adds test-internal helpers via `std::fs::File::set_len`.
- **`std::fs::File::set_len`** is sparse-file-aware on Linux + macOS (and on Windows when the filesystem supports it — NTFS does). Bytes read back as zero; the file's logical size is `n` but the on-disk size is the metadata block plus whatever the filesystem decides. For CI purposes this is "free".

## Patterns & Code

- **The `models/` module mirrors `plugin/`** byte for byte at the dispatch level. `mod.rs` owns `pub fn run(args, mode)` that matches on the subcommand enum and forwards. Each subcommand file owns its own `run(args, mode)`. Errors return `TomeError`; the binary's `main.rs` maps to exit codes via the existing `ExitCode::from(&TomeError)` path.
- **`embedding::download::sha256_file(path: &Path) -> Result<String, TomeError>`** is now `pub`. Streaming, chunked SHA-256 reader; same implementation as the download verifier but reusable from the `--verify` path. Phase 7's `catalog update` reindex may want it for skill-content hash verification.
- **Sparse-fixture pattern is reusable.** If Phase 7 / Phase 8 tests want "model present at realistic size without IO cost", call `fabricate_installed_model`. If they want "all models present (skip path)", call `fabricate_all_installed_models`. The 280 MB reranker is no longer a CI concern.
- **CLI-skip-path test idiom.** For commands that depend on heavy external state (network, GPU, large files), test only the skip path in the CLI binary tests and rely on the library tests for the active path. Slice 2's `tests/models_download.rs` is the template: "every model already installed → all-`skipped` records → exit 0".

## Up-front context (slice 1 start)

### State going in

- `MODEL_REGISTRY` is already pinned in `src/embedding/registry.rs` — two entries: `bge-small-en-v1.5` (embedder, 66 MB, MIT) and `bge-reranker-base` (reranker, 280 MB, MIT). Real upstream SHA-256s. No new registry work.
- `embedding::download::download_model(entry, model_root) -> Result<ModelManifest, TomeError>` already implements the full atomic, SIGINT-aware download pipeline (stream + hash + verify + atomic rename + manifest persist). It maps every failure to the right `TomeError` variant. No new library code needed for download.
- All required error variants exist: `ModelMissing` (30), `ModelCorrupt` (31), `ModelChecksumMismatch` (32), `ModelRegistrationParseError` (33), `Io` (7), `NotATerminal` (54), `PluginAlreadyInState` (21 — not used here), `Usage` (2). Slice 1 wires existing primitives; no new error variants.
- `presentation::tables`, `presentation::progress`, `presentation::prompt::confirm`, `presentation::colour` all exist and have established call-site patterns from the plugin commands.

### What slice 1 needs to ship

- New CLI variant `Command::Models(ModelsArgs)` with subcommand enum `ModelsCommand::{ Download(args), List(args), Remove(args) }`. Flag spellings per the contract: `--force` on download + remove, `--verify` on list. `--json` is the global flag.
- New module tree: `src/commands/models/{mod.rs, download.rs, list.rs, remove.rs}` mirroring the `plugin/` layout.
- `download::run` — iterate `MODEL_REGISTRY`, for each: check manifest existence + file existence + sizes (cheap), skip unless `--force`, call `embedding::download::download_model` with an `indicatif::ProgressBar::new_spinner()` wrapper (handover triage item flagged a byte-progress refactor; defer to a follow-up — minimum-viable Phase 6 ships indeterminate spinners, matching `plugin enable`'s precedent).
- `list::run` — iterate `MODEL_REGISTRY`, classify each into `Ok | Missing | Corrupt | ChecksumMismatched`. Without `--verify`, only check existence + size (cheap). With `--verify`, rehash via the same chunked SHA-256 routine — promote that routine out of `download.rs` if it's currently private; otherwise inline a small variant for the verify path. Render via `presentation::tables`.
- `remove::run` — registry name check (`Usage` if unknown), `ModelMissing` if not installed, confirm prompt (default no, `--force` short-circuits, non-TTY without `--force` → `NotATerminal` exit 54 with the documented pointer), then delete the manifest first (so the install state flips observably) then the model directory. Same pointer-line-then-error pattern as `plugin disable`.
- Wire `Command::Models` into `src/main.rs` dispatch.

### What slice 1 does NOT ship

- Integration tests (slice 2 — `tests/models_download.rs` with local HTTP fixture, `tests/models_list.rs`, `tests/models_remove.rs`).
- No new error variants.
- No byte-progress callback refactor of `download_model` — keep the slice tight; indeterminate spinner ships matching enable's precedent.

### Open questions / triage candidates

- The non-TTY-pointer-message-then-NotATerminal pattern will hit its 4th occurrence in `remove::run` (currently: `enable::run` for `--yes`, `disable::run` for `--force`, `interactive::run` defensively). Per the rule of three+ promotion, this is now the moment to either consolidate into a `presentation::prompt::require_terminal_or_pointer(msg: &str)` helper, or accept duplication and move on. **Decision**: ship the duplication this slice — the 4-line per-site pattern is shorter than the helper's call site would be, and the message text differs per command. Worth one more pass at the helper if a 5th site appears.
- `models list` rehash with `--verify` should reuse a shared chunked-SHA-256 helper. If `download::stream_to_partial` is the only current implementation and it's tangled with the streaming write, extract `sha256_file(path: &Path) -> Result<String, TomeError>` as part of this slice.
