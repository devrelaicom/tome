# Phase 7 Retrospective: User Story 5 — Keep the index in sync with upstream catalogs

## What Worked Well

- **3-slice plan held to the line.** Slice 1 (library), slice 2 (catalog update integration), slice 3 (explicit `tome reindex` CLI) each landed inside the soft size cap and each carried its own integration tests. The library-then-consumer ordering meant the unit tests in slice 1 caught the `sqlite-vec` `INSERT OR REPLACE` bug before either consumer started using the function — a slip into slice 2 or 3 would have triggered a CI fire-drill across an already-large surface.
- **Library-API test entry points are now established as the standard.** `pub fn reindex_catalog_plugins(catalog, enabled, deps)` and `pub fn run_with_deps(scope, plugins, deps, force, mode)` mirror the existing `lifecycle::enable` pattern: take pre-built `LifecycleDeps`, return a deterministic outcome, let the CLI binary be a thin shell over them. Tests inject `StubEmbedder` and assert via `call_count()`; the CLI binary tests cover the parse/scope error surface only. This is now the firm pattern for CLI subcommands that need an embedder.
- **Lazy `FastembedEmbedder` loading** in both `catalog update` and `reindex` means a tome install with zero enabled plugins (the common case before US1 lands for a user) never touches ONNX models. The `GetOrInsertWithResult` helper trait makes the lazy pattern read clean at the call site.
- **Mirroring `enable_plugin_atomic` for `reindex_plugin_atomic`** kept the new function's shape predictable. Same transaction discipline, same SQLite parameter style, same naming convention. Code review can focus on the diff semantics rather than re-validating the transaction boundary.

## What Didn't Work

- **The latent `INSERT OR REPLACE` bug.** `upsert_skill` had been emitting `INSERT OR REPLACE INTO skill_embeddings` since the initial Phase 2 index work — but `skill_embeddings` is a `sqlite-vec` virtual table that doesn't implement REPLACE / `ON CONFLICT`. Every existing test happened to take the no-prior-row code path, so the bug never fired. Reindex's Modified branch hit it on the first test run. **Takeaway**: if a function has a branch that no test exercises, write a unit test for it before relying on it in a new consumer. The "test the branch you're about to use" discipline would have caught this earlier.
- **CI macOS-stable runner flaked once** with the same `cargo fmt` → `rustup-init` shim mis-resolution we saw on the Phase 6 closeout. Re-running the failed job cleared it. Worth tracking — if it happens twice more on macOS-stable specifically, file an upstream issue with `dtolnay/rust-toolchain`.

## Workarounds & Solutions

- **`upsert_skill` bug fix**: `skill_embeddings` is `vec0`. vec0 virtual tables don't support `INSERT OR REPLACE`. The fix is mechanically simple: `DELETE FROM skill_embeddings WHERE skill_id = ?` (idempotent — no-op for first insert) followed by `INSERT INTO skill_embeddings ...`. Both statements live inside the existing transaction, so the atomicity invariant is preserved.
- **`reindex_plugin_atomic` snapshot-then-pass pattern**: collect every existing `(name, hash)` for `(catalog, plugin)` into a `HashMap` at the top of the transaction, then drain entries as you visit on-disk skills. Whatever remains in the map at the end is the Removed set — drop their rows + embeddings. This avoids per-skill SELECT queries and gives O(rows + on-disk skills) instead of O(rows × on-disk skills).
- **Lazy embedder loading via `GetOrInsertWithResult`**: `Option::get_or_insert_with` returns `&mut T`, not `Result<&mut T, E>`. Wrote a one-trait two-line extension to handle the fallible-init case cleanly at the call site. The call reads `embedder.get_or_insert_with_result::<TomeError, _>(|| load_embedder(&paths))?` — no nested `if let None` block. Reusable for any future heavy-init resource.

## Packages & Dependencies

- **`sqlite-vec` vec0 virtual table** — the upstream docs don't prominently mention the REPLACE / ON CONFLICT limitation. The error message is "UNIQUE constraint failed on skill_embeddings primary key", which doesn't obviously point at the virtual-table-doesn't-support-REPLACE issue. Worth checking upstream issues if we touch virtual-table writes again.

## Patterns & Code

- **Mirror `enable_plugin_atomic` shape for `reindex_plugin_atomic`.** Same tx wrapper, same parameter ordering, same closure-based embedder. The diff is the four-way classification (Added/Modified/Removed/Unchanged) and the snapshot-then-drain pattern. Future atomic-batch operations should follow the same template.
- **Library-API entry point alongside CLI `run()`**: `pub fn reindex_catalog_plugins(...)` and `pub fn run_with_deps(...)`. These don't call `Paths::resolve()` and don't construct `FastembedEmbedder`, so they're usable from integration tests with `StubEmbedder`. Same pattern as `lifecycle::enable`. Apply to every new CLI subcommand that needs an embedder.
- **Per-plugin atomicity for batch operations.** Both `catalog update` and `tome reindex` acquire+release the advisory lock per plugin. The spec doesn't require cross-plugin atomicity; per-plugin matches the existing enable/disable contract. A Ctrl-C between plugins leaves earlier plugins committed — by design.
- **Lazy `FastembedEmbedder` pattern**: `Option<FastembedEmbedder>` + `get_or_insert_with_result`. The embedder is ~345 MB resident. Initialise only when there's work that needs it.
- **`PluginNotFound` / `PluginManifestParseError` → auto-disable cascade**: catch the two error variants that mean "plugin gone upstream", call `auto_disable_orphan`, surface a warning. Don't let them propagate up as fatal errors. Other lifecycle errors (`IndexBusy`, `EmbeddingGenerationFailure`, etc.) still propagate.

## For Next Time

- **Write a unit test for every reached branch before consuming it from a new caller.** The `vec0` REPLACE bug would have been a clean slice-1 catch had the existing enable tests exercised the re-embed-of-existing-row branch (which they don't — enable only tests new-row paths).
- **The library-API entry point pattern is now firm.** Phase 8 / US6 (`tome status`) doesn't need an embedder so the pattern doesn't apply there; Phase 9 / US7 (`tome catalog remove --force` cascade) doesn't need one either. But any future `tome <verb>` that needs the embedder should follow `reindex.rs`'s `run_with_deps` shape.
- **The slice-3 CLI binary tests for the parse-error path** are a useful pattern — small, fast, no embedder, cover the public exit-code contract directly. Reuse for any future subcommand with non-trivial scope/argument parsing.
- **`enabled_plugins_for_catalog`** is the third "query the index for a small derived list" function in `index/skills.rs`. The first three (`list_for_plugin`, `mark_all_disabled_for_plugin`, `enabled_plugins_for_catalog`) are still well-organised, but as the surface grows we'll need to think about whether `skills.rs` becomes the right home or whether queries-versus-mutations splits the file.

## Up-front context (slice 1 start)

### Goal

Phase 7 introduces re-embedding of changed skills + auto-disable of orphaned plugins on `tome catalog update`, plus the explicit `tome reindex [scope] [--force]` escape hatch. Three slices planned:

- **Slice 1 (this slice)**: library function `reindex_plugin_atomic` in `src/index/skills.rs` mirroring `enable_plugin_atomic`, plus a `reindex_plugin(id, deps, force)` wrapper in `src/plugin/lifecycle.rs`. Plus `auto_disable_orphan(id, deps)` helper for catalog update's orphan path. Plus unit tests in `lifecycle.rs::tests` covering Added/Modified/Removed/Unchanged + `--force` + orphan deletion.
- **Slice 2**: wire library functions into `src/commands/catalog/update.rs` (walk enabled plugins post-refresh, accumulate summary, surface auto-disable warnings) + `tests/catalog_update_reindex.rs`.
- **Slice 3**: `tome reindex` CLI subcommand (`src/commands/reindex.rs` + `cli.rs` enum) + `tests/reindex.rs`.

### State going in

- All necessary `TomeError` variants exist: `EmbeddingGenerationFailure` (36), `SkillFrontmatterParseError` (23), `ModelMissing` (30), `IndexBusy` (50), `IndexIntegrityCheckFailure` (51). Auto-disable of orphans is a non-error path that emits a stderr warning.
- `index::skills::enable_plugin_atomic` already implements the cheap-reuse semantics (`stored_hash == hash` → flip enabled, no embed). Reindex differs only in three places: (a) does not skip on hash match when `force = true`, (b) deletes rows for skills no longer on disk (Removed), (c) does not require the existing rows to be disabled first.
- `lifecycle::collect_pending_skills` already does the on-disk walk + frontmatter parse + fallback warnings. Slice 1 calls it directly from inside the same module — no need to expose it via `pub`.
- The advisory lockfile + WAL contract from `index::lock` already handles inter-process contention. Slice 1's library acquires the lock the same way `enable` does.

### Slice 1 design

```text
src/index/skills.rs
+ ReindexSummary           # { added, modified, removed, unchanged }
+ reindex_plugin_atomic    # mirrors enable_plugin_atomic; one tx; deletes
                             orphan rows; respects `force` flag

src/plugin/lifecycle.rs
+ ReindexOutcome           # { plugin, summary, duration, warnings }
+ reindex_plugin(id, deps, force) -> ReindexOutcome
                             # acquires lock, calls collect_pending_skills,
                             # calls reindex_plugin_atomic
+ auto_disable_orphan(id, deps) -> u32
                             # called by catalog update when plugin.json
                             # is gone post-refresh; deletes all rows for
                             # (catalog, plugin)
```

### What slice 1 does NOT ship

- No CLI wiring (slice 3).
- No `tome catalog update` integration (slice 2).
- No new error variants — every Phase 7 failure class already has a code.
- No new dependencies.

### Soft size estimate

- `src/index/skills.rs`: ~120 lines (struct + atomic).
- `src/plugin/lifecycle.rs`: +~100 lines public API + ~250 lines unit tests.
- Total slice 1 ≈ 450 lines. Inside the soft cap.
