## Up-front context (slice 1 start)

### State going in

- `plugin::lifecycle::disable(id, paths, config, embedder_seed, reranker_seed) -> Result<DisableOutcome, TomeError>` already exists. The library API was shipped in PR #11 (slice 1a of Phase 3): acquires the advisory lock, runs `disable_locked` under it, returns rows-retained count + duration. Already follows the explicit-release `(LockGuard, Result)` pattern.
- "Already-disabled" detection (FR-008 → `PluginAlreadyInState`, exit 21) is in `disable_locked` and unit-tested in `tests/disable_when_already_disabled_returns_error`.
- Cheap re-enable (FR-006) is already in `index::skills::enable_plugin_atomic` — when `stored_hash == hash` the embedder is NOT invoked; only `UPDATE skills SET enabled = 1` runs. T114 is therefore a *verification* task (read the code, confirm intent matches), with the integration assertion deferred to slice 2.

### What slice 1 needs to ship

- New CLI variant `PluginCommand::Disable(PluginDisableArgs)` with `id: String` and `force: bool`. JSON mode is the global `--json`, inherited like enable.
- New handler `src/commands/plugin/disable.rs` — wraps `lifecycle::disable` with:
  - `--force` short-circuit (skip prompt).
  - Confirmation prompt via `presentation::prompt::confirm` (default no, per `contracts/plugin-commands.md` §"`tome plugin disable`" step 3).
  - Non-TTY without `--force` → `TomeError::NotATerminal` (exit 54) per FR-007 / FR-051. `presentation::prompt::confirm` already enforces this internally, so this falls out for free — but we should still pre-check and emit the documented pointer message to stderr (matching the interactive flow's defensive pattern, P4 §"NotATerminal message duplication").
  - Banner skipped in JSON mode (precedent: `enable::run`).
- Wire into `commands::plugin::run` dispatch.
- No changes to `plugin::lifecycle` or `index::skills`.

### What slice 1 does NOT ship

- Integration tests (slice 2 — `tests/plugin_disable.rs`, cheap re-enable embed-count assertion in `plugin_enable.rs` or sibling, `tests/plugin_repeated.rs`).
- Interactive flow does not need a code change — the existing `interactive::run` already routes Disable via inline `lifecycle::disable` + confirm prompt (P4 slice 1).
- No new error variants. `PluginAlreadyInState` + `NotATerminal` already exist with their exit codes pinned.

### Soft size estimate

- `src/cli.rs`: ~10 lines (new args struct + enum arm).
- `src/commands/plugin/disable.rs`: ~120–150 lines (mirrors the shape of `enable.rs` minus the model-presence dance).
- `src/commands/plugin/mod.rs`: ~3 lines (declare submodule, dispatch arm).
- Total slice 1 ≈ 140–170 lines. Comfortably inside the soft cap; no pre-emptive split.

## Slice 1 — implementation (PR #19)

### Actuals vs estimate

- 4 files / 181 insertions, including the retro skeleton itself. Production code touched: `src/cli.rs` (+13), `src/commands/plugin/disable.rs` (+107, new), `src/commands/plugin/mod.rs` (+2). Estimate of "140–170 production lines" landed at 122 — slightly under, because the disable handler is genuinely simpler than enable (no model-presence dance, no banner needed for a single UPDATE statement).
- All gates green on first push. No mid-flight surprises.

### What worked well

- The library API was so close to the right shape that the CLI handler is essentially a pre-check (`resolve_plugin_dir` for typo-rejection) + a prompt/force gate + a banner + the library call. No new helpers needed; no shared-state surgery.
- The "emit pointer line to stderr BEFORE returning `NotATerminal`" pattern from `commands::plugin::interactive::run` (P4 retro §"NotATerminal message duplication") generalised cleanly. Tests assert the stderr contains `--force`, which is robust to wording tweaks.
- Decision to defer the cheap-reenable assertion to slice 2 was right — slice 1 stays a pure feature ship, slice 2 is pure test coverage. Cleaner PR review surface.

### Gotchas

- None new. The lock-and-call pattern was already in `lifecycle::disable`, so the `(LockGuard, Result)` explicit-release ritual didn't need to be re-applied in `disable.rs`.

## Slice 2 — tests (PR #20)

### What worked well

- **The CLI-binary boundary for disable was a free win.** `lifecycle::disable` doesn't construct `FastembedEmbedder`, so the entire disable CLI surface — `--force`, the non-TTY refusal, repeated-state exit-21, JSON record shape — is exerciseable from a plain `Command::new(tome)`. No pty needed; no fake-embedder threading needed.
- **`StubEmbedder::call_count()` is the right instrumentation hook for FR-006.** The closure-style embedder shared between enable / cheap-reenable lets the same `StubEmbedder` instance be reused across operations, so the counter persists across enable → disable → re-enable. Asserts on `call_count` directly are stronger than relying on `summary.newly_embedded == 0` alone (the latter only tells you the orchestrator's bookkeeping is right; the former tells you the closure genuinely didn't fire).
- **The library-API / CLI-binary split for FR-008 testing.** Enable-of-enabled stays at the library level (handover #10 boundary: real enable CLI loads `FastembedEmbedder`). Disable-of-disabled goes to the CLI binary for an actual `Some(21)` process exit assertion. Both in the same file (`plugin_repeated.rs`) keeps FR-008 coverage discoverable; the mechanism difference is documented inline.
- **`paths_for` promotion at the 4th caller worked exactly as predicted.** Three existing callers converted to imports cleanly; the new test file uses the shared helper from the start. P3 retro's promotion criterion — "3 callers is sticky, 4 is mandatory" — held up.

### Gotchas

- **Subprocess stdin / stdout / stderr inherit TTY-ness from `Command`.** Default `output()` uses pipes, which is the "non-TTY" branch — exactly what the FR-007 / FR-051 test wants. No special setup needed for the non-TTY case; that's the default. Worth recording so future tests don't reach for a custom TTY-disabling shim.

### Patterns to reuse

- **`enabled_row_counts(&Paths) -> (i64, i64)` test helper.** Returns `(total, enabled_sum)` for a fixed `(catalog, plugin)` pair. Useful for any test that needs to inspect `skills` row state — total vs enabled is the most informative split. Worth promoting to `tests/common/mod.rs` if Phase 6+ tests pick it up.
- **CLI-binary tests skip `FastembedEmbedder` when the command path doesn't construct one.** Disable, list, show — all safe to drive from `Command::new(tome)`. Reserve the library-API drive for enable, query (with reranker), reindex, and anything that walks the embedding pipeline. This is the binary-vs-library decision criterion going forward.
