## Up-front context (slice 1 start)

### Goal

Phase 9 / US7 ships the cascade-or-refuse semantics on `tome catalog remove`:

- `tome catalog remove <name>` without `--force` AND the catalog has enabled plugins → exit 53 (`CatalogHasEnabledPlugins`) listing the enabled plugins; no mutation.
- `tome catalog remove <name> --force` with enabled plugins → cascade disable (drop rows for each enabled plugin, all under one advisory-lock acquisition) then proceed with the existing Phase 1 catalog removal (config + cache).
- No enabled plugins → behaves identically to Phase 1 (TTY prompt, cache cleanup, config write).

This is the final feature phase. Phase 10 (polish & cross-cutting) is out of scope per the handover; the agent halts after Phase 9 closes.

### State going in

- `TomeError::CatalogHasEnabledPlugins { catalog, plugins }` already exists in `src/error.rs` (exit code 53) with the contract's error message including the plugin list.
- `index::skills::enabled_plugins_for_catalog(conn, catalog)` already exists from Phase 7 / slice 2 — distinct enabled plugin names ordered alphabetically.
- `index::skills::delete_by_plugin(conn, catalog, plugin)` already exists from Phase 2 foundational — drops `skills` + `skill_embeddings` rows for one plugin inside one transaction.
- `lifecycle::auto_disable_orphan` exists but requires a `LifecycleDeps<'a>` with an embedder reference. The cascade doesn't need an embedder — it's pure deletion. Slice 1 adds a thinner helper `lifecycle::cascade_disable_for_catalog(paths, catalog, plugins, seeds)` that acquires the lock once and deletes each plugin's rows on the same connection.

### What slice 1 needs to ship

- New helper `lifecycle::cascade_disable_for_catalog(paths, catalog, plugins, embedder_seed, reranker_seed) -> Result<u32, TomeError>`: acquires the lock once, opens the index, calls `delete_by_plugin` per plugin, releases on success / drops on failure.
- Extend `src/commands/catalog/remove.rs::run`:
  1. After resolving the entry, query the index for enabled plugins in this catalog.
  2. If non-empty AND `!args.force` → return `Err(CatalogHasEnabledPlugins)`. Exit 53 surfaces.
  3. If non-empty AND `args.force` → run `cascade_disable_for_catalog`. Emit a small "Cascading disable of N enabled plugins" line per the contract.
  4. Continue with the existing Phase 1 flow (TTY prompt for the non-force common case, config write, cache cleanup, emit).
- Adjust JSON output to include a `cascade` array when the cascade fires.

### Tests (slice 2)

- `tests/catalog_remove_cascade.rs` (3 tests):
  - **Refuse case**: enable a plugin, then `tome catalog remove <name>` without `--force` → exit 53, output mentions the plugin id.
  - **Cascade case**: enable a plugin, then `tome catalog remove <name> --force` → exit 0, plugin rows dropped, catalog removed from config.
  - **No-enabled-plugins case**: register a catalog without enabling anything, then `tome catalog remove <name> --force` → exit 0, behaves identically to Phase 1.

The catalog-remove flow doesn't require the embedder (the cascade is pure deletion), so the test path stays library-API for the enable step but otherwise drives the CLI binary.

### What slice 1 does NOT ship

- Tests (slice 2).
- No new error variants — `CatalogHasEnabledPlugins` is pre-allocated.
- No new dependencies.
