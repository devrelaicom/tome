# Phase 10 Retrospective: Polish & Cross-Cutting Concerns (Phase 2 close-out)

## Context

Phase 10 followed the autonomous Phase 7–9 push. Phase 2 feature work was complete at PR #33 (240 tests / 36 suites). The handover requested a multi-agent review pass to surface anything the feature phases missed, fix the blockers in dedicated PRs, then execute the Phase 10 task list.

Total PRs in Phase 10: **#34–#40** (will be **#34–#41** with the closeout PR). 7 PRs, all auto-merged on CI green; 1 macOS msrv flake on PRs #36 and #39 — both cleared on `gh run rerun --failed`.

## What Worked Well

- **Four reviewers in parallel was the right shape.** Contract audit + Rust-lens code review + test audit + security audit dispatched concurrently. Each one had a focused scope and produced a structured report; collation into `review/findings.md` was a 5-minute job. The reviewers found different things — none of the four reports duplicated more than ~20% of another's findings.
- **Reviewer-recommended doc reconciliation over code retrofit.** In every contract-vs-impl divergence flagged (~15 items in PR #35), the reviewer's recommendation was "update the spec, not the code" — because the shipped behaviour was deliberate and the contract was the stale draft. Treating the spec as the moving target made PR #35 a pure-docs PR and avoided shipping a behaviour change cycle.
- **Phase-aligned PR splits worked.** Each PR had a single theme: blocker fix (#34), spec reconciliation (#35), security hardening (#36), small refactors (#37), heavy tests (#38), exit-code E2E + atomicity (#39), docs (#40). No PR exceeded ~400 lines. Reviewers can mentally swap in / out one theme at a time.
- **Pre-allocated test infrastructure paid off.** PRs #38–#39 added ~250 lines of test code without touching any helper signatures — `paths_for`, `fabricate_*`, `config_with_catalog`, `copy_sample_plugin_catalog` all existed and fit. The investment from earlier phases keeps amortising.
- **Coverage matrix as test doc.** `tests/exit_codes_e2e.rs` documents (in a module-level comment) which Phase 2 exit codes are E2E-tested vs library-level vs deferred to manual repro. The "deferred" entries are explicit rather than hidden — making the gap auditable in one place.
- **Cross-process lock test via cheap path.** `tests/concurrency.rs` uses `tome catalog remove --force` against an enabled-plugin catalog to exercise the lock — the cascade path is pure deletion, so no `FastembedEmbedder` load. The pattern reuses the existing library/CLI test boundary.

## What Didn't Work

- **macOS msrv flake recurrence.** Same rustup-shim signature (`error: unexpected argument 'fmt' found`) hit PR #36 and PR #39. Both cleared with `gh run rerun --failed`. The flake is in the `actions/setup-toolchain` step, not in any of our code. Worth filing a Renovate-tracked CI workflow update at some point but not a Phase 10 blocker.
- **Local `git pull --ff-only` rejected mid-flow.** Same `.midnight-expert/settings.local.json` regeneration issue that P9 retro flagged. The `git stash push -u` step sometimes stages files instead of stashing on the first attempt; the workaround is `git reset HEAD <path>` + retry the stash. Re-iterated here as a documented harness quirk.
- **`tests/exit_codes.rs` rustfmt edited my code mid-commit.** The pre-commit hook ran `cargo fmt --check` which caught a too-wide `assert!` invocation; I had to fmt + re-add. Not blocking but worth noting that the hook is correctly enforcing fmt — write to clippy/fmt-pass shape from the start when typing fresh code.

## Workarounds & Solutions

- **Pulling main on stale `.midnight-expert/`.** Sequence that worked reliably:
  ```sh
  git reset HEAD review/                                    # unstage anything we don't want
  git restore --staged .midnight-expert/settings.local.json # if accidentally staged
  git stash push -- .midnight-expert/settings.local.json
  git pull --ff-only
  ```
  Don't combine `git stash push -u` with `git pull` in one chained command; separate them so each step can fail independently with a readable error.
- **Documenting deferred test coverage in-source.** The `tests/exit_codes_e2e.rs` module doc contains a coverage matrix that names each Phase 2 exit code, where it's tested, and why the embedder/inference codes are deferred to library-level (~345 MB ONNX cost). Keeping this in the test file rather than only in the spec means anyone running the test suite sees the audit trail.
- **`mid_stream_connection_drop` as SIGINT proxy.** T196 asked for a slow-trickle interrupt test with `CANCELLED` mid-stream flip. The global `CANCELLED` static is shared across the test binary's parallel tests — flipping it from one test races with every other. The proxy: a server that delivers part of the body then drops the socket. The cleanup path is identical (both `Interrupted` and `Io(connection reset)` propagate through `download_model`'s pipeline closure to the partial-dir cleanup). Document the proxy in the test comment; the actual SIGINT signal is covered by the source-visible `was_cancelled()` check + manual repro.

## Patterns & Code

- **Multi-agent review as a phase boundary.** When wrapping up a substantial body of work, dispatch reviewers across the four orthogonal axes (contract correctness, code quality, test coverage, security/supply chain) in parallel. Collate into a single triage table; pick a PR shape per theme.
- **Severity → PR shape mapping.** Blocker = dedicated PR before any other work. Major = dedicated PR. Minor = fold into the relevant polish slice. Nit = wontfix unless trivial-while-nearby. Applied this rule mechanically; it kept the PR count proportional to the surface area.
- **Reviewer-deferred items belong in the retro.** Test gaps with named alternatives (library-level coverage, manual repro instructions) ship in the test file's module doc. Bigger refactors get a one-line entry in this retro + the disposition.md.
- **Single-source-of-truth for the closed error set.** PR #37 collapsed three `_force_*` unused-import idioms — the pattern of "preserve an import via a dummy function" doesn't survive refactors. If clippy flags an unused import, delete it; let the compiler tell us when it's needed again.
- **`pub fn x_for_tests()` is not a thing in this codebase.** All test-only helpers are gated on `#[cfg(test)]` and stay `pub(crate)`. Integration tests that need crate-internal helpers either route through a public API or live in `src/` as unit tests. The discipline keeps the public surface small.

## What's Deferred (post-v0.2.0 candidates)

These were surfaced by the review pass and consciously not shipped:

- **Read-only DB open refactor.** `commands::plugin::open_index_for_read` and `commands::status::*` use a read-write open today. The security audit flagged it; PR #35 dropped the `writer_pid` claim from `status.md` which lowered the urgency. A future PR can plumb `OpenFlags::SQLITE_OPEN_READ_ONLY` through these paths.
- **`tome query` library entry point.** Tests for `--strict` (exit 40), embedder drift (41/42), and `--no-rerank` `scoring` field require either a `query::run_with_deps`-style library bypass (precedent: `commands::reindex::run_with_deps`) or real embedder loading. Both unblocked once that entry point lands.
- **Phase 2 `TomeError` Display tests.** `tests/error_messages.rs` covers only `ManifestInvalid` variants today. ~10 Phase 2 variants ship without Display-format assertions — low-risk because the closed-set discipline catches missing variants at compile time, but tighter tests would catch wording regressions.
- **`ModelManifest` strictness grep guard.** `tests/manifest_strictness.rs` checks `src/catalog/manifest.rs` and `src/config.rs` but not `src/embedding/registry.rs`. Add the path + a negative test that proves an unknown field in a `manifest.json` is refused.
- **`tome catalog update --json` CLI-binary schema test.** Library-API tested via `catalog_update_reindex.rs`; the JSON envelope shape is not asserted at the binary boundary.
- **Byte-progress callback on `download_model`.** TD-010 in `CONCERNS.md`. Currently `tome models download` shows an indeterminate spinner; a determinate bar requires a callback.
- **`tome show` git-log `Last updated:` integration.** Author surfaces today; the timestamp + "X ago" cell renders as `—`. PR #35 documented the partial state in the contract.
- **Manual SC-001 / SC-002 against real BGE models.** SC-001 = `tome plugin enable` end-to-end timing on a real laptop with real models. SC-002 = canonical query produces a relevant skill in the top 3 against a known good catalog. Tracked in T088 (P3 retro) and T202/T203 (this phase's task list). Both need a developer with the models installed; CI cost makes them unsuitable for automation.

## For Next Time (Phase 11 if it happens)

- **Run the multi-agent review at every phase boundary, not just the end.** Phase 10 caught issues that had been sitting since Phase 4–7. Running the same four reviewers after each user story would have surfaced them earlier — at lower repair cost.
- **Document "deliberate divergence from contract" in the contract itself, not in code comments.** Both the `tome query` no-prompt design and the `tome models download` spinner-not-bar design lived only in source comments before PR #35. The contract still claimed the older behaviour. Future intentional divergences should land in the spec at the same time as the code.
- **Pre-allocate error variants for anticipated cases.** Worked for `CatalogHasEnabledPlugins` in Phase 9; the variant existed since Phase 2 foundational. Cheap to add; saves a cross-file edit later.
- **The library/CLI test boundary is firm now.** Heavy-state paths → library + `StubEmbedder`; light/error paths → CLI binary. Don't try to test FastembedEmbedder paths through the CLI binary.
- **Closeout the artefacts: review/, disposition.md, retro/Pn.md.** Phase 10's review/ directory contains five files that capture the reviewer pass. Future phases can consume them as inputs to the next planning round.
