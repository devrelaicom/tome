## Workarounds

### Bounded char_indices walk replaced O(n) truncate_description
US4.d C-2 / Security HIGH: pre-fix `truncate_description` did TWO `chars()` passes per result (early `chars().count()` for `<=max` check + `chars().take().collect()` for truncation). Even when no truncation occurred, the full input was walked. With caller-controlled `description_max_chars` × `top_k` results × multi-KB descriptions, this was a measurable DoS amplifier. Fix: single-pass `char_indices` walk — at most `max+1` chars examined, then either return verbatim (no truncation) or slice at the (`max+1`)-th char's byte offset and append the ellipsis. No `take().collect()` allocation in the truncation path; no full-string traversal in the no-truncation path.

### Orchestrator brief had a stale assumption about commands+searchable
The US4.c brief said "commands default `searchable=false`" — sub-agent corrected: `resolved_searchable()` returns `true` unless `disable-model-invocation: true` is explicitly set. Commands DO appear in search results by default; the exclusion test sets the opt-out flag. Lesson: don't assume from contract; verify against implementation. Briefs should say "test what the contract says" not "test what I think the contract says".

### Stub MetaSeed pattern in test files
US4.c tests needed `MetaSeed` matching `StubEmbedder`'s identity. Declared `static STUB_EMBEDDER_ENTRY: ModelEntry { name: "stub-embedder", version: "0" }` (and corresponding reranker) inside the test file. The `mcp_get_skill_info` tests don't hit this because they never run the embedder/reranker pipeline. Promote to `tests/common/` when a third caller needs it.

### invalid_kind + resource_enum_failed envelope tests deferred
Both are documented error envelopes but hard to trigger artificially:
- `invalid_kind`: caller would send `{"kind": "skil"}` → rmcp serde dispatch rejects before reaching our handler (returns a generic INVALID_PARAMS not our slug)
- `resource_enum_failed`: requires filesystem permission tricks (read-only dir mid-walk)

Test surface accepts these as deferred test coverage. The wire envelopes are documented in the contract; the production code paths exist; the test gap is in artificial triggering, not in implementation.

## Patterns established

### Bounded-walk discipline for caller-controlled string operations
When a caller supplies a "max" parameter that bounds output, the implementation should be O(max) not O(input). Pattern: use `char_indices` to walk at most `max+1` characters then stop; never call `chars().count()` as an early-exit gate. Applicable to any string-truncation surface where caller controls the cap.

### Sanity cap above documented contract surface
`MAX_DESCRIPTION_MAX_CHARS = 100_000` is INTENTIONALLY above the documented contract behaviour. `u32` deserialisation handles the negative branch (FR-092's `< 0` rejection); the constant catches "absurd but legal u32 values" that no real agent would request. Pattern: when serde handles one rejection axis, add a code-level constant for the orthogonal axis that the type can't reject. Document both branches in the contract.

### BTreeMap for byte-stable JSON object keys
`get_skill_info`'s resource enumeration uses `BTreeMap<String, ResourceSubdir>` instead of `HashMap` so the JSON serialisation produces alphabetically-ordered keys deterministically. Beats "HashMap + post-sort" pattern. Pattern: when JSON output must be byte-stable AND has object-shaped data, default to BTreeMap.

### `walk_resources` symlink defence carried from US1.d
Per-entry `file_type()` lstat skip (matches `get_skill::walk_dir` pattern). Residual TOCTOU between lstat and second `read_dir(sub)` documented as accepted risk per Phase 4 trust model (catalog is trusted-on-enrol, not trusted-on-read). Hardening to `cap-std`'s `openat`/`O_NOFOLLOW` deferred to v0.6+ if a real threat materialises.

### `EntryKind` JsonSchema for single source of truth
Adding `schemars::JsonSchema` derive to `EntryKind` (US4.a) meant the new tool's input schema could reference it directly without a local wrapper enum. Single source of truth — kind values stay in sync across serialisation, deserialisation, and JSON schema generation.

### Reviewer pass severity continues to drop
US1.d: 1 BLOCKER + 8 majors. US2.d: 2 BLOCKERS + 6 majors. US3.d: 0 BLOCKERS + 2 majors-applied. US4.d: 0 BLOCKERS + 1 HIGH + 2 majors-applied. The structural fixes from earlier USs continue to pay compounding dividends — each subsequent US inherits cleaner architecture and surfaces fewer issues. The US4.d HIGH was a perf/security finding (truncation O(n)) that a non-substitution-trained eye would have missed.

## For next time

### Hot-path string ops deserve a sanity-cap perf test
US4.d C-2 was caught by the reviewer pass, not by tests. A perf-tier test pinning that `truncate_description("a".repeat(N), small_max)` doesn't allocate proportionally to N would have caught the regression at write time. Worth a future test idiom for any "max" parameter that bounds output: assert "small bounded output regardless of large input".

### Multibyte truncation tests are cheap and high-signal
The added `truncation_at_multibyte_char_boundary_does_not_split_codepoint` test is 30 lines + uses an emoji. Catches Unicode boundary bugs cheaply. Pattern worth carrying to any future "char-bounded slice" surface.

### Stub MetaSeed pattern ready to promote on next consumer
3rd test file with file-local stub MetaSeed. Next consumer in US5 should promote to `tests/common/mod.rs`.

### Reviewer pass remains the structural-safety net
US4.d HIGH (truncate_description DoS) would NOT have been caught by:
- Contract review (contract is silent on perf)
- Test review (no test exercised large descriptions × top_k)
- Security audit alone (the connection between description_max_chars and CPU cost was non-obvious until the Rust reviewer surfaced the double-pass)

Took the Rust-lens reviewer specifically calling out the double-iteration to surface this as HIGH. Pattern: keep all 4 reviewer perspectives even when previous slices were clean. The reviewer cost is amortised across the phase; the catch frequency is unpredictable per-slice.

### `cap-std` evaluation worth doing in v0.6+
Multiple US closeouts have flagged TOCTOU residuals on filesystem walks (US1.d, US4.d). `cap-std` would give us `openat`/`O_NOFOLLOW` and make the symlink defence structurally airtight. Worth a v0.6 spike: dependency footprint check + binary-size impact + API ergonomics.
