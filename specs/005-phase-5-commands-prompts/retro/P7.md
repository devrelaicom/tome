## Workarounds + surprises

- **`unchecked_transaction()` for snapshot consistency on read-only connection**. `open_read_only` produces a non-writing connection, but SQLite still serves snapshot semantics inside a transaction. We needed the wrapper because `query_map` over two different `prepare`s could see two different snapshots. `unchecked_transaction()` is the right name (the "unchecked" refers to nesting; doesn't validate that you're allowed to write — which is what we want here, since we don't intend to write anyway). Implicit rollback on Drop is also what we want — no `tx.commit()` call, just let it fall out of scope.

- **`tracing::warn!` "silent-bail" middle ground.** US5.c R-m3 surfaced 3 sites where `read_dir` errors were silently swallowed via `Ok(_) => ..., Err(_) => return out`. The wrong fix would have been propagating the error (the diagnostic surface must never crash per FR-561). The right fix turned out to be `match` with explicit `NotFound` short-circuit + `tracing::warn!` for all other errors. NotFound is legitimate state (an unbootstrapped install has no `plugin-data/` dir); permission-denied or transient I/O failure is operationally relevant and shouldn't be invisible.

- **The reviewer-pass discipline saved the closeout cycle from quietly shipping latent issues** for the 5th US in a row. Even though US5 produced the cleanest pass, the 4 reviewers still surfaced 4 majors that would have been silent maintenance drift (the layout constant duplication, the snapshot inconsistency, the unbounded description, the misleading carve-out doc). Each was a 2-line fix in the right hand. Not catching them now would have cost more later.

- **Mapper run during background `cargo test`**. The full test suite (`cargo test --workspace`) runs ~10-15 min on a quiet machine; under CPU contention from concurrent rustc processes on other projects it stretches past 30 min. Dispatching mappers (read-only, mostly file reads) during the run is fine — they don't compete for the same CPU shape. **Pattern**: while waiting on long-running validation, dispatch the closeout-stage parallel work.

## Patterns established (Phase 5 cumulative)

- **Single-source-of-truth accessors at first cross-module consumer.** Phase 5 reinforced this 3x: `Paths::plugin_data_root()` (US5.c R-M1), `Paths::workspace_data_dir_for` (US1.c), `MCP_SLASH_PREFIX` (US5.c R-m11). The trigger to promote: at the first call-site that DUPLICATES an inline layout/string convention from a writer site. The rule-of-3 still holds for non-layout helpers; layout convention deserves promotion at 2 sites.

- **Snapshot consistency for multi-statement diagnostic reads via `unchecked_transaction()`** on `open_read_only` connections. The pattern is now established for any future doctor / status surface that reads multiple tables in sequence. Drop-rollback is automatic.

- **Output `Serialize` types stay lenient.** Phase 5's `PromptsReport`, `OrphanDataDirReport`, `EntryCountsByKind`, `CollisionRecord`, `ResolvedCollisionEntry`, `EntryIdentity`, `EntryView`, `EntryKind` all carry `Serialize` without `#[serde(deny_unknown_fields)]`. The strictness boundary applies to inputs only; the rule continues to hold across all Phase 5 surfaces.

- **`Option<*>` field semantics in doctor reports**: `None` means "explicitly no user intent" (e.g. implicit GlobalFallback), NOT "not applicable" or "errored". Errored states route through `tracing::warn!` + `None` — but the docstring should clarify what "no value" means in the specific surface. R-M5 was the doc fix that taught this lesson.

- **Negative + positive test pairs for boolean-logic features.** T-G1 reinforced this — testing that `[dormant]` IS present requires also testing it's NOT present under the opposite states. Standard pattern from here on for any annotation / classification with boolean inputs.

- **Exact-count assertions over deterministic fixtures.** T-W1 promoted `assert!(counts.skills >= 1)` to `assert_eq!(counts.skills, 4)`. The lower-bound form was masking off-by-one regressions the deterministic fixture should reveal. Lower-bound is reserved for genuinely-non-deterministic state (timing-sensitive heuristics like `pending_re_embedding`).

- **4-reviewer parallel pass dispatched as ONE message with 4 Agent calls.** Established across all 5 USs; the cost is ~5 min wall clock for the parallel work. Saves vs serial reviewer dispatch by ~15-20 min per closeout. The pattern is now firm for v0.6+ as well.

- **Findings + disposition committed BEFORE applying fixes.** Two-commit pattern (`docs(review):` + `fix(...):` / `test(...):`) keeps the audit trail clean for posterity. The disposition doc explicitly records "applied vs deferred" so future readers (or v0.6+ Polish) can find the deferred backlog.

- **Reviewer pass severity descends through a phase.** Phase 5 trend: US1.d 1B+8M → US2.d 2B+6M → US3.d 0B+2M → US4.d 0B+1H+2M → US5.b 0B+4M. As discipline solidifies (closed-error-enum, byte-stable pin, FR-124-style invariants, single-sweep regex, bounded loops), each successive reviewer pass surfaces fewer issues. Phase 6 should expect a baseline 0B + 2-4M ceiling — anything higher means a pattern regressed.

## Decisions deferred to v0.6+ / Polish backlog

- **R-M2 — orphaned-workspace-dir flood in `OrphanDataDirReport`**. When an entire workspace dir is on disk but has no row in the `workspaces` table (e.g. `tome workspace remove` was interrupted mid-cascade), every `(catalog, plugin)` triple under it gets emitted as an orphan. Should be ONE entry "workspace `foo` not in DB" instead. Requires either a new `workspace_dirs: Vec<PathBuf>` field on `OrphanDataDirReport` or refactoring `workspace_data` to carry workspace-root paths when the workspace itself is orphaned. Either choice is a contract amendment + Phase 4 doctor JSON shape impact.

- **6 cosmetic Rust nits + 1 naming convention** (R-m1, R-m6, R-m7, R-m8, R-m9, R-n1, R-n2). Tracked in `us5-disposition.md`.

- **MEDIUM symlink path disclosure in doctor output**. Doctor emits absolute paths to orphan dirs; the walk SKIPS symlinks (correct), but the path itself is disclosed. Under the trust model this is accepted-risk; v0.6+ cap-std hardening would moot it entirely.

- **LOW terminal escape sequences in plugin/entry names**. The lenient frontmatter parser doesn't sanitise control characters. v0.6+ could add parse-time control-char rejection or display-time escaping; for now the trust boundary (catalog trusted-on-enrol) is the gate.

- **TOCTOU residual in `walk_resources`** (from US4.d disposition; carried forward). Same resolution path as above — cap-std hardening.

- **Read-only DB open refactor for Phase 4 read-side calls** (from Phase 3 backlog). Still tracked; not on the v0.5.0 critical path.
