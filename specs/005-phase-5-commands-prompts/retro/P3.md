## What this phase will land

The headline Phase 5 deliverable: commands accessible as slash commands across
harnesses via the MCP prompts capability.

- Schema v2 → v3 migration (kind/searchable/user_invocable/when_to_use columns)
- Lenient frontmatter parsing for new fields (`arguments`, `when_to_use`,
  `searchable`, `user-invocable`)

- Commands directory walk in `plugin::components`
- MCP `prompts` capability (`prompts/list` + `prompts/get`)
- Prompt name derivation (`<plugin>__<entry>` format with sanitisation +
  truncation + collision counter-suffix)
- Wiring substitution::render() into prompts/get (still a no-op stub from F3;
  US2 + US3 wire real transforms)

## Workarounds

### F2 reduced to no-op (Foundational, surfaced before US1)
`regex = "1"` was already a direct dep since Phase 1 — research §R-2's "promote from transitive" was stale. F2 reduced to a binary-size record. Documented in P2.

### F3 contract amendment: `regex_sets.rs` not `regex.rs`
The original tasks.md T027 name shadowed the `regex` crate inside the module. Renamed to `regex_sets.rs` inline. Documented in P2.

### US1.a schema migration: SQLite autoindex naming
The contract's literal `DROP INDEX IF EXISTS skills_unique` no-ops against a real v2 DB because SQLite materialises inline `UNIQUE` table constraints as `sqlite_autoindex_*`, not as a developer-named index. Used the SQLite "12-step" table-rebuild pattern (mirroring Phase 4's `phase_4_v1_to_v2`). Contract amendment documented inline.

### US1.a framework FK toggle promotion
SQLite silently ignores `PRAGMA foreign_keys` inside a transaction. The Phase-4 migration's belt-and-braces toggle in the migration body was therefore a no-op. Promoted the toggle to `apply_pending` (disable before each step's transaction; restore on commit/error). Both Phase 4 and Phase 5 migrations now benefit. Beneficial side-effect.

### US1.a SCHEMA_VERSION cascade through bootstrap
`src/index/schema.rs::CREATE_STATEMENTS` had to be updated so a fresh install creates the v3 `skills` table directly. Otherwise a fresh install would bootstrap at v3 but with v2-shape columns.

### US1.b rmcp 1.7.0 deviations (4 documented)
The T122 verification report (`notes/rmcp-prompts-api.md`) records 4 deviations from the contract:
1. Alias `rmcp::model::Prompt` directly as `PromptDescriptor` (avoid double-marshalling)
2. Dynamic registration via `PromptRoute::new_dyn` (NOT `#[prompt_router]` macro — Tome prompts aren't compile-time-known)
3. `PromptListResponse` wrapper unnecessary
4. `PromptsCapability` declared via typestate builder (non_exhaustive struct)

### US1.b lifetime wrinkle: `PromptRoute::new_dyn` HRTB
The trivial `move |ctx| Box::pin(async move { ... })` shape fails the HRTB because the inferred future type doesn't carry `'a` and `Box` is invariant. Resolved via free `fn stub_future<'a, S>(...) -> Pin<Box<dyn Future + Send + 'a>>` whose return type binds the input lifetime.

### US1.b `#[prompt_handler]` macro unusable
Defaults to `Self::prompt_router()` (macro-generated) and expects a static set of `#[prompt]`-marked methods. Tome's prompts are workspace-driven. Manual `list_prompts` + `get_prompt` `ServerHandler` methods inside the existing `#[tool_handler] impl ServerHandler` block.

### US1.b latent path bug (fixed in US1.c)
`get_skill::Output.path` was `PathBuf::from(&row.path)` treating `row.path` as absolute. `skills.path` actually stores RELATIVE paths. US1.b's PromptRegistry resolved this correctly via a private helper. US1.c promoted that helper to `resolve_entry_body_path`, consumed by both `prompts/get` and `get_skill`.

### US1.c `time::now_local()` requires feature flag
`time::OffsetDateTime::now_local()` needs the `local-offset` cargo feature, not enabled in Tome's tree. Used `now_utc()` as default with `SUBSTITUTION_CLOCK_OVERRIDE` test seam.

### US1.d sub-agent didn't auto-commit (US1.b reprise)
First US1.b sub-agent dispatch left work uncommitted, claiming "test still running". The orchestrator verified locally and committed. Pattern: prefer running `cargo test` + `git commit` in the orchestrator after sub-agent return rather than relying on sub-agent self-verification — round-trip cost is the same.

## Patterns established

### 4-reviewer parallel-pass discipline
Phase 4's established pattern carries forward cleanly. 4 agents (contract / Rust-lens / test / security) dispatched in a SINGLE message (multiple Agent tool uses) with self-contained briefs. Each writes a structured report to `/tmp/tome-phase5-us1-{contract,rust,test,security}.md`. Orchestrator consolidates to `review/us1-findings.md` + `us1-disposition.md`. Apply blockers + selected majors; defer rest with explicit rationale.

### Latent-bug surfacing via slice composition
US1.b's `PromptRegistry` exercise of catalog path resolution surfaced the latent `get_skill` abs/rel path bug. The bug was pre-existing in `get_skill` but no integration test exercised the file-read branch. US1.c's promotion of the shared resolver fixed it. Lesson: when a new slice adds a code path that mirrors an existing one, check the existing one for parallel correctness.

### Contract amendments inline at fix time
Phase 4 P3 retro pattern continues: when a contract issue surfaces mid-implementation (US1.a's autoindex naming; US1.d's new exit code 9 for `PluginDataDirWriteFailed`), amend the contract in the SAME commit as the fix. Avoids re-litigation.

### Pre-allocation slice (F3 archetype)
F3's `src/substitution/` module ships with `#[allow(dead_code)]` on every internal helper and stub-returning public APIs. Production consumers (US2, US3) light up the stubs in subsequent slices. Each `#[allow(dead_code)]` carries a doc comment naming which US activates it. The override-seam statics (`SUBSTITUTION_CLOCK_OVERRIDE`, `PLUGIN_DATA_DIR_OVERRIDE`, `WORKSPACE_DATA_DIR_OVERRIDE`) + matching RAII test guards land in F3 even though no test exercises them yet — keeping the seams testable from day one.

### Sub-agent for high-LOC slices, orchestrator for fixes
US1.a + US1.b + US1.c each dispatched a `devs:rust-dev` sub-agent for the heavy implementation work. US1.d dispatched a single sub-agent for fix application (the disposition was already split, brief was concrete). Reviewer pass + retro fill + mapper refresh + CLAUDE.md updates handled by orchestrator. Sub-agents shine for self-contained "implement these tasks on these files" briefs; orchestrator handles coordination + judgement calls.

### JSON wire-shape pin discipline
Every new `#[derive(Serialize)]` type that flows through `--json` or MCP gets a byte-stable pin test in the SAME PR. US1.b: `prompts/list`. US1.c: `prompts/get`. US1.d: 4 error envelopes (`prompt_not_found`, `prompt_argument_mismatch`, `substitution_failed`, `workspace_data_dir_write_failed`, `plugin_data_dir_write_failed`). Phase 4 P8 lesson reinforced.

### `PoisonError::into_inner` recovery on mutex test seams
Established Phase 4 P5 pattern carries through to F3's three new override-seam statics. Tests don't panic when prior tests panicked.

### `MAX_ARGUMENTS = 256` size cap at parse boundary
S-M1 fix establishes a defensive cap on parsed YAML list length at the third-party-input boundary. Pairs with existing `bounded_read_to_string` discipline. Future lenient-parse surfaces should default-cap list lengths.

### Mapper-refresh-at-phase-boundary
Phase 4 P8 retro recommended mid-phase mapper rotation. US1 close runs `/sdd:map incremental` with 4 parallel mappers — captures both the structural delta (new modules, schema column changes) and conventions delta (new RAII guards, override-seam pattern, byte-stable JSON pin pattern). The 8 docs refresh in parallel, ~5 min wall clock.

### Reviewer pass dispatch tip
The 4 reviewer briefs should each include:
- The 3-5 specific contracts they audit (not just "the contracts dir")
- The "what NOT to flag" section listing pre-approved deviations (don't re-litigate)
- File:line-cite output format
- Where to write the report
- Severity tiers explicitly (BLOCKER / MAJOR / MINOR)

US1.d's briefs were tight. ~5-7 min per reviewer in parallel.

### Sub-agent verification pattern
When dispatching, ALWAYS specify: "Run `cargo fmt && cargo clippy --all-targets --all-features -- -D warnings && cargo test` BEFORE committing". The first US1.b sub-agent dispatch skipped this; the orchestrator had to verify and commit manually. Adds 1-2 min to the sub-agent's wall time but eliminates the orchestrator round-trip.

### Disposition.md is the contract for the closeout PR
The closeout PR's body should mirror the disposition.md structure: "Applied: X. Deferred: Y." Reviewers scanning the PR diff want a one-screen view of what landed vs what didn't.
