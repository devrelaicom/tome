# tome-site Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Turn the vanilla Docusaurus scaffold in `tome-site` into the Tome public-beta promo + docs site, implementing the locked "workshop manual" visual identity, the custom landing page, the trimmed docs IA, a launch blog post, and Netlify deployment at `tome-mcp.com`.

**Architecture:** Docusaurus 3.10 classic preset. The landing page is a custom React page (`src/pages/index.tsx`) composed of section components in `src/components/landing/`, styled with one CSS module + global theme tokens in `src/css/custom.css`. Navbar is themed via config + CSS; the dark "back-cover" footer is a swizzled `@theme/Footer`. Docs and blog use standard Docusaurus content. Fonts are self-hosted via `@fontsource` (no CDN). Light theme is pinned for v1.

**Tech Stack:** Docusaurus 3.10.1, React 19, TypeScript, pnpm, `@fontsource/space-grotesk`, `@fontsource/ibm-plex-mono`, Playwright (smoke tests), Netlify (host).

**Design source-of-truth:** `specs/2026-06-01-tome-site-design.md` (spec) and `specs/reference/landing-mockup.html` + `specs/reference/hero-mockup.html` (approved visual mockups — exact colours, type, spacing values live there). `specs/reference/tome-exploded-v1.png` is the approved hero illustration.

**Testing strategy (read this first):** This is a presentational content site, not a logic library — so "tests" are: (1) `pnpm typecheck` clean, (2) `pnpm build` succeeds (Docusaurus `onBrokenLinks: 'throw'` makes the build a link-integrity test), (3) a Playwright **smoke test** asserting the landing renders with all sections + nav works. Add component unit tests **only** where there is real logic (there is almost none here). Commit after every task.

```markdown
# Security model

Tome treats catalog content as untrusted **input** and defends the *mechanical* boundary. It cannot — and does not claim to — vet the *content* a catalog ships.

## What Tome defends (the mechanical boundary)

- Bounded reads (no out-of-memory from a hostile manifest)
- Path-segment validation (no path traversal)
- Symlink-safe writes (no escaping target directories)
- Managed-marker collision refusal (no corrupting your config files)
- Credential scrubbing on every git/network/log boundary

A hostile catalog cannot crash Tome, escape its target directories, or corrupt your files.

## What Tome cannot defend (the semantic boundary)

A catalog's skills, commands, and agents are **instructions your AI agent executes**. Tome delivers them faithfully; it cannot judge whether they are benign. A hostile `SKILL.md` can tell your agent to exfiltrate secrets or run destructive commands — and it won't look like "code."

**Adding a catalog is trusting it — exactly like `npm install` or running a script. Only add catalogs you trust.**

## Reporting a vulnerability

See `SECURITY.md` in the Tome repository.
```
