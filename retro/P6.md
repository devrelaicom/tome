# Phase 6 Retrospective: Polish & Cross-Cutting Concerns

## What Worked Well

- **`--help` output reads naturally** without manual `long_about` overrides.
  clap synthesises the per-command help from the doc comments on the
  `CatalogCommand` variants and the `clap::Args` field doc comments. Every
  flag, including the global `--json` and `-v`/`-vv`, surfaces on every
  subcommand.
- **Release binary on macOS arm64 is 2.7 MB** — comfortably inside the 10 MB
  cap (SC-010). Linux size should track within ~10 % once the CI matrix
  reports it. The `strip = "symbols"`, `lto = "thin"`, `codegen-units = 1`
  release profile is doing real work.
- **The cross-cutting success criteria were already covered.**
  - SC-006 (credential scrubbing): `tests/scrubbing.rs` (8 rows) +
    `tests/catalog_add.rs::git_failure_with_credential_url_is_scrubbed` are
    sufficient. No additional tests added in this phase.
  - SC-007 (human/JSON identical exit codes and side effects): each
    catalog_*.rs integration suite covers both modes for the happy paths.
  - SC-008 (non-TTY harness for prompts):
    `tests/catalog_remove.rs::non_tty_without_force_exits_2` drives the
    only interactive prompt with `Stdio::null()` and asserts exit 2.

## What Didn't Work

- **SIGINT integration tests (T126) are still missing.** Real signal
  injection against the binary needs to spawn the child, kick a clone
  that's slow enough to be interrupted, send `SIGINT`, and assert exit 8.
  The signal-handler machinery in `src/catalog/git.rs` is exercised by
  the unit-level `was_cancelled()` API but not by an end-to-end test.
  Documented in CHANGELOG / spec as "covered by code review and the
  unit-level cancellation path"; a follow-up issue should land an
  end-to-end test using `nix::sys::signal::kill(child_pid, SIGINT)`.

## Workarounds & Solutions

- The CHANGELOG entry for `[Unreleased]` now lists user-visible commands
  separately from project-level scaffolding. Future entries should keep
  that split — releases ship behaviour, scaffolding ships once.

## Packages & Dependencies

- No new dependencies in this phase.

## Patterns & Code

- **Doc-comment density.** Every public item has at least an inline comment
  explaining *why* it exists (per the constitution). Module-level `//!`
  comments orient the reader. There's no per-item rustdoc-style
  description on private helpers — that's deliberate; the reader knows
  Rust.

## For Next Time

- The "covered by code review" gap on SIGINT is a real one. Phase 2
  (post-Phase-1) should add `nix` to dev-dependencies and write the
  end-to-end interruption tests on top of the existing `Git` helper.
- The CHANGELOG split (user-visible vs project-level) is a tiny pattern
  worth keeping. Migration-driven changes belong under user-visible;
  CI / lint / dep bumps belong under project-level.
