# Phase 3 Retrospective: User Story 1 — Register and Inspect a Remote Catalog

## What Worked Well

- **The Phase 2 atomic-write helpers shipped untouched.** `store::save` and the
  `tempfile::NamedTempFile::persist` machinery worked the first time against
  the real subcommand bodies. The integration tests confirmed atomicity in
  practice (registry write rolling back when subsequent steps fail).
- **`file://` URLs as a Git source.** Spinning up a fixture catalog with
  `tempfile::TempDir` + `git init -b main && git add -A && git commit -m init`
  and pointing `tome catalog add` at it gave the integration tests real-Git
  coverage without any mocks (constitution VIII).
- **`CARGO_BIN_EXE_tome`.** Cargo populates this env var with the absolute
  path of the freshly-built binary for the package, so integration tests
  don't have to second-guess the target directory layout. Used by
  `tests/common/mod.rs::tome_bin()`.
- **Splitting `commands/catalog.rs` into a directory module.** Each subcommand
  has its own file (`add`, `list`, `show`, `update`, `remove`, `source`),
  keeping the per-file diff small and the responsibility boundary obvious.

## What Didn't Work

- **First credential-scrubber pass leaked through the `catalog` field.** The
  scrubber was applied to `git`'s stderr but not to the source string passed
  into `Git::new(...)`. When git failed, the formatted error was
  `git failed for `<unscrubbed source>`: <scrubbed stderr>`. Fix: scrub the
  source string before constructing the `Git` helper in `commands/catalog/add.rs`.
  Caught by `tests/catalog_add.rs::git_failure_with_credential_url_is_scrubbed`
  on the first run.
- **`directories::ProjectDirs` routes through `~/Library/Application Support`
  on macOS** regardless of `XDG_*` env vars. The integration tests rely on
  isolating XDG paths via env vars; the dependency had to go. Replaced with
  ~20 lines of hand-rolled `XDG_CONFIG_HOME` / `XDG_DATA_HOME` resolution in
  `paths.rs` that works identically on macOS and Linux. The `directories`
  crate is now gone from `Cargo.toml` entirely.

## Workarounds & Solutions

- **`git clone --depth 1 --branch <sha>` doesn't work** with arbitrary SHAs
  (only branches and tags). The SHA-pinned no-op test patches the registry
  entry directly to set `ref = "<sha>"` after a normal add, instead of trying
  to add with `--ref <sha>` against a shallow clone. Recorded in the contract;
  matches the FR-008 semantics ("`tome catalog add --ref <sha>` is the only
  way to set a SHA pin", but Phase 1's `add` SHA support is best-effort and
  is documented separately).
- **`gh repo create` did not run automatically.** The Pre-Tool hook flagged
  pushes to `main` as destructive even though the user authorised "Create
  main". Worked around by having the user push `main` themselves after I
  prepared the local branch.

## Packages & Dependencies

- **Removed**: `directories` (~30 KB, macOS path mismatch with the spec).
- **Confirmed working as expected**: `tempfile` for both fixture tempdirs
  and the `NamedTempFile::persist` atomic-write path.
- **`cargo deny`**: After removing `directories`, the lockfile no longer
  pulls in the `dirs-sys` → `redox_users` → `thiserror 1.x` chain that
  caused the duplicate-thiserror warning on the Phase 2 CI run.

## Patterns & Code

- **Per-subcommand directory module.** `src/commands/catalog/{add,list,…}.rs`
  reads like a manual: file name matches command name, each file owns its
  output formatting (human + JSON) and its error mapping. Reusable for
  future subcommand groups.
- **Subcommand contract per file.** Each command file imports its own slice
  of the lib (manifest, store, git, paths, output) — no module-level
  shared state, no helpers that drift apart. The `source.rs` resolver is
  the one shared helper because all five commands could end up needing it
  for diagnostics.
- **Integration test isolation.** `ToolEnv` pipes `HOME`,
  `XDG_CONFIG_HOME`, `XDG_DATA_HOME` into a fresh `tempfile::TempDir` per
  test. No test ever touches the host's real config. Pattern is reusable
  for any future CLI that uses XDG paths.
